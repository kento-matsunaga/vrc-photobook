# UsageLimit 集約 データモデル設計

> 上流: [UsageLimit ドメイン設計](./ドメイン設計.md) / [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) §3.7, §7.2

---

## 1. 扱うデータ

- 時間窓ごとの操作カウント（作成・通報・アップロード・メール送信）
- bot検証は永続化しない（セッションスコープ）
- 画像制限値（枚数・サイズ）は設定として保持
- **IPハッシュソルトはインフラ層（環境変数）で管理し、Report集約と共有する**（v4 P1-4）

---

## 2. テーブル一覧

| テーブル | 役割 |
|---------|-----|
| `usage_windows` | 操作カウントの時間窓 |
| `usage_policies`（任意） | 操作別の上限値設定（ハードコードでも可） |

MVPでは `usage_policies` をコード側の定数で持ち、DBテーブルは `usage_windows` のみで始めて問題ない。

ソルトそのものは DB ではなく**環境変数 / Secret Manager で管理**する（§8 参照）。

## 3. `usage_windows`

| カラム | 型 | NULL | 既定 | 制約・備考 |
|-------|-----|------|------|----------|
| `actor_hash` | `bytea` | NOT NULL | - | PK 一部。SHA-256(IP + serverSalt) |
| `salt_version` | `smallint` | NOT NULL | `1` | PK 一部。ハッシュ化時のソルトバージョン（v4 P1-4） |
| `operation` | `text` | NOT NULL | - | PK 一部。CHECK: `create_photobook / submit_report / upload_image / manage_url_delivery` |
| `window_start` | `timestamptz` | NOT NULL | - | PK 一部。時間窓の開始（例: 毎正時） |
| `count` | `int` | NOT NULL | `0` | CHECK: `>= 0` |
| `last_incremented_at` | `timestamptz` | NOT NULL | `now()` | |

### 主キーと索引

| 索引 | カラム | 用途 |
|------|-------|------|
| PK | `(actor_hash, salt_version, operation, window_start)` | 集約の一意性 |
| INDEX | `window_start` | 古い窓のGC |
| INDEX | `operation, window_start, count` | 不審挙動の検出（将来の分析） |
| INDEX | `salt_version, window_start` | ソルトローテーション後の古いバージョンの掃除 |

### なぜ `salt_version` を並列カラムに持つか（v4 P1-4）

- 同じ生IPでも、ソルトバージョンが異なれば `actor_hash` は別値になる
- ローテーション直後は新旧バージョンが一時的に混在しうる（ローテーション時点では旧windowがまだ残っている）
- `salt_version` を明示することで、「どのバージョンのソルトで生成されたハッシュか」をクエリで識別でき、古いバージョンの行を掃除できる
- **Report集約の `reports.source_ip_hash_salt_version` と同じ意味論を持つ**（共通ソルトポリシー / 同じ `SaltPolicy.currentVersion` を共有）

### カウンタ更新の実装パターン

```sql
INSERT INTO usage_windows (actor_hash, salt_version, operation, window_start, count, last_incremented_at)
VALUES ($1, $2, $3, $4, 1, now())
ON CONFLICT (actor_hash, salt_version, operation, window_start)
DO UPDATE SET
  count = usage_windows.count + 1,
  last_incremented_at = now()
RETURNING count;
```

返却値の `count` が上限超過していれば、呼び出し側が拒否を返す。

## 4. `usage_policies`（任意、運用パラメータをDB化する場合）

| カラム | 型 | NULL | 既定 | 制約・備考 |
|-------|-----|------|------|----------|
| `operation` | `text` | NOT NULL | - | PK |
| `plan_tier` | `text` | NOT NULL | `'anonymous'` | PK 一部。`anonymous / registered / paid` |
| `limit_count` | `int` | NOT NULL | - | 時間窓あたり上限 |
| `window_duration_seconds` | `int` | NOT NULL | `3600` | 時間窓の長さ（秒） |
| `updated_at` | `timestamptz` | NOT NULL | `now()` | |

MVPでは定数でもよいが、運用中に閾値を変更したいなら早期にDB化が楽。

### 初期値（v4 §3.7 + 本書での設定）

| operation | plan_tier | limit_count | window |
|-----------|-----------|-------------|--------|
| `create_photobook` | `anonymous` | 5 | 1時間 |
| `submit_report` | `anonymous` | 10 | 1時間 |
| `upload_image` | `anonymous` | 100 | 1時間 |
| `manage_url_delivery` | `anonymous` | 3 | 1時間 |

## 5. 画像制限値（設定）

テーブル化する必要はなく、アプリ設定または `usage_policies` の拡張で対応。

| 設定キー | 初期値 | 備考 |
|---------|-------|------|
| `image.max_per_photobook` | 20 | v4 §3.7 |
| `image.max_byte_size` | 10,485,760 (10MB) | v4 §3.7 |

UsageLimit はこれら**上限値の提供**とアップロード枚数のカウント（`OperationKind = upload_image` の `usage_windows` 行）を担う。

**画像実体の検証**（MIME / マジックナンバー / 最大長辺 8192px / 最大 40MP / アニメーション拒否 / SVG禁止 / デコンプレッションボム対策）は Image 集約の `ImageIngestionService` が担う（v4 §3.7 / §3.10、UsageLimit ドメイン設計 §12 参照）。

## 6. 保持期間と削除ポリシー

v4 §7.2 に従う最小保持。

| 対象 | 保持期間 | アクション |
|------|---------|----------|
| `usage_windows` の行 | `window_start < now() - INTERVAL '24 hours'` | 削除（日次バッチ） |
| `usage_windows` で `salt_version` が `SaltPolicy.currentVersion` 未満の行 | ローテーションから 24 時間経過後 | 削除（ローテーション後バッチ） |

時間窓が過ぎた行は意味がないため、24時間以上経過したものは物理削除して問題ない。監査目的があれば別テーブルに集計を退避する。

## 7. bot検証の永続化方針

**永続化しない**。

- Turnstile等のトークンはサーバーが受理した時点で検証APIに送る
- 検証結果はセッション／リクエストスコープで保持
- 公開操作が完了したら破棄
- DB保存は不要（v4 §7.2 最小保持原則）

ただし、検証失敗が多発した場合のアラート用に、集計値のみ `usage_windows` とは別に `bot_challenge_failures` テーブルを持つ選択肢はある（Phase 2以降で検討）。

## 8. ソルト管理（v4 新規 / P1-4）

### 8.1 保存場所

ソルトは DB には保存しない。以下のいずれかで管理する。

- 環境変数（MVP 推奨、`IP_HASH_SALT_V1=<base64>` 等）
- Cloud Secret Manager（Phase 2 以降で運用が複雑化した場合）

### 8.2 バージョニング

環境変数のキー名に明示的にバージョンを含める。

| 環境変数 | 意味 |
|---------|-----|
| `IP_HASH_SALT_CURRENT_VERSION` | 現在の `SaltPolicy.currentVersion`（例: `1`） |
| `IP_HASH_SALT_V1` | バージョン1のソルト値（base64エンコード、32バイト以上） |
| `IP_HASH_SALT_V2` | バージョン2のソルト値（ローテーション時に追加） |

新規ハッシュ生成時は `IP_HASH_SALT_CURRENT_VERSION` が示すバージョンのソルトを用いる。

### 8.3 Report集約との共有

- UsageLimit の `actor_hash` と Report の `source_ip_hash` は**同じソルト実体、同じバージョン管理**を共有する
- インフラ層では、両集約の Hasher が同一のソルト取得関数（例: `GetIpHashSalt(version int) []byte`）を参照する
- `IP_HASH_SALT_CURRENT_VERSION` は両集約で共通
- ドメイン層では `SaltPolicy` が単一インスタンス（DIコンテナで共有）

### 8.4 ローテーション運用

1. 新しいソルト（`IP_HASH_SALT_V2`）を環境に追加
2. `IP_HASH_SALT_CURRENT_VERSION=2` に切り替え（アプリ再起動またはホットリロード）
3. 以後の新規 `usage_windows` 行は `salt_version=2` で記録される
4. `salt_version=1` の既存行は 24時間の時間窓満了後、日次バッチで掃除される
5. 運用ログに切替日時を記録（監査用）

### 8.5 ローテーションの効果と代償

| 項目 | 効果 |
|------|-----|
| プライバシー | 長期追跡性の喪失（同一IPでもバージョンが違えば別集計） |
| 相関検出 | 同一ローテーション期間内のみ UsageLimit / Report 相関が成立 |
| ソルト漏洩時の影響範囲 | ローテーション期間で打ち切り可能 |

長期追跡性の喪失は**意図的な設計**であり、MVP 要件（同一期間内での大量投稿・大量通報の相関検出）を満たす。

## 9. インフラ層マッピング

```
backend/usage_limit/
├── domain/
│   ├── entity/
│   │   └── usage_window.go
│   └── vo/
│       ├── actor_identifier/
│       ├── salt_policy/                # v4 新規
│       ├── operation_kind/
│       ├── time_window/
│       ├── limit_policy/
│       └── bot_challenge_result/
├── infrastructure/
│   ├── repository/rdb/
│   │   └── usage_window_repository.go
│   ├── hasher/
│   │   └── actor_identifier_hasher.go  # v4 新規、Reportと同じソルト実体を共有
│   └── bot/
│       ├── bot_verifier.go
│       └── turnstile_verifier.go
└── internal/
    └── usecase/
        ├── rate_limit_service.go
        └── image_constraint_validator.go

shared/
└── ip_hash_salt/                       # v4 新規、UsageLimit と Report で共有
    └── salt_provider.go                # 環境変数からソルトを取得
```

`shared/ip_hash_salt/` は UsageLimit と Report の両方から参照される。ここで `SaltPolicy.currentVersion` とバージョン別ソルト値を一元管理する。

## v3からの主な変更点

| 項目 | v3 | v4 |
|------|----|----|
| `usage_windows.salt_version` カラム | なし（ソルトの版管理はドキュメント記述のみ） | **追加**。PK の一部として `(actor_hash, salt_version, operation, window_start)` で一意（v4 P1-4） |
| ソルト管理セクション | §6「保持期間と削除ポリシー」内で簡潔に触れるのみ | **§8 を独立節として新設**。保存場所（環境変数）、バージョニング規約、Report との共有方針、ローテーション手順、代償を明文化 |
| Report 集約とのソルト共有 | 言及なし | **明示**。`reports.source_ip_hash` と同じソルト実体・同じ `SaltPolicy.currentVersion` を共有（v4 P1-4） |
| `shared/ip_hash_salt/` インフラ層 | なし | **新設**。UsageLimit と Report の両方が参照する単一のソルト提供層 |
| 画像制限値の責務 | `image.max_per_photobook` / `image.max_byte_size` を保持、Photobook集約・Image集約が参照 | 同左に加え、**Image集約 `ImageIngestionService` と UsageLimit の責務分担を明記**（MIME/マジックナンバー/ピクセル/ボム対策は Image 側、枚数・サイズ上限値と枚数カウントは UsageLimit 側）（v4 P0-12関連） |
| ローテーション後の掃除 | 24時間経過で削除のみ | **ローテーション後 24 時間で旧 `salt_version` の行を掃除するバッチ**を追加（§6） |
| 索引 | `window_start`, `(operation, window_start, count)` | 左記に加え `(salt_version, window_start)` を追加 |
