# UsageLimit 集約 ドメイン設計

> 上流: [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) 第3.7節 / §6.11 / §7.2

---

## 1. 集約の責務

ログイン不要サービスでの**荒らし・スパム・大量投稿を抑止**するための利用制限を管理する。

### 担う責務

- 同一作成元（IPアドレス等）からの作成レートを計測する
- 1フォトブックあたりの画像枚数上限を適用する
- 1画像あたりのファイルサイズ上限を適用する
- 通報送信のレート制限を適用する
- bot検証（Turnstile等）の結果を受理する
- **IPハッシュソルトポリシーの保持と、Report集約への共有**（v4 P1-4）

### 担わない責務

- 実際のbot検証の実装（外部サービス、インフラ層）
- 画像ファイルの保存や形式変換（Image集約）
- **画像形式の検証（MIME/マジックナンバー/実デコード）**（Image集約の `ImageIngestionService` が担う）
- センシティブ画像の自動検出（v4 §3.7、MVPはスコープ外）
- 通報されたフォトブックの自動削除（運営判断）

## 2. 集約境界と集約ルート

### 集約ルート: `UsageWindow`

「ある作成元」×「ある時間窓」×「ある操作」ごとの集計を1集約とする。

```
UsageWindow (集約ルート)
  ├── actor: ActorIdentifier     // IPハッシュ（ソルトバージョン付き）
  ├── operation: OperationKind   // create_photobook / submit_report など
  ├── window: TimeWindow         // 1時間窓
  └── count: int
```

利用制限は本質的に**短寿命の集計データ**であり、長期保管はしない。

---

## 3. エンティティ

### 3.1 UsageWindow（集約ルート）

| 属性 | 型 | 説明 |
|------|----|----|
| id | `UsageWindowId` | 識別子（actor + operation + window のハッシュでもよい） |
| actor | `ActorIdentifier` | 操作主体の識別子（ソルトバージョン情報を内包） |
| operation | `OperationKind` | 何の操作か |
| windowStart | `Timestamp` | 時間窓の開始時刻 |
| windowDuration | `Duration` | 時間窓の長さ（例: 1時間） |
| count | `int` | 時間窓内のカウント |
| lastIncrementedAt | `Timestamp` | 最終更新時刻 |

## 4. 値オブジェクト

### 4.1 `ActorIdentifier`

| 属性 | 型 | 制約 |
|------|----|-----|
| hash | `bytea` | SHA-256(rawIp + saltBySaltVersion) |
| saltVersion | `int` | ハッシュ化時に用いたソルトのバージョン番号 |

**不変条件**:

- 生のIPアドレスは永続化しない（プライバシー配慮、v4 §7.2）
- `hash` は必ず `saltVersion` とセットで保持される（どのソルトで作られたかが追跡可能）
- **Report集約の `SourceIpHash` と同じソルトポリシー（同じ `SaltPolicy`）を用いて生成する**（v4 P1-4）

### 4.2 `SaltPolicy`（v4 新規）

UsageLimit と Report で共有されるIPハッシュソルトのポリシーを表す概念。

| 属性 | 型 | 説明 |
|------|----|----|
| currentVersion | `int` | 現在有効なソルトのバージョン番号 |
| algorithm | `enum` | ハッシュアルゴリズム（MVP: `SHA-256`） |
| rotationPolicy | `text` | ローテーション方針の説明（運用的な記述） |

**業務ルール**:

- `SaltPolicy` は UsageLimit と Report の双方が参照する単一のポリシーである
- ソルト値そのものはドメイン層には露出せず、インフラ層（環境変数）で管理する
- 新しい `currentVersion` に切り替わったら、以後の新規ハッシュは新バージョンで生成される
- 過去のバージョンでハッシュ化された `ActorIdentifier` / `SourceIpHash` は、新バージョンとは別集計として扱う（同一主体であっても区別される）
- これは**意図的な設計**であり、長期追跡性の喪失はローテーションの対価として許容する

### 4.3 `OperationKind`

| 値 | 対応操作 |
|----|---------|
| `create_photobook` | フォトブック新規作成（公開成立） |
| `submit_report` | 通報送信 |
| `upload_image` | 画像アップロード |
| `manage_url_delivery` | 管理URL控えメール送信 |

### 4.4 `TimeWindow`

| 属性 | 値 | 備考 |
|------|----|----|
| windowStart | `Timestamp` | 窓の開始時刻 |
| windowDuration | `Duration` | 1時間固定（MVP） |

### 4.5 `LimitPolicy`（ドメインサービス的ポリシー）

v4 §3.7 付随ルールを実装する純粋関数。

| OperationKind | 上限値（MVP無料枠） | 備考 |
|---------------|-------------------|-----|
| `create_photobook` | **5冊 / 1時間** | v4 §3.7 |
| `submit_report` | **10件 / 1時間** | スパム通報抑止（本書で設定） |
| `upload_image` | **100枚 / 1時間** | 過剰アップロード抑止（本書で設定） |
| `manage_url_delivery` | **3件 / 1時間** | メール爆撃抑止（本書で設定） |

### 4.6 `BotChallengeResult`

| 属性 | 値 |
|------|----|
| verified | `bool` |
| provider | `turnstile` / `recaptcha` / その他 |
| verifiedAt | `Timestamp` |
| token | 検証トークン（検証後は即破棄、保存しない） |

**業務ルール**:

- `verified == true` でなければ、フォトブックの公開操作は拒否される
- bot検証結果はセッションスコープで短期保持し、公開操作完了後に破棄

### 4.7 画像制限系VO

| VO | 値 | 業務ルール |
|----|-----|----------|
| `ImageCountLimit` | 整数 | 1フォトブックあたり **20枚** まで（MVP） |
| `ImageSizeLimit` | バイト | 1画像 **10MB** まで（MVP） |

これらは Image集約の `ImageIngestionService` から参照される「上限値の提供」に徹する。実際の画像バイトの中身を検証するのは Image 集約の責務（§12参照）。

## 5. 不変条件（Invariants）

- **I1**: `count >= 0`
- **I2**: `(actor, operation, windowStart)` の組は一意
- **I3**: `count > LimitPolicy.limit(operation)` のとき、同じ `(actor, operation)` に対する新規操作は拒否される
- **I4**: `windowStart + windowDuration` を過ぎた `UsageWindow` は無効化される（物理的には削除または新窓を生成）
- **I5**: `ActorIdentifier` は生のIPを含まない
- **I6（v4）**: `ActorIdentifier` は `saltVersion` を必ず持ち、ハッシュ値と整合する
- **I7（v4）**: UsageLimit と Report の `SaltPolicy.currentVersion` は常に同じ値を指す（同じソルト管理単位を共有する）

## 6. ドメインサービス

### 6.1 `RateLimiter`

操作試行ごとに `UsageWindow` を参照・更新し、上限超過時に拒否する。

```
tryAcquire(actor, operation) -> Allowed | RateLimited
```

- 現在の時間窓の `count` を取得
- `LimitPolicy.limit(operation)` と比較
- 許可されれば `count` をインクリメント
- 許可されなければ `RateLimited` を返し、再試行可能時刻を付随させる

**実装上の選択肢**:

- RDBの楽観ロックで実装（テーブル `usage_windows`）
- Redis等のインメモリキャッシュで実装（高速だがMVPでは必須ではない）

### 6.2 `BotVerifier`

外部bot検証サービス（Turnstile等）のトークンを検証し、`BotChallengeResult` を返す。

- 検証失敗時は公開操作を拒否
- 検証結果はセッションスコープで保持し、永続化しない

### 6.3 `ImageConstraintValidator`

アップロード受付時に画像の**枚数・バイトサイズ**上限を検証する純粋関数。

```
validateUpload(photobookId, imageByteSize, currentPhotoCount) -> Ok | LimitViolation
```

**担うこと**: 枚数の超過判定、バイトサイズの超過判定。

**担わないこと**: 画像形式の検証（MIME/マジックナンバー/ピクセル数/アニメーションフレーム）。これらは Image 集約の `ImageIngestionService` が担う（§12参照）。

### 6.4 `ActorIdentifierHasher`（v4 新規）

`SaltPolicy` を用いて、生のIPアドレスから `ActorIdentifier` を生成するドメインサービス。

```
hash(rawIp: string) -> ActorIdentifier
```

- 現在の `SaltPolicy.currentVersion` に対応するソルトで SHA-256 ハッシュを計算する
- 生成される `ActorIdentifier` は `(hash, saltVersion)` のペアを持つ
- **Report 集約の `SourceIpHasher` と同じ `SaltPolicy` を共有する**ことが必須（同一の DI コンテナで同一インスタンスを注入する等）

## 7. ドメインイベント

| イベント | 発火タイミング | 用途 |
|---------|--------------|-----|
| `RateLimitExceeded` | 上限超過時 | 運営通知、不審挙動検出 |
| `BotChallengeFailed` | bot検証失敗時 | ログ、運営通知 |

MVPでは通常のエラーレスポンスで十分で、イベントは将来の拡張用。

---

## 8. 操作

| 操作 | 説明 |
|------|-----|
| `RateLimiter.tryAcquire(actor, op)` | 上限チェック＋インクリメント |
| `BotVerifier.verify(token)` | bot検証 |
| `ImageConstraintValidator.validate(...)` | 画像枚数・サイズ上限検証 |
| `ActorIdentifierHasher.hash(rawIp)` | ソルトポリシーに従って `ActorIdentifier` を生成 |

UsageWindowエンティティ自体への直接操作はサービス層で閉じ、外部には露出しない。

## 9. 他集約との境界

| 関係先 | 参照方向 | 内容 |
|--------|---------|-----|
| Photobook | Photobook公開時にサービス層で参照 | `tryAcquire('create_photobook')` |
| Report | 通報送信時にサービス層で参照 | `tryAcquire('submit_report')` + **`SaltPolicy` を共有** |
| Image | アップロード時にサービス層で参照 | `ImageConstraintValidator`（枚数・サイズのみ） |
| ManageUrlDelivery | メール送信時にサービス層で参照 | `tryAcquire('manage_url_delivery')` |

UsageLimit集約は他集約を参照しない（受動的）。

### Report集約とのソルト共有（v4 P1-4）

- `Report.source_ip_hash` と `UsageLimit.actor_hash` は、**同じ `SaltPolicy`（同じソルト値、同じバージョン管理）** で生成される
- 両集約は独立しているが、インフラ層で同じソルト実体を共有する
- 目的: 同一作成元からの**大量投稿（UsageLimit）と大量通報（Report）の相関検出**を MVP から可能にするため
- ローテーション時は両集約で同時に新バージョンに切り替わる

### Image集約との責務分担

| 責務 | 担当 |
|------|-----|
| 枚数上限（20枚/冊）の保持 | UsageLimit |
| バイトサイズ上限（10MB）の保持 | UsageLimit |
| 枚数・バイトサイズ超過判定 | UsageLimit の `ImageConstraintValidator` |
| MIME / マジックナンバー検証 | Image の `ImageIngestionService` |
| 実デコードによる形式確認 | Image の `ImageIngestionService` |
| SVG 拒否 / アニメーション拒否 | Image の `ImageIngestionService` |
| 最大長辺 8192px / 最大 40MP 判定 | Image の `ImageIngestionService` |
| デコンプレッションボム対策 | Image の `ImageIngestionService` |
| EXIF / XMP / IPTC 除去 | Image の配信用正規化処理 |

## 10. 将来拡張の余地

v4 §3.7 付随ルールで「将来の任意ユーザー枠」「将来の有料プラン枠」が言及されている。

- `LimitPolicy` は `PlanTier`（`anonymous` / `registered` / `paid`）を引数に取れる設計とする
- MVP では `anonymous` のみを実装
- Phase 2 以降で `registered` 追加、Phase 5 で `paid` 追加

## 11. IPハッシュソルトポリシー（v4 新規 / P1-4）

### 11.1 共有スコープ

- UsageLimit集約の `ActorIdentifier.hash`
- Report集約の `SourceIpHash`

この2つは **同一の `SaltPolicy`** を共有する。Report担当の `SourceIpHasher` と UsageLimit の `ActorIdentifierHasher` は、インフラ層で同一のソルト実体を参照する。

### 11.2 現在のバージョン

| 項目 | 値 |
|------|----|
| `currentVersion` | `1`（MVP 初期値） |
| `algorithm` | `SHA-256` |
| ソルト長 | 32バイト以上（暗号学的乱数） |
| 保存場所 | 環境変数 / Secret Manager（ドメイン層には露出させない） |

### 11.3 ローテーション手順

1. 新しいソルト値（例: `version=2`）を環境に追加する（旧ソルトとの並行保持は不要）
2. `SaltPolicy.currentVersion` を `2` に切り替える
3. 以後の新規ハッシュは `version=2` で生成される
4. 既存の `version=1` のハッシュは、同一主体であっても別物として扱われる
5. 運用ドキュメントに切替日時を記録する

### 11.4 ローテーションの影響

- **長期追跡性の喪失**: 同一主体からの累積行動（数ヶ月以上）は追跡できなくなる
- **許容する理由**:
  - プライバシー保護（生IP非保存原則と整合）
  - ソルト漏洩時の影響を一定期間で打ち切れる
  - 相関検出は「同一ローテーション期間内」で成立していればMVP要件を満たす
- **ローテーション推奨頻度**: 四半期〜半年程度（運用実態に応じて調整）

## 12. 画像アップロード制限とImage集約との責務分担（v4 / P0-12関連）

v4 §3.7 では画像アップロード時の安全検証が明文化された。この責務を UsageLimit と Image で以下のように分担する。

### UsageLimit が担う部分

- **枚数上限値の保持**（20枚/冊、`ImageCountLimit`）
- **バイトサイズ上限値の保持**（10MB、`ImageSizeLimit`）
- **アップロード枚数のカウント**（`OperationKind = upload_image` の `UsageWindow`）
- 枚数・バイトサイズの超過判定（`ImageConstraintValidator`）

### Image集約（`ImageIngestionService`）が担う部分

- MIME 判定
- マジックナンバー判定
- 実デコードによる形式確認
- SVG 拒否
- アニメーション WebP / APNG 拒否
- 最大長辺 8192px 判定
- 最大ピクセル数 40MP 判定
- デコンプレッションボム対策
- EXIF / XMP / IPTC 除去

この分担により、UsageLimit は**利用量の制限ポリシー**に専念し、Image は**画像実体の安全性**に専念する。

## v3からの主な変更点

| 項目 | v3 | v4 |
|------|----|----|
| `ActorIdentifier` | SHA-256 + サーバーソルト、バージョンは不変条件としてのみ言及 | **`saltVersion` を属性として明示保持**。Report と同じソルトポリシー（P1-4） |
| `SaltPolicy` VO | なし | **新設**。UsageLimit と Report で共有されるソルトポリシーを表現 |
| `ActorIdentifierHasher` ドメインサービス | なし | **新設**。Report の `SourceIpHasher` と同じ `SaltPolicy` 実体を共有 |
| Report集約との境界 | 通報送信時のレート制限のみ言及 | **ソルトポリシー共有を明記**（P1-4） |
| 画像アップロード制限の責務 | `ImageConstraintValidator` のみ | **§12 を新設**し、UsageLimit（枚数・サイズ上限値と枚数カウント）と Image（MIME/マジックナンバー/ピクセル/ボム対策）の責務分担を明確化（P0-12関連） |
| IPハッシュソルトポリシー節 | 未定義（データモデル側で簡潔に触れるのみ） | **§11 を新設**。現在バージョン、ローテーション手順、長期追跡性喪失の許容を明文化 |
| 不変条件 | I1〜I5 | **I6（saltVersion整合）、I7（UsageLimit/Report のソルトポリシー一致）を追加** |
