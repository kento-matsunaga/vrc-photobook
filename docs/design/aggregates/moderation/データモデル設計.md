## 1. 扱うデータ

- 運営アクションの履歴（**追記型・イミュータブル**）
- 通報（Report）との紐付け（`source_report_id`）
- 運営ラベル、実行理由、実行時刻、補正ペアリング（`correlation_id`）

本集約は状態を持つ集約ではなく、**監査ログ的な追記型集約**として扱う。

---

## 2. テーブル一覧

| テーブル | 役割 |
|---------|-----|
| `moderation_actions` | 運営アクション履歴（追記のみ、UPDATE / DELETE 禁止） |

1 テーブルで完結する。Photobook / Report / Outbox など他集約の更新は、**同一 DB トランザクション内**で行うが、それぞれ各集約のテーブル（`photobooks` / `reports` / `outbox_events`）に書かれる（ドメイン設計 §1）。

---

## 3. `moderation_actions` テーブル

| カラム | 型 | NULL | 既定 | 制約・備考 |
|-------|-----|------|------|----------|
| `id` | `uuid` | NOT NULL | `gen_random_uuid()` | PK |
| `kind` | `text` | NOT NULL | - | CHECK: `hide / unhide / soft_delete / restore / purge / reissue_manage_url` |
| `target_photobook_id` | `uuid` | NOT NULL | - | Photobook への参照（FK なし、§4） |
| `source_report_id` | `uuid` | NULL | - | Report への参照（FK なし、§4）。指定時、`hide`/`soft_delete`/`purge` で Report.status を `resolved_action_taken` に同一 TX で遷移 <!-- 付録C P0-20 --> |
| `actor_label` | `text` | NOT NULL | - | 運営識別子（個人情報不可、例: `ops-1` / `legal-team`） |
| `reason` | `text` | NOT NULL | - | CHECK: `report_based_harassment / report_based_unauthorized_repost / report_based_sensitive_violation / report_based_minor_related / report_based_subject_removal / rights_claim / creator_request_manage_url_reissue / erroneous_action_correction / policy_violation_other` |
| `detail` | `text` | NULL | - | 0–2000 文字、内部参照用（公開しない） |
| `correlation_id` | `uuid` | NULL | - | 関連アクション（`hide` ↔ `unhide`、`soft_delete` ↔ `restore` のペアリング等） |
| `executed_at` | `timestamptz` | NOT NULL | `now()` | 実行時刻 |

v3 のスキーマ定義を維持する。v4 で新設カラムはない（責務拡張はアプリ層の同一トランザクション化で吸収する）。

## 4. 外部キー方針（v3 維持） <!-- 付録C P0-22 -->

`target_photobook_id` と `source_report_id` には**FK 制約を付けない**。

**理由**:

- Moderation は監査ログ的性質を持ち、対象 Photobook が物理削除（`purged`）された後も**記録として残す必要がある**（v4 §7.3 権利侵害対応証跡、§7.4 未成年保護対応の監査証跡）
- FK を付けると `ON DELETE` の挙動が複雑になる（`SET NULL` は監査証跡を失わせ、`CASCADE` は歴史を破壊する）
- Report 側でも同様に `reports.target_photobook_id` に FK を付けない方針（v4 §3.6）。Moderation と Report は整合的に「FK なし + snapshot または id 保持」戦略を採用
- データ整合性はアプリ層（`ModerationActionExecutor` の同一トランザクション原則）と Reconcile スクリプトで保証する

## 5. 索引（v3 維持）

| 索引 | カラム | 用途 |
|------|-------|------|
| PK | `id` | - |
| INDEX | `target_photobook_id, executed_at DESC` | 特定 Photobook のアクション履歴（最新順） |
| INDEX | `source_report_id` WHERE `source_report_id IS NOT NULL` | 通報からの逆引き |
| INDEX | `kind, executed_at DESC` | アクション種別別の集計 |
| INDEX | `actor_label, executed_at DESC` | 運営の活動ログ |
| INDEX | `reason, executed_at DESC` | 理由別の集計（`report_based_minor_related` の頻度監視等） |
| INDEX | `correlation_id` WHERE `correlation_id IS NOT NULL` | ペア参照（hide↔unhide 等） |

## 6. イミュータビリティの強制（v3 維持）

- **アプリ層**: `moderation_actions` に対しては INSERT のみ発行する。UPDATE / DELETE 文は発行しない
- **DB 層**: 可能なら UPDATE / DELETE を拒否するトリガまたは RULE を設置する

```sql
-- 例: UPDATE を拒否するトリガ
CREATE OR REPLACE FUNCTION moderation_actions_forbid_update()
RETURNS trigger AS $$
BEGIN
  RAISE EXCEPTION 'moderation_actions is append-only';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER moderation_actions_no_update
BEFORE UPDATE OR DELETE ON moderation_actions
FOR EACH ROW EXECUTE FUNCTION moderation_actions_forbid_update();
```

補正は新規レコード（`unhide` / `restore` / `erroneous_action_correction` 等）として追加する（ドメイン設計 I3）。

## 7. 保持期間と削除ポリシー

Moderation は**監査ログ**として**恒久保持**する。

| 対象 | 保持期間 | 備考 |
|------|---------|-----|
| `moderation_actions` 行 | **恒久保持** | 物理削除しない |

ただし、関連 Photobook が `purged` された後、`detail` フィールドに含まれうる個人情報類（v4 §7.2 準拠）は匿名化する運用を推奨する。

```sql
-- Photobook 物理削除時に detail を匿名化（任意、運営判断）
UPDATE moderation_actions
SET detail = '[purged]'
WHERE target_photobook_id = $1
  AND detail IS NOT NULL;
```

ただし、この UPDATE は §6 のイミュータビリティ原則と衝突するため、運用上は「`detail` に個人情報を書かない」ことを優先し、匿名化バッチは**例外運用**として別 RDB ユーザー権限で実行する。

## 8. 運営スクリプトからの利用（v4: 同一トランザクション）

MVP では運営 UI を持たないため、運営スクリプト経由でアプリケーションのユースケース層を呼び出す（v4 §5.4）。

### 想定スクリプト（`scripts/ops/`）

| スクリプト | 機能 |
|-----------|-----|
| `ops-hide.sh <photobook_id> --reason <R> --operator <L> [--report <RID>]` | 一時非表示 |
| `ops-unhide.sh <photobook_id> --operator <L> [--correlation <AID>]` | 解除 |
| `ops-soft-delete.sh <photobook_id> --reason <R> --operator <L> [--report <RID>]` | 論理削除 |
| `ops-restore.sh <photobook_id> --operator <L> [--correlation <AID>]` | 復元 |
| `ops-purge.sh <photobook_id> --reason <R> --operator <L> [--report <RID>]` | 物理削除 |
| `ops-reissue-manage-url.sh <photobook_id> --reason <R> --operator <L>` | 管理URL再発行 |

### 同一 DB トランザクションで行う更新（v4 P0-5） <!-- 付録C P0-19, P0-20, P0-21 -->

各スクリプトは内部で `ModerationActionExecutor` 経由のユースケースを呼び、**以下すべてを単一トランザクションで**実行する。

1. `photobooks` テーブルの状態更新（`hidden_by_operator` / `status` / `manage_url_token` / `manage_url_token_version` 等）
2. `moderation_actions` への INSERT
3. `reports` テーブルの更新（`--report` が指定され、かつ kind が `hide` / `soft_delete` / `purge` の場合のみ。ドメイン設計 §9）
   - `reports.status = 'resolved_action_taken'`
   - `reports.resolved_by_moderation_action_id = <new action id>`
   - `reports.resolved_at = now()`
4. `outbox_events` への INSERT（ActionKind に対応する event_type。ドメイン設計 §7 のマッピング表）

いずれか失敗したら全体ロールバック。`unhide` / `restore` / `reissue_manage_url` では Report を自動更新しない（ドメイン設計 I7）。

### 擬似 SQL（`hide` 通報起点）

```sql
BEGIN;

-- 1. Photobook 状態更新
UPDATE photobooks
SET hidden_by_operator = true, updated_at = now()
WHERE id = $photobook_id;

-- 2. moderation_actions 追記
INSERT INTO moderation_actions (
  id, kind, target_photobook_id, source_report_id,
  actor_label, reason, detail, executed_at
) VALUES (
  $action_id, 'hide', $photobook_id, $report_id,
  $operator, 'report_based_minor_related', $detail, now()
);

-- 3. Report 状態更新（source_report_id があるため）
UPDATE reports
SET status = 'resolved_action_taken',
    resolved_by_moderation_action_id = $action_id,
    resolved_at = now()
WHERE id = $report_id;

-- 4. Outbox イベント追記
INSERT INTO outbox_events (
  id, aggregate_type, aggregate_id, event_type, payload_json, status, created_at
) VALUES (
  gen_random_uuid(), 'photobook', $photobook_id, 'PhotobookHidden',
  jsonb_build_object('action_id', $action_id, 'source_report_id', $report_id,
                     'reason', 'report_based_minor_related', 'actor_label', $operator),
  'pending', now()
);

COMMIT;
```

DB を直接 UPDATE する運用は禁止する（監査証跡が壊れ、Outbox が発火されず副作用が走らない）。

## 9. 他集約との整合性

### 9.1 Photobook との整合性

- Photobook の `hidden_by_operator` は、直近の `hide` / `unhide` アクションと一致すべき
- `manage_url_token_version` は `reissue_manage_url` アクションの回数と一致すべき
- `photobooks.status` は直近の `soft_delete` / `restore` / `purge` と一致すべき

**整合性検証クエリ（hidden_by_operator）**:

```sql
-- hidden_by_operator=true なのに直近アクションが hide でない Photobook を検出
SELECT p.id, p.hidden_by_operator,
  (SELECT kind FROM moderation_actions ma
   WHERE ma.target_photobook_id = p.id
     AND ma.kind IN ('hide', 'unhide')
   ORDER BY ma.executed_at DESC LIMIT 1) AS latest_hide_kind
FROM photobooks p
WHERE p.hidden_by_operator = true
  AND (SELECT kind FROM moderation_actions ma
       WHERE ma.target_photobook_id = p.id
         AND ma.kind IN ('hide', 'unhide')
       ORDER BY ma.executed_at DESC LIMIT 1) <> 'hide';
```

### 9.2 Report との整合性（v4 新規）

- `source_report_id` が指定された `hide` / `soft_delete` / `purge` アクションに対応する Report は、アクション実行後に `status='resolved_action_taken'` かつ `resolved_by_moderation_action_id` が当該アクションを指しているべき
- `unhide` / `restore` / `reissue_manage_url` では Report 状態は自動変更されない（ドメイン設計 I7）

**整合性検証クエリ（Report 連動）**:

```sql
-- 通報起点の hide/soft_delete/purge を実行したのに、対応 Report が resolved_action_taken でないケースを検出
SELECT ma.id AS action_id, ma.kind, ma.source_report_id,
       r.status AS report_status, r.resolved_by_moderation_action_id
FROM moderation_actions ma
JOIN reports r ON r.id = ma.source_report_id
WHERE ma.kind IN ('hide', 'soft_delete', 'purge')
  AND ma.source_report_id IS NOT NULL
  AND (r.status <> 'resolved_action_taken' OR r.resolved_by_moderation_action_id <> ma.id);
```

### 9.3 Outbox との整合性（v4 新規）

- すべての `moderation_actions` 行は、**同一時刻に作成された対応する `outbox_events` 行**を持つべき

**整合性検証クエリ（Outbox 連動）**:

```sql
-- ModerationAction があるのに対応 Outbox イベントが存在しないケースを検出
SELECT ma.id AS action_id, ma.kind, ma.target_photobook_id, ma.executed_at
FROM moderation_actions ma
LEFT JOIN outbox_events oe
  ON oe.aggregate_type = 'photobook'
 AND oe.aggregate_id = ma.target_photobook_id
 AND oe.payload_json->>'action_id' = ma.id::text
WHERE oe.id IS NULL;
```

これらの検証クエリは Reconcile スクリプト（例: `photobook_image_integrity.sh` 相当の拡張）の一部または運営の監査バッチに組み込む。

## v3 からの主な変更点

- **§8（刷新）**: 運営スクリプトの内部実装として、**Photobook + `moderation_actions` + Report + `outbox_events` を単一 DB トランザクション**で更新する要件を明示（v4 P0-5）。擬似 SQL で 4 要素の INSERT/UPDATE を具体化。
- **§9.2（新規）**: Report との整合性検証クエリを追加。`hide` / `soft_delete` / `purge` 実行時に対応 Report が `resolved_action_taken` に遷移しているかを検出。`unhide` / `restore` は自動変更されないことを明示。
- **§9.3（新規）**: Outbox との整合性検証クエリを追加。`moderation_actions` 行と対応 `outbox_events` 行のペア欠損を検出（v4 P0-13）。
- **§10**: インフラ層マッピングに `executor.go`（`ModerationActionExecutor`）を明示し、ユースケース層で同一 `sql.Tx` を共有する方針を追記。
- **§1 / §4**: Report 側も FK を持たない方針（v4 §3.6）と整合することを明示。
- スキーマ本体（§3 カラム定義、§5 索引、§6 イミュータビリティ、§7 保持期間）は v3 をそのまま維持（v4 の責務拡張はアプリ層の同一 TX 化で吸収するため）。
