## 1. 集約の責務

運営による**フォトブックへの介入アクション**を記録し、同時に他集約（Photobook / Report）に対する状態変更指示を発行し、副作用伝搬のための Outbox イベントを発火する。

### v4 同一トランザクション原則（最上位不変条件）

運営の1アクションは、以下**すべてを同一DBトランザクション**で原子的に更新する（v4 §5.4, P0-5）。

1. **Photobook 状態変更**（`hidden_by_operator` / `status` / `manage_url_token` 等）
2. **`moderation_actions` への追記**（イミュータブルな履歴）
3. **Report 状態更新**（`sourceReportId` が指定されている場合のみ）
4. **`outbox_events` への INSERT**（副作用伝搬イベント）

いずれか1つでも失敗したら全体をロールバックする。中途半端な状態を許容しない。

### 担う責務

- 一時非表示の開始・解除の記録と Photobook 状態反映
- 管理URLの失効と新規発行の記録と Photobook トークン更新
- 論理削除・物理削除・復元の記録と Photobook 状態反映
- 通報を起点とした運営判断の根拠の追跡（`sourceReportId`）
- 通報起点アクション（hide / soft_delete / purge）における Report 状態の自動遷移
- 誤削除・誤通報への対応（restore, unhide）の履歴
- 副作用（CDNパージ、OGP無効化、画像物理削除、メール送付）への伝搬イベント発火

### 担わない責務

- フォトブック本体のドメインロジック → **Photobook 集約**（Moderation はそこへの指示を出す）
- 通報の受付・記述フィールドの管理 → **Report 集約**
- 実際の CDN パージ / メール送信 / 画像削除 → **Outbox ワーカーとそのハンドラ**
- 運営向け管理 UI → MVP スコープ外（v4 §5.4。運営スクリプトおよび Reconcile で代替）

## 2. 集約境界と集約ルート

### 集約ルート: `ModerationAction`

運営の1アクション = 1集約。追記型（イミュータブルな履歴）。

```
ModerationAction (集約ルート)
  ├── id: ActionId
  ├── kind: ActionKind
  ├── targetPhotobookId: PhotobookId
  ├── sourceReportId: ReportId?
  ├── actorLabel: OperatorLabel
  ├── reason: ActionReason
  ├── detail: ActionDetail?
  ├── correlationId: ActionId?
  └── executedAt: Timestamp
```

補正（取り消しに相当する操作）は**既存アクションを変更せず、新しい補正アクションを追記する**形で表現する。例: `hide` の取り消しは `unhide` アクションの追記。

## 3. エンティティ

### 3.1 ModerationAction（集約ルート / イミュータブル）

| 属性 | 型 | 説明 |
|------|----|----|
| id | `ActionId` | アクション ID（UUIDv4 / ULID） |
| kind | `ActionKind` | アクション種別 |
| targetPhotobookId | `PhotobookId` | 対象 Photobook |
| sourceReportId | `ReportId?` | 起点となった通報（任意） |
| actorLabel | `OperatorLabel` | 実行運営の識別子（個人情報を含まないラベル） |
| reason | `ActionReason` | 実行理由カテゴリ |
| detail | `ActionDetail?` | 自由記述（内部参照用、公開しない） |
| correlationId | `ActionId?` | ペア参照（例: `hide` ↔ `unhide`） |
| executedAt | `Timestamp` | 実行時刻 |

**追記型・イミュータブル**。生成後の属性変更は認めない（不変条件 I2 / I3）。

## 4. 値オブジェクト

### 4.1 `ActionKind`

v4 §5.4 の運営対応手段に対応。

| 値 | 意味 | Photobook への指示 |
|----|-----|--------------------|
| `hide` | 一時非表示の開始 | `Photobook.hide()` |
| `unhide` | 一時非表示の解除 | `Photobook.unhide()` |
| `soft_delete` | 論理削除 | `Photobook.softDelete()` |
| `restore` | 論理削除からの復元 | `Photobook.restore()`（保持期間内） |
| `purge` | 物理削除 | `Photobook.purge()` |
| `reissue_manage_url` | 管理URL失効と新規発行 | `Photobook.reissueManageUrl()` |

### 4.2 `ActionReason`

v4 §3.6（通報カテゴリ、`minor_safety_concern` 追加）、§7.3（権利侵害）、§7.4（未成年保護）に準拠。

| 値 | 意味 | 対応する通報カテゴリ |
|----|-----|---------------------|
| `report_based_harassment` | 通報起因: 嫌がらせ・晒し | `harassment_or_doxxing` |
| `report_based_unauthorized_repost` | 通報起因: 無断転載 | `unauthorized_repost` |
| `report_based_sensitive_violation` | 通報起因: センシティブ違反 | `sensitive_flag_missing` |
| `report_based_minor_related` | 通報起因: 未成年関連の性的表現・年齢に関する問題（優先対応） | **`minor_safety_concern`**（v4 §3.6 で新設） |
| `report_based_subject_removal` | 通報起因: 被写体からの削除依頼 | `subject_removal_request` |
| `rights_claim` | 権利者からの削除申立て（Report 非経由も可） | - |
| `creator_request_manage_url_reissue` | 作成者からの管理URL再発行申請 | - |
| `erroneous_action_correction` | 誤操作の補正（restore / unhide 等） | - |
| `policy_violation_other` | その他の規約違反 | `other` 等 |

`report_based_minor_related` は v4 §7.4 未成年保護の観点から**最優先対応**の対象。通知レベルは `ReportSubmitted` 発火時点で上がる（Outbox ハンドラ側で分岐）。

### 4.3 `OperatorLabel`

| 属性 | 制約 |
|------|-----|
| 文字列 | 運営内で合意された識別子（`ops-1`, `legal-team` など）。**個人情報は含めない** |

MVP では運営が少人数のため単純ラベル。将来運営アカウント機能ができれば `OperatorId` へ置換可能。

### 4.4 `ActionDetail`

| 属性 | 制約 |
|------|-----|
| 文字列 | 0–2000 文字、内部参照用。作成者・閲覧者には露出しない |

### 4.5 `ActionId`

UUIDv4 または ULID。

## 5. 不変条件（Invariants）

- **I1**: `kind`, `targetPhotobookId`, `actorLabel`, `reason`, `executedAt` は必須
- **I2**: `ModerationAction` は**不変（immutable）**。作成後の属性変更不可
- **I3**: 補正は新規アクション（`unhide` / `restore` 等）として追加する。既存アクションの UPDATE / DELETE は禁止
- **I4**: `sourceReportId` が指定されている場合、該当 Report は `under_review` 以降の状態でなければならない（§9）
- **I5**: `kind == reissue_manage_url` のアクションは Photobook の `manage_url_token_version` をインクリメントする
- **I6（v4）**: **同一トランザクション原則**。Photobook 状態変更 / `moderation_actions` 追記 / Report 状態更新（該当時）/ `outbox_events` 追記は単一 DB トランザクションで commit される
- **I7（v4）**: `unhide` / `restore` では Report の状態を**自動変更してはならない**（§9）

## 6. ドメインサービス

### 6.1 `ModerationActionExecutor`（v4 責務拡張）

運営の操作要求を受け、以下を**原子的に1トランザクション内で**実行する。

**入力**: `kind`, `targetPhotobookId`, `reason`, `actorLabel`, `sourceReportId?`, `detail?`, `correlationId?`

**実行ステップ**:

1. トランザクション開始
2. Photobook 集約をロード（行ロック取得）
3. `kind` に応じた Photobook ドメインメソッド呼び出し（`hide()` / `unhide()` / `softDelete()` / `restore()` / `purge()` / `reissueManageUrl()`）
4. `ModerationAction` を生成し `moderation_actions` に INSERT
5. `sourceReportId` がある場合、`kind` に応じて Report 状態更新（§9 のルール表に従う）
6. `kind` に対応する Outbox イベントを生成し `outbox_events` に INSERT（§7）
7. commit

上記のいずれかが失敗したら全体をロールバック。中途半端な状態（Photobook だけ更新されて Moderation が無い / Report だけ更新されて Outbox が無い等）を許容しない。

**契約**: Photobook 集約の `hide()` / `softDelete()` / `purge()` / `reissueManageUrl()` 等の運営用メソッドは、**この Executor 経由でのみ呼ばれる**という規約とする。

### 6.2 `RestoreWindowPolicy`

v4 §3.4「論理削除後の一定の保持期間内は、運営判断で復元可能」。

```
canRestore(photobook) -> bool
```

- `status == deleted` かつ `deletedAt + retentionPeriod > now()` なら復元可
- 保持期間内であれば複数回 `restore` 可能（v4 §3.4 Slug 復元ルール）
- `status == purged` は**復元不可**
- 保持期間超過の deleted は物理削除候補

### 6.3 `ManageUrlReissuePolicy`

v4 §3.4 付随ルール: 漏洩・不正利用が疑われる場合に運営が失効と新規発行を行う。

```
canReissue(photobook, reason) -> bool
```

- `reason == creator_request_manage_url_reissue` または `policy_violation_other` 等で許可
- 再発行後、旧トークンは即座に無効化（Photobook の `manage_url_token_version` を増分）
- 新 URL の控え送付は `ManageUrlReissued` イベント経由（新 `ManageUrlDelivery` を別ユースケースで作成）

## 7. ドメインイベント（Outbox 連携）

v4 §6.11, §2.8 に従い、**全 ModerationAction は対応する Outbox イベントを発火する**（P0-13）。発火は `ModerationActionExecutor` 内で `outbox_events` への INSERT として行われる（同一トランザクション）。

### ActionKind → Outbox event_type マッピング

| ActionKind | event_type | 主要な副作用（Outbox ハンドラ側） |
|------------|-----------|------------------------------|
| `hide` | `PhotobookHidden` | 公開ページ CDN 無効化、OGP キャッシュ無効化 |
| `unhide` | `PhotobookUnhidden` | 公開ページ CDN 無効化、必要なら OGP 再生成 |
| `soft_delete` | `PhotobookSoftDeleted` | 公開ページ CDN 無効化、OGP キャッシュ無効化 |
| `restore` | `PhotobookRestored` | 公開ページ CDN 無効化、OGP 再生成 |
| `purge` | `PhotobookPurged` | 所属 Image / OGP 画像の物理削除、CDN 削除 |
| `reissue_manage_url` | `ManageUrlReissued` | 旧 URL 無効化、新 URL 控え送付（任意） |

イベントの payload には最低限 `aggregate_id=photobookId`, `action_id`, `source_report_id?`, `reason`, `actor_label` を含める（監査上の追跡のため）。

## 8. 実行フロー（同一トランザクション）

すべての運営アクションは `ModerationActionExecutor` を通る。以下に代表例を示す。

### 8.1 `hide`（通報起点、v4 §8.3 相当）

```
[tx start]
 1. photobook = repo.lock(photobookId)
 2. photobook.hide()                                  # hidden_by_operator=true に
 3. insert moderation_actions(
      kind='hide', targetPhotobookId, sourceReportId,
      actorLabel, reason='report_based_*', detail?, executedAt=now()
    )
 4. if sourceReportId is not null:
      update reports
        set status='resolved_action_taken',
            resolved_action_id=<new action id>,
            resolved_at=now()
        where id = sourceReportId
 5. insert outbox_events(
      aggregate_type='photobook',
      aggregate_id=photobookId,
      event_type='PhotobookHidden',
      payload_json={action_id, source_report_id?, reason, actor_label, ...}
    )
[tx commit]
```

### 8.2 `soft_delete`（通報起点、v4 §8.4 相当）

```
[tx start]
 1. photobook = repo.lock(photobookId)
 2. photobook.softDelete()                            # status='deleted', deleted_at=now()
 3. insert moderation_actions(kind='soft_delete', ...)
 4. if sourceReportId is not null:
      update reports set status='resolved_action_taken', resolved_action_id=...
 5. insert outbox_events(event_type='PhotobookSoftDeleted', ...)
[tx commit]
```

### 8.3 `purge`

```
[tx start]
 1. photobook = repo.lock(photobookId)
 2. photobook.purge()                                 # status='purged', purged_at=now()
 3. insert moderation_actions(kind='purge', ...)
 4. if sourceReportId is not null:
      update reports set status='resolved_action_taken', resolved_action_id=...
 5. insert outbox_events(event_type='PhotobookPurged', ...)
[tx commit]
```

後続で Outbox ワーカーが所属 Image と OGP 画像の物理削除、CDN 削除を実行する（冪等）。

### 8.4 `unhide`（Report 自動更新なし）

```
[tx start]
 1. photobook = repo.lock(photobookId)
 2. photobook.unhide()                                # hidden_by_operator=false
 3. insert moderation_actions(
      kind='unhide', correlationId=<元のhideのid>?,
      reason='erroneous_action_correction' 等
    )
 4. # Report 状態は自動変更しない（I7）
 5. insert outbox_events(event_type='PhotobookUnhidden', ...)
[tx commit]
```

### 8.5 `restore`（Report 自動更新なし）

```
[tx start]
 1. photobook = repo.lock(photobookId)
 2. RestoreWindowPolicy.canRestore(photobook) を検証
 3. photobook.restore()                               # status='published' に復帰
 4. insert moderation_actions(kind='restore', correlationId=<元のsoft_deleteのid>?)
 5. # Report 状態は自動変更しない（I7）
 6. insert outbox_events(event_type='PhotobookRestored', ...)
[tx commit]
```

### 8.6 `reissue_manage_url`

```
[tx start]
 1. photobook = repo.lock(photobookId)
 2. ManageUrlReissuePolicy.canReissue(photobook, reason) を検証
 3. photobook.reissueManageUrl()                      # manage_url_token_version++、新トークン採番
 4. insert moderation_actions(kind='reissue_manage_url', ...)
 5. insert outbox_events(event_type='ManageUrlReissued', ...)
[tx commit]
```

### 8.7 unhide / restore で Report を自動変更しない理由

`unhide` と `restore` は意味が多義的である。

- 以前の通報対応の誤りだった（= Report も巻き戻したい）
- 状況変化（作成者が問題箇所を修正した等）で復旧してよくなった（= Report は解決済みのままでよい）
- 権利者との合意が成立し復旧した（= Report は状況に応じて運営判断）

自動で Report を `under_review` 等に巻き戻してしまうと、上記の多義性が失われ、監査上も誤解を生む。そのため **`unhide` / `restore` では Report を自動変更せず、必要な場合は運営が別途 Report 側の状態を操作する**運用とする（I7）。

## 9. Report 状態更新ルール（kind 別）

`sourceReportId` が指定されたときの Report 状態遷移。

| ActionKind | sourceReportId あり | sourceReportId なし |
|------------|--------------------|--------------------|
| `hide` | `reports.status = 'resolved_action_taken'`（対応済み） | 何もしない |
| `soft_delete` | `reports.status = 'resolved_action_taken'` | 何もしない |
| `purge` | `reports.status = 'resolved_action_taken'` | 何もしない |
| `unhide` | **自動変更しない**（I7） | 何もしない |
| `restore` | **自動変更しない**（I7） | 何もしない |
| `reissue_manage_url` | **自動変更しない**（通報由来の操作ではないため） | 何もしない |

Report 更新時は `resolved_action_id`（この ModerationAction の id）と `resolved_at=now()` も同時に記録する。Report 集約のスキーマに準拠。

## 10. 操作（集約ルートのファクトリ）

`ModerationAction` は不変のため、操作は「新規作成」のみ。実際の呼び出しは `ModerationActionExecutor` 経由に集約される。

| ファクトリ | 説明 |
|-----------|-----|
| `Hide(photobookId, reason, operator, sourceReportId?, detail?)` | 一時非表示アクション |
| `Unhide(photobookId, reason, operator, correlationId?, detail?)` | 解除アクション |
| `SoftDelete(photobookId, reason, operator, sourceReportId?, detail?)` | 論理削除アクション |
| `Restore(photobookId, reason, operator, correlationId?, detail?)` | 復元アクション |
| `Purge(photobookId, reason, operator, sourceReportId?, detail?)` | 物理削除アクション |
| `ReissueManageUrl(photobookId, reason, operator, detail?)` | 管理URL再発行アクション |

## 11. 他集約との境界

| 関係先 | 参照方向 | 内容 |
|--------|---------|-----|
| Photobook | Moderation → Photobook | `targetPhotobookId` による参照。状態変更は Photobook のドメインメソッド経由（Executor が呼ぶ） |
| Report | Moderation → Report | `sourceReportId` による参照。§9 のルールで状態を同一 TX 更新 |
| Image | Moderation → Image（間接） | `purge` 実行後、Outbox ハンドラ（`PhotobookPurged` 処理）が `owner_photobook_id` 一致 Image を削除 |
| ManageUrlDelivery | Moderation → ManageUrlDelivery（間接） | `reissue_manage_url` → `ManageUrlReissued` イベント → 後続ユースケースで新 `ManageUrlDelivery` 作成 |
| Outbox | Moderation → Outbox | 各 ActionKind に対応する `outbox_events` 行を同一 TX で INSERT |

## 12. MVP での実装方針（v4 §5.4）

v4 §5.4「MVP では、運営は DB 直接操作ではなく運営スクリプト経由で対応する」に従う。

- 運営 UI は作らない
- 運営スクリプト（CLI）を `scripts/ops/` 配下に配置し、内部で `ModerationActionExecutor` を呼ぶ
  - 例: `ops-hide.sh <photobook_id> --reason <r> --operator <l> [--report <rid>]`
- スクリプト実行はアプリケーション層（ユースケース）を必ず通すことで、§1 の同一トランザクション原則を自動的に保証する
- DB を直接 UPDATE する運用は禁止（監査証跡が壊れ、Outbox が発火されない）
- Reconcile スクリプト群（`scripts/ops/reconcile/`）は別途提供され、Outbox failed や不整合を検出・修復する（横断設計書参照）

## 13. 権利侵害申立てへの対応（v4 §7.3）

v4 §7.3「明らかな権利侵害と判断された場合、作成者への事前通知なく措置を講じる場合がある」に対応。

- `ActionReason.rights_claim` で実行
- Report 経由でない直接の申立ても受理（`sourceReportId` は空）
- この場合でも §1 の同一トランザクション原則は変わらず、`outbox_events` には必ず対応イベントを記録する
- 権利者への返答は運営が手動で行う（MVP）。将来的に `rights_claim_cases` テーブルの追加を検討

## v3 からの主な変更点

- **§1 / I6（新規）**: `ModerationActionExecutor` の責務を拡張し、**Photobook 状態変更 + `moderation_actions` 追記 + Report 状態更新 + `outbox_events` INSERT を同一 DB トランザクション**で原子的に実行することを最上位不変条件として明示（v4 P0-5）。
