# Image 集約 ドメイン設計

> 上流: [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) §3.10, §6.14, §3.7 / [Outbox設計](../../cross-cutting/outbox.md) / [OGP生成設計](../../cross-cutting/ogp-generation.md) / [Reconcileスクリプト設計](../../cross-cutting/reconcile-scripts.md)
>
> v4 の主題: (1) 画像の所有モデルを `owner_photobook_id` 方式に切り替え（`reference_count` 廃止）、(2) `failed` 状態の追加、(3) 画像アップロード安全検証の厳格化、(4) EXIF/XMP/IPTC 等メタデータの原則全除去。

---

## 1. 集約の責務

作成者がアップロードした**画像ファイルのライフサイクル**を管理する。v4 では「画像は1つの Photobook に所有される」という明確な所有モデルに切り替え、参照カウント方式は廃止する。

- 画像の永続化と識別
- 閲覧用派生画像（リサイズ、サムネイル、OGP 用等）の生成と管理
- アップロード時の安全検証（MIME / マジックナンバー / 実デコード、サイズ上限、デコンプレッションボム対策）
- EXIF / XMP / IPTC 等の埋め込みメタデータの原則全除去
- 公開URLからの配信を成立させるためのストレージキー管理
- 所有者 Photobook の物理削除および draft 期限切れに伴う画像の連鎖削除
- 処理失敗（`failed`）時の状態管理と長期放置対策

### 担わない責務

- アップロードUIや入力バリデーション表示 → Photobook 集約／UI 層の責務（UI 側で枚数・サイズ上限を表示）
- 画像の加工・編集・切り抜き → MVP スコープ外（v4 §3.10）
- 具体ストレージサービス選定 → 技術設計（ADR）
- OGP 画像の状態管理そのもの → OGP 集約（`photobook_ogp_images`）が担う。Image はその実体のみ保持する
- Photobook との所属関係の決定 → Photobook 集約側で `owner_photobook_id` を割り当てる

---

## 2. 集約境界と集約ルート

### 集約ルート: `Image`

1つのアップロード原本とその派生バリアントをまとめる。**1 Image は必ず 1 つの Photobook に所有される**。

```
Image (集約ルート)
├── ownerPhotobookId : PhotobookId     ← v4 新規。所有する Photobook を指す
├── usageKind        : ImageUsageKind  ← v4 新規。photo / cover / ogp
└── ImageVariant [1..*]
       kind ∈ { original?, display, thumbnail, ogp }
```

- Photobook 集約からは `ImageId` で参照される
- Image → Photobook 方向の参照は `ownerPhotobookId`（所有関係）
- 1画像が複数 Photobook から共有されることは MVP では起きない想定（v4 §3.10 付随ルール）
- `ownerPhotobookId` は Image の生成時に決まり、基本的に変更しない

## 4. 値オブジェクト

| VO | 値・形式 | 制約・業務ルール |
|----|---------|----------------|
| `ImageId` | UUIDv4 or ULID | 生成時不変 |
| `PhotobookId` | UUIDv4 or ULID | Photobook 集約から参照 |
| `ImageUsageKind` | `photo` / `cover` / `ogp` | **v4 新規**。Phase 2 で `creator_avatar` 追加可能（MVP では追加しない） <!-- 付録C P0-9 --> |
| `ImageFormat` | `jpg` / `png` / `webp` / `heic` | v4 §3.10。受付対応は JPG / PNG / WebP / HEIC、**正規化形式は JPG または WebP のみ**。HEIC は内部で変換 |
| `ImageDimensions` | `{ width: int, height: int }` | 1 以上。最大長辺 8192px、最大ピクセル数 40MP |

| `MimeType` | 文字列 | ホワイトリスト照合（`image/jpeg` / `image/png` / `image/webp`） |
| `StorageKey` | 文字列 | R2 バケット内パス。命名規則は `photobooks/{photobook_id}/images/{image_id}/{variant}/{random}.{ext}`（OGP は `photobooks/{photobook_id}/ogp/{ogp_id}/{random}.png`）。`{random}` は 12B 暗号論的乱数を base64url 化 (参照: ADR-0005) <!-- 付録C P0-12 --> |
| `VariantKind` | `original` / `display` / `thumbnail` / `ogp` | `display` / `thumbnail` は `photo` / `cover` で必須。`ogp` は `usageKind=ogp` のみ。**MVP では `original` は保持しない**（U9） |
| `ImageStatus` | `uploading` / `processing` / `available` / `failed` / `deleted` / `purged` | **v4 で `failed` を追加**。`rejected` は採用しない <!-- 付録C P0-10 --> |
| `FailureReason` | 12 種固定（§11.2 表参照） | **v4 / 付録C P0-11 で確定**。CHECK 制約で値域固定（データモデル設計 §3 参照） <!-- 付録C P0-11 --> |

### 4.1 `ImageUsageKind` 詳細（v4 新規）

| 値 | 用途 | 備考 |
|----|----|------|
| `photo` | Photobook の本文ページに使われる写真 | MVP 標準 |
| `cover` | Photobook の表紙に使われる代表画像 | 表紙は `photobooks` にインライン化（v4 §2.1） |
| `ogp` | OGP 共有画像の実体 | `photobook_ogp_images.image_id` から参照される |
| （`creator_avatar`） | 作成者アイコン | **Phase 2 以降**（v4 §5.2） |

### 4.2 `normalizedFormat` の扱い（v4 P1-3）

- 受付対応: JPG / PNG / WebP / HEIC
- 正規化形式: **JPG または WebP のみ**
- 変換ルール:
  - 入力 JPG → JPG または WebP
  - 入力 PNG → 透過ありなら WebP、透過なしなら JPG または WebP
  - 入力 WebP → WebP（アニメーション WebP は拒否）
  - 入力 HEIC → JPG または WebP（HEIC を `normalizedFormat` に保持しない）
- `original` variant として PNG 原寸を保持する運用は許容（ADR で判断）。ただし配信向けの `display` / `thumbnail` / `ogp` variant は JPG または WebP に限る

## 5. 不変条件（Invariants）

- **I1**: `sourceFormat` がサポート対象外（SVG、アニメーション WebP / APNG、HEIC 以外の非 JPG/PNG/WebP、その他未対応形式）のとき、`status=available` へ遷移しない
- **I2**: `status == available` のとき、`variants` は `usageKind` に応じた必要バリアントを含む
  - `usageKind=photo` / `cover` → 最低 `display` と `thumbnail`
  - `usageKind=ogp` → `ogp` バリアント
- **I3**: `status == available` のとき、`metadataStrippedAt` は設定済み（EXIF / XMP / IPTC 除去が完了している）
- **I4**: `originalByteSize` は 10MB（UsageLimit の制限値）以内。超過は受付時点で拒否され Image は生成されない
- **I5**: `originalDimensions` の長辺は 8192px 以下、総ピクセル数は 40MP 以下
- **I6**: `ownerPhotobookId` は生成時に必ず設定され、基本的に不変（運用上の移動は想定しない）
- **I7**: `status == deleted` の Image は配信停止される
- **I8**: HEIC は必ず内部で JPG または WebP に変換され、`normalizedFormat` に HEIC は入らない
- **I9**: **v4 新規**。`status == failed` の Image は、**公開中の Photobook に紐付けてはならない**（参照側の Photobook が公開状態なら Photobook 側の不変条件で拒否／非表示化）
- **I10**: **v4 新規**。`status == failed` の Image は `failedAt` と `failureReason` が設定済み
- **I11**: `status ∈ { uploading, processing, failed }` の Image は公開ページから配信されない
- **I12**: アニメーション WebP / APNG / SVG は受付時点で拒否され、Image は生成されないか、生成されたとしても即 `failed` に遷移する

## 6. ライフサイクル

v4 で `failed` 状態が追加され、処理失敗ケースを明示的に表現する。

```
      upload受付
         │
         ▼
    ┌──────────┐
    │ uploading│  ← バイナリ受信中
    └────┬─────┘
         │ 受信完了
         ▼
    ┌──────────┐
    │processing│  ← マジックナンバー検証、HEIC変換、
    │          │    EXIF/XMP/IPTC除去、リサイズ/サムネ生成
    └────┬─────┘
         │
    ┌────┴──────────────┐
    │                   │
    │ 成功              │ 失敗
    ▼                   ▼
┌──────────┐       ┌──────────┐
│available │       │  failed  │ ← failedAt, failureReason 記録
└────┬─────┘       └────┬─────┘
     │                  │ 長期（例: 7日）経過で Reconcile 対象
     │ 所有 Photobook   │ または owner_photobook_id の Photobook が purged/draft期限切れ
     │ の purge / draft │
     │ 期限切れ / 孤児  │
     ▼                  ▼
┌──────────────────────────┐
│         deleted          │  ← 論理削除
└────┬─────────────────────┘
     │ 保持期間経過（例: 30日）
     ▼
┌──────────┐
│  purged  │  ← ストレージからも物理削除、行削除
└──────────┘
```

### 6.1 `failed` に遷移する代表ケース（v4 P0-2, P0-3） <!-- 付録C P0-10, P0-11 -->

- デコードに失敗した（破損ファイル、不正な構造、デコンプレッションボム疑い）→ `failure_reason = decode_failed`
- 申告 MIME / マジックナンバーで判定した結果がサポート外形式 → `failure_reason = unsupported_format`
- SVG が混入している → `failure_reason = svg_not_allowed`
- アニメーション WebP / APNG が混入している → `failure_reason = animated_image_not_allowed`
- 寸法上限（長辺 8192px / 40MP）超過 → `failure_reason = dimensions_too_large`
- 申告サイズと実サイズが不一致 → `failure_reason = size_mismatch`
- 10MB 超過（実測） → `failure_reason = file_too_large`
- EXIF / XMP / IPTC 除去処理に失敗した → `failure_reason = exif_strip_failed`
- HEIC → JPG / WebP 変換に失敗した → `failure_reason = heic_conversion_failed`
- リサイズ / サムネ / OGP variant 生成に失敗した → `failure_reason = variant_generation_failed`
- complete 時に R2 オブジェクトが存在しない → `failure_reason = object_not_found`
- 上記に分類できない内部エラー → `failure_reason = unknown`

### 6.2 `failed` からの回収

- `failed` 状態の Image は公開に使われない
- `failed_at` から一定期間（例: **7 日**）経過した Image は `image_references.sh` Reconcile で `deleted` へ遷移
- 所有 Photobook が draft 期限切れ / purge になった場合は `failed` を経由せず直接 `deleted` へ

## 7. ドメインサービス

### 7.1 `ImageIngestionService`

アップロードされたバイナリを受け、正規化・メタデータ除去・派生生成を行う。v4 で責務が厳格化された。

**入力**: 生バイナリ + 宣言 MIME タイプ + `ownerPhotobookId` + `usageKind`

**出力**: `Image`（`available` または `failed` 状態）

**責務（v4 で強化）**:

1. **宣言 MIME に頼らないフォーマット判定**
   - 拡張子・宣言 MIME は信用せず、マジックナンバーと実デコードで判定する
   - SVG は拒否、アニメーション WebP / APNG は拒否
2. **サイズ・寸法の事前検証**
   - `originalByteSize` ≤ 10MB
   - 長辺 ≤ 8192px、総ピクセル数 ≤ 40MP
   - デコンプレッションボム対策: 展開前にヘッダ情報で推定ピクセル数を確認し、上限超過が見込まれるものは拒否
3. **HEIC → JPG / WebP 変換**
4. **EXIF / XMP / IPTC 等の埋め込みメタデータを原則全除去**（詳細は §12）
5. **派生バリアント生成**
   - `usageKind=photo` / `cover` → `display`（例: 長辺 1600px）と `thumbnail`（例: 長辺 400px）
   - `usageKind=ogp` → `ogp`（1200 × 630px）
6. **ストレージへの配置**（推測困難なキー）
7. **成功時は `status=available`、失敗時は `status=failed` に遷移**し、`failedAt` / `failureReason` を設定

### 7.2 `OrphanImageCollector`

v4 では `reference_count` ベースではなく、**所有 Photobook の状態**をもとに孤児を判定する（`image_references.sh` Reconcile として実装）。

- 所有 Photobook が存在しない Image
- 所有 Photobook が `purged` 状態の Image
- 所有 Photobook が `deleted` かつ保持期間経過した Image
- 所有 Photobook が draft 期限切れの Image
- `status=failed` で `failed_at` から 7 日経過した Image
- `status=available` だが Photobook 本文から参照されていない Image

これらを `deleted` に遷移させる。

### 7.3 `ImagePurger`

`deleted` 状態で保持期間（例: 30 日）経過した Image について、`image_variants` のストレージ実体を削除し、`images.status` を `purged` にした後、行を削除する。CDN キャッシュパージもここで実施する。

## 8. ドメインイベント

| イベント | 発火タイミング | 主な用途 |
|---------|--------------|---------|
| `ImageAvailable` | `processing → available` | Photobook 集約で参照可能化（Photobook 側で公開成立条件を満たす） |
| `ImageIngestionFailed` | `processing → failed` | 作成者への再アップロード要求（UI 層へ通知）、OGP なら `photobook_ogp_images` を `failed` へ |
| `ImageDeleted` | `→ deleted` | CDN キャッシュ無効化の予約 |
| `ImagePurged` | `→ purged`（行削除直前） | ストレージからの実削除完了確認、CDN 最終パージ |

Image 集約のイベントは基本的に Photobook 集約や OGP 集約から駆動される。また、Photobook 側の Outbox イベント（`PhotobookPurged` 等）のハンドラが、所属 Image の物理削除ジョブを投入する（v4 §6.11, Outbox 設計書 §4.1）。

## 9. 操作（集約ルートに生えるメソッド）

| 操作 | 説明 |
|------|-----|
| `NewIngestion(ownerPhotobookId, usageKind, sourceFormat, declaredMime)` | アップロード受付。`status=uploading` |
| `attachBytes(bytes)` | バイナリを付与し `processing` に遷移 |
| `completeProcessing(normalizedFormat, variants, metadataStrippedAt)` | 派生生成完了、`available` に遷移。`metadataStrippedAt` を設定 |
| `fail(reason)` | **v4 新規**。処理失敗、`failed` に遷移。`failedAt=now()`, `failureReason=reason` を設定 |
| `softDelete()` | 論理削除（`deleted`）。所有 Photobook の purge、draft 期限切れ、孤児回収から呼ばれる |
| `purge()` | 物理削除（`purged`）。ストレージ削除後に呼ばれ、行削除へ進む |

v3 にあった `incrementReference() / decrementReference()` は **v4 で廃止**。所有関係は `ownerPhotobookId` 1 つで表現する。

所有権の移動（ownerPhotobookId の変更）は MVP では行わない。Photobook を分割・結合する機能は MVP にないため、移動メソッドも用意しない。

## 10. 他集約との境界

| 関係先 | 参照方向 | 内容 |
|--------|---------|-----|
| Photobook | Image → Photobook（`ownerPhotobookId`） | Image は Photobook に所有される。Photobook 側も本文の `photo_image_id` / `cover_image_id` で Image を参照するが、これは参照であって所有ではない |
| OGP（`photobook_ogp_images`） | OGP → Image（`image_id`） | OGP 集約は状態のみ管理し、実体は Image を参照。Image 側は `usageKind=ogp` として存在 |
| UsageLimit | サービス層で参照 | アップロード前にサイズ・回数上限を検証 |
| Outbox | Photobook 経由で間接 | `PhotobookPurged` ハンドラが所属 Image の物理削除ジョブを投入 |
| Moderation | Moderation → Photobook → Image | Photobook の hide / soft_delete / purge に連動 |

### 10.1 Photobook との境界（v4）

- Photobook 側は本文で Image を**参照**する（`photo_image_id` 等）
- Image 側は `ownerPhotobookId` で所有を表現する
- 通常はこの両者は一致する（Photobook A が参照する Image の `ownerPhotobookId` は A）
- 不一致は `photobook_image_integrity.sh` Reconcile で検出される

### 10.2 OGP との境界（v4）

- OGP の**状態**は `photobook_ogp_images` が単独で管理（pending / generated / failed / fallback / stale）
- OGP の**実体**は Image 集約が保持（`usageKind=ogp`）
- OGP 再生成時、旧 Image は所有関係上は同じ Photobook に属するまま孤児化し、Reconcile で回収される

## 11. 画像安全検証ルール詳細（v4 P0-12）

`ImageIngestionService` の検証段階で以下を順に実施する。いずれかを突破できなかった時点で `failed` に遷移させる（ただし受付拒否が望ましいものは Image を生成しない）。

### 11.1 受付段階（Image 生成前に拒否）

| チェック | 判定基準 | 不合格時の対応 |
|---------|---------|--------------|
| 宣言 MIME ホワイトリスト | `image/jpeg` / `image/png` / `image/webp` / `image/heic` / `image/heif` | HTTP 415 で受付拒否 |
| 拡張子は判定に使わない | 宣言 MIME と実マジックナンバーで判定 | （参考情報としてのみ使用） |
| バイトサイズ | ≤ 10MB | HTTP 413 で受付拒否 |

### 11.2 処理段階（Image 生成後、`processing` 中に検査） <!-- 付録C P0-10, P0-11 -->

| チェック | 判定基準 | 不合格時の対応 |
|---------|---------|--------------|
| マジックナンバー | 実バイナリ先頭から JPG / PNG / WebP / HEIC を判定 | `failed(unsupported_format)` |
| SVG 検出 | XML / `<svg` ヘッダを検出 | `failed(svg_not_allowed)` |
| アニメーション検出 | WebP の ANIM チャンク / APNG の acTL チャンク | `failed(animated_image_not_allowed)` |
| デコード可否 | 実デコードライブラリで読み込める。デコンプレッションボム疑いを含む | `failed(decode_failed)` |
| 寸法上限 | 長辺 ≤ 8192px、総ピクセル数 ≤ 40MP | `failed(dimensions_too_large)` |
| サイズ整合 | 申告 Content-Length と実サイズの一致 | `failed(size_mismatch)` |
| 実サイズ上限 | 実測 ≤ 10MB | `failed(file_too_large)` |
| HEIC 変換 | 成功すること | `failed(heic_conversion_failed)` |
| メタデータ除去 | EXIF / XMP / IPTC 除去が成功すること | `failed(exif_strip_failed)` |
| リサイズ / サムネ / OGP 生成 | 成功すること | `failed(variant_generation_failed)` |
| R2 オブジェクト存在確認（complete 時） | HeadObject が成功すること | `failed(object_not_found)` |
| 上記に分類できない | 内部例外発生 | `failed(unknown)` |

`failureReason` には上記の括弧内の分類キーを入れる。実装側で機械的に集計・Reconcile 判断できる。語彙は v4 §3.10 / 付録C P0-11 / ADR-0005 と一致する 12 種で固定し、データモデル設計 §3 の CHECK 制約で値域を縛る。 (参照: v4 §3.10 / ADR-0005)

## 12. EXIF / XMP / IPTC 除去ポリシー（v4 P0-14）

### 12.1 基本方針

**公開配信用画像から、埋め込みメタデータは原則として全除去する**。

- 除去対象（v4 §3.10）:
  - GPS 位置情報
  - デバイスシリアル番号、カメラ機種
  - PC ユーザー名、ローカルファイルパス
  - 撮影日時
  - Author / Artist / Photographer / Copyright 等のクレジット情報
  - その他ベンダー固有拡張
- **保持してよい情報はない**（Orientation は画像処理内部で解釈してピクセル配置に反映させた上で除去）
- 除去処理が失敗した場合は `failed(exif_strip_failed)` とし、`available` へ遷移させない

### 12.2 クレジット情報の扱い

v4 では Author / Artist / Photographer / Copyright も画像メタデータとしては**残さない**。クレジット表示が必要な場合は、**Photobook のメタ情報として明示入力させる**（ページメタの Photographer 項目等）。

理由:

- PC ユーザー名や編集ソフト由来の意図しない文字列が公開されるリスクを防ぐ
- クレジットは作成者が明示的にコントロールできるべき情報である
- 画像バイナリから分離することで、誤公開時の影響範囲を限定できる

### 12.3 原本の扱い

`original` バリアントを保持する場合も、メタデータ除去は適用する。削除済みメタデータを作成者に返す機能は MVP で提供しない（業務上の必要性がなく、情報管理リスクが増えるため）。

## 14. 次集約への引き継ぎ事項

本集約の P0 反映を後続設計に渡すための申し送り。付録C 番号を追跡 ID として用いる。

### 14.1 OGP 横断（`docs/design/aggregates/outbox/` または `cross-cutting/ogp-generation.md` で扱う）

- **OGP 状態管理は `photobook_ogp_images` 側、実ファイルは Image 集約**（`usage_kind = 'ogp'`）。責務境界の詳細はデータモデル設計 §7 参照 <!-- 付録C P0-13 -->
- 失敗理由カラムは Image 側 `failure_reason`（画像処理失敗）と OGP 側 `ogp_failure_reason`（OGP 生成プロセス失敗）で**別語彙体系**として管理する（混同防止）
- `photobook_ogp_images.image_id` は NULL 許容（生成前 / 失敗時 / fallback 使用時）
- `PhotobookPublished` / `PhotobookUpdated` Outbox イベントの OGP 再生成ハンドラが、Image 集約に新しい `usage_kind = 'ogp'` Image を作成し、生成完了後に `photobook_ogp_images.image_id` を更新する

### 14.2 Photobook publish / update UseCase

- cover image の `usage_kind = 'cover'` 設定は Photobook 側 UseCase の責務
- **`Image.owner_photobook_id` は publish 時に変更しない**（Draft と Published は同一 Photobook ID） (参照: v4 §6.14) <!-- 付録C P0-8 -->
- Photobook purge 時、`owner_photobook_id` 一致の Image を削除対象にする（Outbox `PhotobookPurged` ハンドラ経由）

### 14.3 Image upload-intent UseCase

- upload-intent は upload-verification 機構（`docs/design/auth/upload-verification/`）の `validateAndConsume(rawToken, photobookId)` を呼び出してから実行する
- Image レコード作成時の既定: `status='uploading'`, `usage_kind='photo'`, `owner_photobook_id=対象Photobook`
- `storage_key` は `photobooks/{photobook_id}/images/{image_id}/{variant}/{random}.{ext}` 形式で生成（U7 / U8 参照） <!-- 付録C P0-12 -->
- complete API で R2 オブジェクト存在確認後、`outbox_events` に `ImageIngestionRequested` を INSERT（Image 状態更新と同一 TX）

### 14.4 image-processor ワーカー

- `ImageIngestionRequested` を購読し、§11 の安全検証ルールを実行
- 成功で `completeProcessing()` → `available` 遷移、失敗で `fail(failure_reason)` → `failed` 遷移
- variant 生成は §4.2 / データモデル §4.2 の表に従う（**MVP では `original` を生成・保持しない**、U9）
- HEIC 変換に libheif を使用（cgo 必要、ADR-0005 / ADR-0001 のコンテナ構築方針）

### 14.5 Reconcile / GC

- **`image_references.sh`**: 孤児 Image 検出（所有 Photobook 不存在 / `purged` / `deleted` 期限経過 / draft 期限切れ / 本文未参照） → `deleted` 遷移
- **`failed` 7 日経過**: `image_references.sh` で `deleted` 遷移
- **`deleted` 30 日経過**: 物理削除バッチで R2 オブジェクト削除 → `purged` → 行削除
- **CDN パージ**: `purged` 遷移時に Outbox 経由でキャッシュ無効化
- 各 reconciler の自動 / 手動の区別は v4 §6.16 / `aggregates/outbox/` 設計（後続）に従う

### 14.6 ManageUrlDelivery / Moderation との関係

- 本集約から直接の依存はない
- ただし運営の `purge` 操作（Moderation 経由）が Outbox 経由で本集約に削除を波及させる
