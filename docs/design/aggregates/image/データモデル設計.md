# Image 集約 データモデル設計

> 上流: [Image ドメイン設計](./ドメイン設計.md) / [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) §3.10, §6.14, §3.7 / [Outbox設計](../../cross-cutting/outbox.md) / [Reconcileスクリプト設計](../../cross-cutting/reconcile-scripts.md)

---

## 1. 扱うデータ

- 画像原本のメタデータ（サイズ、形式、メタデータ除去状態）
- 派生画像（`display` / `thumbnail` / `ogp` / `original?`）のメタデータとストレージキー
- 所有 Photobook（`owner_photobook_id`）と用途（`usage_kind`）
- ライフサイクル状態（`uploading` / `processing` / `available` / `failed` / `deleted` / `purged`）
- 処理失敗の記録（`failed_at` / `failure_reason`）

画像バイナリ自体はオブジェクトストレージ（S3 / R2 等）に置き、DB にはキーのみ保存する。

**v4 での主要変更**: `reference_count` カラムを廃止し、`owner_photobook_id` + `usage_kind` に置き換え。`status` に `failed` を追加し、`failed_at` / `failure_reason` を追加。

## 3. `images`（v4）

| カラム | 型 | NULL | 既定 | 制約・備考 |
|-------|-----|------|------|----------|
| `id` | `uuid` | NOT NULL | `gen_random_uuid()` | PK |
| `owner_photobook_id` | `uuid` | **NOT NULL** | - | **v4 新規**。所有する Photobook の ID。FK → `photobooks.id` は `ON DELETE RESTRICT`（§3.2 参照）。publish 時も変更しない <!-- 付録C P0-8 --> |
| `usage_kind` | `text` | **NOT NULL** | - | **v4 新規**。CHECK: `photo / cover / ogp`（Phase 2 で `creator_avatar` 追加可能） <!-- 付録C P0-9, P0-13 --> |
| `source_format` | `text` | NOT NULL | - | CHECK: `jpg / png / webp / heic` |
| `normalized_format` | `text` | NULL | - | CHECK: `jpg / webp`。`status=available` 以降で必須 |
| `original_width` | `int` | NULL | - | `status=available` 以降で必須。`<= 8192` |
| `original_height` | `int` | NULL | - | 同上 |
| `original_byte_size` | `bigint` | NULL | - | `<= 10485760`（10MB）。`status=available` 以降で必須 |
| `metadata_stripped_at` | `timestamptz` | NULL | - | EXIF / XMP / IPTC 除去完了時刻。`status=available` で必須 |
| `status` | `text` | NOT NULL | `'uploading'` | **v4 更新**。CHECK: `uploading / processing / available / failed / deleted / purged`。`rejected` は採用しない <!-- 付録C P0-10 --> |
| `uploaded_at` | `timestamptz` | NOT NULL | `now()` | |
| `available_at` | `timestamptz` | NULL | - | `status=available` 遷移時に設定 |
| `failed_at` | `timestamptz` | NULL | - | **v4 新規**。`status=failed` 遷移時に設定 |
| `failure_reason` | `text` | NULL | - | **v4 新規**。失敗分類キー。**CHECK 制約で 12 種固定**（§3.3 参照）。`status=failed` 遷移時に必須 <!-- 付録C P0-10, P0-11 --> |
| `deleted_at` | `timestamptz` | NULL | - | `status=deleted` で必須 |

**v4 で削除されたカラム**:

- `reference_count`（`int`）— 所有モデルへの移行により廃止。以後、参照関係は `owner_photobook_id` 単独で表現する。EXIF 除去完了時刻は `exif_removed_at` → `metadata_stripped_at` に名称変更（XMP / IPTC も対象に含まれるため）

### 3.0 `failure_reason` の CHECK 制約（v4 / 付録C P0-11）

`failure_reason` は次の 12 種に値域を固定する。CHECK 制約で DB 側で縛り、未知の値が混入しないようにする。 (参照: v4 §3.10 / ADR-0005) <!-- 付録C P0-11 -->

| 値 | 意味 |
|----|----|
| `file_too_large` | 申告または実サイズが 10MB 超 |
| `size_mismatch` | 申告 Content-Length と実サイズが不一致 |
| `unsupported_format` | 受け入れ対象外の形式（GIF / BMP / TIFF 等、または申告 MIME とマジックナンバー不一致） |
| `svg_not_allowed` | SVG が混入 |
| `animated_image_not_allowed` | アニメーション WebP / APNG |
| `dimensions_too_large` | 長辺 8192px 超 または 40MP 超 |
| `decode_failed` | 実デコード失敗（破損ファイル、デコンプレッションボム疑い含む） |
| `exif_strip_failed` | EXIF / XMP / IPTC 除去失敗 |
| `heic_conversion_failed` | HEIC からの変換失敗 |
| `variant_generation_failed` | display / thumbnail / ogp variant 生成失敗 |
| `object_not_found` | complete 時に R2 オブジェクトが存在しない |
| `unknown` | 上記に分類できない内部エラー |

**CHECK 制約**:

```sql
CHECK (
  failure_reason IS NULL OR failure_reason IN (
    'file_too_large', 'size_mismatch', 'unsupported_format',
    'svg_not_allowed', 'animated_image_not_allowed',
    'dimensions_too_large', 'decode_failed', 'exif_strip_failed',
    'heic_conversion_failed', 'variant_generation_failed',
    'object_not_found', 'unknown'
  )
)
CHECK (status != 'failed' OR failure_reason IS NOT NULL)
```

旧語彙からの移行マッピング（v3 → v4）:

| 旧（v3 例示） | 新（v4 / 付録C） |
|--------------|-------------------|
| `magic_mismatch` | `unsupported_format`（拡張子 / MIME / マジック不一致を統合） |
| `decompression_bomb_suspected` | `decode_failed`（v4 §3.10 で「破損・ボム疑い含む」と統合） |
| `metadata_strip_failed` | `exif_strip_failed` |
| `animation_rejected` | `animated_image_not_allowed` |
| `svg_rejected` | `svg_not_allowed` |
| `too_large_dimensions` | `dimensions_too_large` |

### 3.1 索引

| 索引 | カラム | 用途 |
|------|-------|------|
| PK | `id` | |
| INDEX | `owner_photobook_id` | Photobook の purge / draft 期限切れ時に所属 Image を高速抽出 |
| INDEX | `owner_photobook_id, usage_kind` | 用途別抽出（例: cover 画像取得） |
| INDEX | `status, deleted_at` | 物理削除バッチ |
| INDEX | `status, failed_at` WHERE `status = 'failed'` | `image_references.sh` で長期 failed を抽出 |
| INDEX | `status, available_at` WHERE `status = 'available'` | 孤児検出の補助 |

### 3.2 `owner_photobook_id` の削除連鎖方針

v4 では **`ON DELETE CASCADE` にはしない**。方針は以下。

- `owner_photobook_id` は `photobooks.id` への FK を張るが **`ON DELETE RESTRICT`** とする
- Photobook 側の purge ハンドラ（Outbox の `PhotobookPurged` ハンドラ）が、所属 Image（`owner_photobook_id` 一致）を明示的に `deleted` → `purged` に遷移させてから、Photobook 本体を削除する
- これによりストレージ実体削除・CDN パージ・監査ログ記録を漏らさない
- Reconcile（`image_references.sh`）で取りこぼしを追いかける多層防御構成にする

**選定理由**:

- CASCADE だと DB 行は消えるがストレージ実体と CDN キャッシュが残る
- 明示削除フロー＋Reconcile の二重化が v4 §6.14, §6.12 の方針に沿う

## 4. `image_variants`

派生画像を個別行で管理する。MVP では 1 画像あたり最大 4 行（`original` / `display` / `thumbnail` / `ogp`）。用途により実際に作成される variant の組は異なる。

| カラム | 型 | NULL | 既定 | 制約・備考 |
|-------|-----|------|------|----------|
| `id` | `uuid` | NOT NULL | `gen_random_uuid()` | PK |
| `image_id` | `uuid` | NOT NULL | - | FK → `images.id` ON DELETE CASCADE |
| `kind` | `text` | NOT NULL | - | CHECK: `original / display / thumbnail / ogp` |
| `storage_key` | `text` | NOT NULL | - | ストレージ上のキー（バケット名は含めない） |
| `width` | `int` | NOT NULL | - | `>= 1` |
| `height` | `int` | NOT NULL | - | `>= 1` |
| `byte_size` | `bigint` | NOT NULL | - | `>= 1` |
| `mime_type` | `text` | NOT NULL | - | **v4 更新**。CHECK: `image/jpeg / image/png / image/webp` のみ（HEIC は内部で JPG / WebP に変換済みのため variant には出現しない） |
| `created_at` | `timestamptz` | NOT NULL | `now()` | |

### 4.1 索引

| 索引 | カラム | 用途 |
|------|-------|------|
| PK | `id` | |
| UNIQUE | `(image_id, kind)` | 1 画像 1 種 1 行 |
| INDEX | `storage_key` | ストレージキーからの逆引き（運用保守） |

### 4.2 用途別の variant 構成 <!-- 付録C P0-12, P0-13 -->

| `images.usage_kind` | 生成される variant | 備考 |
|---------------------|-------------------|------|
| `photo` | `display`（必須）, `thumbnail`（必須） | 配信用の正規化済みのみ |
| `cover` | `display`（必須）, `thumbnail`（必須） | photo と同構成。表紙は `photobooks.cover_image_id` から参照される |
| `ogp` | `ogp`（必須、1200 × 630 PNG） | OGP 単一 variant。`display` / `thumbnail` は生成しない |

**MVP 基本: `original` variant は保持しない**（U9）。

理由:
- EXIF / XMP / IPTC の完全除去を徹底しやすい（原本がないため漏れ得ない）
- ストレージコストを抑えられる
- 物理削除・GC が単純になる（`display` / `thumbnail` / `ogp` のみ追えばよい）
- 公開配信用途には `display` / `thumbnail` で足りる

Phase 2 以降: 高画質ダウンロードや物理フォトブック出力の要件が出た場合に再検討。

`image_variants.kind` は `original / display / thumbnail / ogp` を CHECK で許容しているが、**MVP では `original` 行を作らない**運用とする（CHECK は将来のために残す）。

### 4.3 重要な設計判断

- **`storage_key` は URL ではなく相対パス的なキー**: ストレージプロバイダ切替を容易にするため、配信時に CDN ベース URL を結合
- **`storage_key` 命名規則は ADR-0005 に統一**（§6 参照） <!-- 付録C P0-12 -->
- **MVP は `original` variant を保持しない**（§4.2 / U9）
- **`mime_type` から PNG 以外の HEIC 系を排除**: 変換済み形式のみが variant に入る
- **派生生成失敗時は `image_variants` に行を残さず、`images.status` を `failed` に遷移**（v3 の「processing に留める」は v4 で廃止）

## 5. 保持期間と削除ポリシー

v4 §6.14 に従う。`reference_count` ベースではなく、**所有 Photobook の状態**と**Image 自身の状態**で判定する。

| 状態・条件 | 判定基準 | アクション |
|-----------|---------|----------|
| `available` で所有 Photobook が存在かつ `published` / `unlisted` / `private` | 通常利用中 | 削除しない |
| `available` で所有 Photobook が `draft` | 編集中 | 削除しない（`draft_expires_at` までは保持） |
| `available` で所有 Photobook が `draft` かつ `draft_expires_at` 経過 | draft 期限切れ | `draft_expired.sh` が Photobook 削除と同時に Image を `deleted` に |
| `available` で所有 Photobook が `deleted`（論理削除）かつ保持期間経過 | Photobook 論理削除期間経過 | `image_references.sh` が Image を `deleted` に |
| `available` で所有 Photobook が `purged` | Photobook 物理削除 | Photobook の purge ハンドラが Image を `deleted` に（Reconcile が取りこぼしを回収） |
| `available` だが Photobook 本文から参照されていない | 孤児 | `image_references.sh` が `deleted` に |
| `failed` かつ `failed_at` から 7 日経過 | 長期 failed（v4 新規） | `image_references.sh` が `deleted` に |
| `deleted` かつ `deleted_at` から 30 日経過 | 保持期間経過 | 物理削除バッチが `image_variants` のストレージ実体削除 → `images.status='purged'` → 行削除 |
| `purged` | - | 行削除（監査ログに痕跡のみ） |

### 5.1 物理削除バッチ

- 日次バッチで `status='deleted' AND deleted_at < now() - INTERVAL '30 days'` を抽出
- ストレージから各 `image_variants.storage_key` を削除
- ストレージ削除成功後に `images.status='purged'` へ遷移、直後に行を削除
- **CDN キャッシュの無効化**: `purged` 遷移時に CDN 無効化をトリガ（Outbox / Reconcile 経由）

### 5.2 Photobook からの削除連鎖（v4 更新）

```
photobook の purge / draft 期限切れ
  ↓
Outbox イベント（PhotobookPurged）or draft_expired.sh
  ↓
ハンドラが owner_photobook_id 一致の Image を列挙
  ↓
各 Image を status='deleted' に遷移
  ↓
30日後の物理削除バッチで image_variants のストレージ実体削除
  ↓
images.status='purged' → 行削除
  ↓
CDN キャッシュパージ
```

v3 にあった「`reference_count` のデクリメント → `reference_count == 0` で孤児 TTL 待ち」のフローは v4 で**廃止**。所有 Photobook の状態変更が Image 削除の単一のトリガになる。

### 5.3 `failed` 状態の長期放置対策

`failed` 状態の Image はストレージにバイナリを持たない or 中途半端な状態のことが多いが、DB 行が残る。長期放置を防ぐため以下を行う:

- `image_references.sh --execute` を日次実行し、`failed_at` から 7 日経過した Image を `deleted` に遷移
- 監視ダッシュボードで `status='failed'` の件数を追い、急増時は `failure_reason` 分布を確認

## 6. ストレージ構成 <!-- 付録C P0-12 -->

ADR-0005 で確定した命名規則に統一する。R2（Cloudflare）を前提とするが、S3 互換 API のため他プロバイダにも移植可能。

### 6.1 storage_key 命名規則

| バリアント | キー形式 | 備考 |
|-----------|---------|------|
| `display` | `photobooks/{photobook_id}/images/{image_id}/display/{random}.webp` | 長辺 1600px 目安、WebP 配信 |
| `thumbnail` | `photobooks/{photobook_id}/images/{image_id}/thumbnail/{random}.webp` | 長辺 400px 目安、WebP 配信 |
| `ogp` | `photobooks/{photobook_id}/ogp/{ogp_id}/{random}.png` | 1200 × 630px、PNG 形式（ADR-0005 確定）。`{ogp_id}` は `photobook_ogp_images.id` |
| ~~`original`~~ | （**MVP では保持しない**、U9） | Phase 2 以降で再検討時に `photobooks/{photobook_id}/images/{image_id}/original/{random}.{ext}` 形式 |

### 6.2 `{random}` の生成方針（U7）

- 12 バイトの暗号論的乱数を base64url 化（`crypto/rand`、Go 標準ライブラリ）
- 衝突時は Application 層で再生成して再試行（`{image_id}` が UUIDv7 で一意なので `{random}` 衝突確率は実質ゼロ、保険的措置）
- 詳細実装は M3 / M6 で確定

### 6.3 path 統一の意図

- `photobooks/{photobook_id}/...` プレフィックスにより、Photobook 単位での bucket 列挙・一括削除（purge）が容易
- `cover` 用 storage_key は別ルート（`photobooks/{photobook_id}/cover/...`）を**採用しない**（U8）。cover 画像も Image 集約の実体として扱うため、通常の `photobooks/{photobook_id}/images/{image_id}/...` で統一する
- OGP のみ別ルート `photobooks/{photobook_id}/ogp/{ogp_id}/...` とするのは、`photobook_ogp_images` 側のライフサイクル（再生成・stale・version 管理）と紐付けるため

### 6.4 配信

- 配信は CDN ベース URL を `storage_key` に結合して行う
- キャッシュキーに `?v={photobook_ogp_images.version}` を付ける等のバージョニングは OGP 設計書 §7.3 参照
- 配信用 URL を直接外部に露出させず、認可チェック後に署名付き URL or CDN Worker 経由配信（M6 で確定）

## 7. `photobook_ogp_images` との責務分離 <!-- 付録C P0-13 -->

OGP は **状態管理を `photobook_ogp_images` 側、実ファイルを `images` 側** に分離する。両テーブルのカラム責務を明確化することで、混同による事故を防ぐ。 (参照: v4 §6.17 / ADR-0005)

### 7.1 責務分離表

| 関心事 | `images`（Image 集約） | `photobook_ogp_images`（OGP 状態管理、横断テーブル） |
|--------|----------------------|------------------------------------------------|
| 実ファイル参照 | ✅ `storage_key`（`image_variants` 配下） | ❌（`image_id` で参照のみ） |
| 物理ファイルの形式・サイズ・寸法 | ✅ `normalized_format` / `original_byte_size` / `original_width` / `original_height` / `image_variants` 配下 | ❌ |
| Image ライフサイクル | ✅ `status`（`uploading` 〜 `purged`） | ❌ |
| Image 処理失敗理由 | ✅ `failure_reason`（12 種、画像処理失敗） | ❌ |
| OGP 生成状態 | ❌ | ✅ `pending` / `generated` / `failed` / `fallback` / `stale` |
| OGP version（再生成連動） | ❌ | ✅ `version` |
| OGP 生成成功時刻 | ❌（`available_at` は Image の生成完了時刻） | ✅ `generated_at` |
| OGP 生成失敗理由 | ❌ | ✅ `ogp_failure_reason`（**Image の `failure_reason` とは別語彙**） |
| Photobook 公開連動 | ❌ | ✅ `PhotobookPublished` / `PhotobookUpdated` ハンドラから状態遷移 |
| 新規生成時の image_id | ❌ | ✅ `image_id` NULL 許容（生成前 / 生成失敗時 / fallback 使用時） |
| `usage_kind` | ✅ `'ogp'` | ❌ |
| `owner_photobook_id` | ✅（OGP Image も Photobook に所有される） | ❌（`photobook_id` を別カラムで持つ） |

### 7.2 失敗理由カラムの命名分離

混同防止のため、失敗理由カラム名は次のように分ける:

- **`images.failure_reason`**: Image 処理（マジックナンバー / デコード / EXIF 除去 / variant 生成）の失敗理由。語彙は §3.0 の 12 種
- **`photobook_ogp_images.ogp_failure_reason`**: OGP 生成プロセス（Photobook メタ取得 / レンダリング / R2 PUT 等）の失敗理由。Image 層の失敗とは別の事象を扱うため別語彙

両者は CHECK 制約の値域も別管理する（`photobook_ogp_images` 側の語彙は OGP 設計書側で確定）。

### 7.3 image_id NULL 許容の意味

`photobook_ogp_images.image_id` は NULL 許容で運用する:

- **生成前**: Photobook publish 直後、OGP Image がまだ作成されていない（`status='pending'`、`image_id=NULL`）
- **生成失敗時**: image-processor で variant 生成が失敗（`status='failed'`、`image_id` は NULL のまま、または途中まで作成された Image の id を保持して後で `deleted` 化）
- **fallback 使用時**: 既定の OGP 画像で代替（`status='fallback'`、`image_id=NULL`）

生成成功時のみ `image_id` を Image 集約の新しい行に紐付ける。再生成時は新しい Image を作り、`image_id` を更新、旧 Image は `owner_photobook_id` 一致のまま孤児化 → Reconcile で回収される（v4 §6.14 / Image ドメイン §10.2）。

## 10. 未解決事項

`auth/` 配下の U1〜U6 とは別管理。Image 集約固有の未解決事項を U7 以降に割り当てる。

### U7: storage_key の `{random}` 実装

**MVP 基本案**: 12 バイトの暗号論的乱数（`crypto/rand`）を base64url 化する。

- `{image_id}` が UUIDv7 で一意なため、`{random}` 衝突確率は実質ゼロ
- 衝突時は Application 層で再生成して再試行（保険的措置、リトライ上限 3 回）
- 詳細実装は M3 / M6 で確定

### U8: cover 用 storage_key path

**MVP 基本案**: cover も Image 実体として扱うため、通常の Image と同じパス形式を使う。

```
photobooks/{photobook_id}/images/{image_id}/{variant}/{random}.{ext}
```

別ルート `photobooks/{photobook_id}/cover/...` は **採用しない**。

将来、cover 専用の生成・差し替え・キャッシュ管理が必要になった場合に再検討（Phase 2 以降）。

### U9: `original` variant の保持判断

**MVP 基本案**: `original` variant は保持しない。

理由:
- EXIF / XMP / IPTC の完全除去を徹底しやすい（原本がないため漏れ得ない）
- ストレージコストを抑えられる
- 物理削除・GC が単純になる
- 公開配信用途には `display` / `thumbnail` で足りる

Phase 2 以降:
- 高画質ダウンロード機能や物理フォトブック出力（印刷発注）が必要になった場合に再検討
- `image_variants.kind` の CHECK は `original` を許容したまま残し、運用で「行を作らない」だけの方針にする（将来の DDL 変更を最小化）

## v3 からの主な変更点

| 項目 | v3 | v4 |
|------|----|----|
| 所有モデル | `reference_count`（`int`, `CHECK >= 0`） | **`owner_photobook_id`（`uuid`, NOT NULL, FK RESTRICT）+ `usage_kind`（`text`, CHECK `photo/cover/ogp`）** |
| FK 削除ルール | なし（論理のみ） | **`owner_photobook_id` を FK、`ON DELETE RESTRICT`。Photobook purge ハンドラで明示削除 + Reconcile で追跡** |
| `status` CHECK | `uploading / processing / available / deleted / purged` | **左記 + `failed`** |
| 失敗記録 | なし | **`failed_at`, `failure_reason` カラム追加** |
| メタデータ除去 | `exif_removed_at` | **`metadata_stripped_at` に改名**（EXIF / XMP / IPTC すべて対象） |
| 変換形式制約 | `normalized_format` CHECK `jpg / webp`（記述薄） | **明示: PNG は入力のみ、出力は JPG / WebP。HEIC は内部変換後のため出力には現れない** |
| `image_variants.kind` | `original / display / thumbnail` | **左記 + `ogp`** |
| `image_variants.mime_type` | 記述あるが制約弱 | **CHECK: `image/jpeg / image/png / image/webp` のみ** |
| 索引 | `status, deleted_at`、`(status, available_at) WHERE reference_count=0 AND status='available'` | **`owner_photobook_id`、`(owner_photobook_id, usage_kind)`、`(status, failed_at) WHERE status='failed'`、`(status, deleted_at)` を採用** |
| 孤児検出 | `reference_count=0` かつ TTL | **所有 Photobook 状態ベース + `failed` 長期化検出**（`image_references.sh`） |
| 削除連鎖 | `photo_image_id` の外れ → `reference_count` 減 → TTL → `deleted` | **Photobook purge / draft 期限切れで `owner_photobook_id` 一致 Image を直接 `deleted` に**。Outbox + Reconcile の二重化 |
| 物理削除時の CDN 無効化 | 言及あり | **Outbox / Reconcile 経由の明示的トリガに整理** |
| 用途区分 | なし | **`usage_kind` で `photo` / `cover` / `ogp` を明示**（Phase 2 で `creator_avatar` 追加可能） |
