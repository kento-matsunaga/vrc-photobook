- 通報レコード（対象 Photobook ID + snapshot、理由、詳細、連絡先、対応状況、IP ハッシュ）

### 3.1 `target_photobook_id` の FK 方針（v4 確定） <!-- 付録C P0-22 -->

v3 では「要検討（候補B推奨: ON DELETE RESTRICT）」としていたが、**v4 では FK 制約を付けない（= 候補 C）** に確定した（v4 §3.6, P0-11）。

- 理由 1: Photobook の物理削除（purge）後も、監査証跡として最低限の文脈（slug / title / creator_display_name）を snapshot で残したい
- 理由 2: FK RESTRICT では「未対応通報があると Photobook を物理削除できない」という業務制約になり、悪質な通報者が purge を阻止できてしまう
- 理由 3: Photobook 側のライフサイクル（draft → published → deleted → purged）と Report 側のライフサイクルを疎結合にしたい

アプリ層で以下を担保する。

- `Submit` ユースケース時、`target_photobook_id` に対応する Photobook の **現在の slug / title / creator_display_name** を取得し、snapshot カラムに必ず埋める
- Photobook が存在しない（purged 済み）場合の通報は受け付けない（ユースケース層でエラー）
- Photobook purge 後も Report 行はそのまま残る（snapshot を通じて文脈復元可能）

### 3.2 `reason` の CHECK 制約

```sql
CHECK (reason IN (
  'subject_removal_request',
  'unauthorized_repost',
  'sensitive_flag_missing',
  'harassment_or_doxxing',
  'minor_safety_concern',
  'other'
))
```

v4 で `minor_safety_concern` を追加。既存行への影響はなし（新規追加のため）。

### 3.3 `source_ip_hash` のソルトポリシー

v4 P1-4 に従い、**UsageLimit の `actor_hash` と同一ソルトを共有**する。

- ソルトは秘密管理（環境変数 / Secret Manager）
- ソルトは `version` を持ち、ローテーション可能
- ローテーション時は長期追跡性が失われることを許容（v4 §3.7）
- 同一作成元からの「大量投稿 + 大量通報」の相関検出に必要

## 4. 索引

| 索引 | カラム | 用途 |
|------|-------|------|
| PK | `id` | |
| INDEX | `target_photobook_id` | Photobook 別通報集計 |
| INDEX | `status, submitted_at` | 未対応通報一覧 |
| INDEX | `reason, submitted_at` | カテゴリ別抽出 |
| **部分 INDEX** | `submitted_at` WHERE `reason = 'minor_safety_concern' AND status IN ('submitted', 'under_review')` | **v4 新規**: 未成年関連通報の最優先抽出（v4 §3.6, §7.4） |
| INDEX | `source_ip_hash, submitted_at` | 同一 IP からの大量通報検出（UsageLimit と連携） |
| INDEX | `resolved_by_moderation_action_id` WHERE `resolved_by_moderation_action_id IS NOT NULL` | Moderation 処置起因の通報逆引き |

## 5. 保持期間と削除ポリシー

v3 方針を維持（v4 §7.2 に基づく「用途完了後、適切な期間内に削除」）。

| 対象 | 保持期間 | 備考 |
|------|---------|-----|
| 通報レコード本体 | 対応完了から **90 日** | 運用調整値。snapshot により Photobook purge 後も文脈復元可能 |
| `reporter_contact` | 対応完了から **30 日** 以内に NULL 化 | 最小保持（v4 §7.2） |
| `source_ip_hash` | **90 日** で NULL 化 | UsageLimit と同期。ソルトローテーション時は自動的に追跡不能化 |
| `target_*_snapshot` | 通報レコードと同じ | Photobook 削除後の証跡として保持 |

### 実装メモ

- バッチで `status IN ('resolved_action_taken', 'resolved_no_action', 'dismissed') AND resolved_at < now() - INTERVAL '30 days'` の `reporter_contact` を NULL 化
- 通報レコード自体の削除は 90 日後。ただし運用実績蓄積のため、完全匿名化（個人特定情報を NULL 化）する運用でも代替可能
- snapshot カラムは個人特定情報を含まない想定だが、`target_creator_display_name_snapshot` にハンドル名が残るため、90 日後の削除/匿名化対象に含める

## 6. 運営 UI との関係（Outbox 経由の運営通知）

v4 §5.4「MVP では専用 UI を持たず、メールや通知で対応」に従い、運営通知は **Outbox 経由**で送出する。

- 通報作成時に `ReportSubmitted` イベントを `outbox_events` に INSERT（同一トランザクション）
- Outbox ワーカーが `notify_operators` ハンドラを実行
  - 通常: メール / Slack 通知
  - `reason == minor_safety_concern` の場合: **通知レベルを上げる**（別チャネル、即時通知等）
- 運営は DB 直接クエリ or `scripts/ops/reports/` 配下の運営スクリプトで対応
- `status` の更新は、Moderation 処置経由（自動、同一 TX）または運営スクリプト経由（手動）

Phase 2 以降で運営 UI を整備する想定。

## v3 からの主な変更点

- **P0-11（FK 方針確定）**: v3 で「要検討（候補B推奨）」としていた `target_photobook_id` の FK を、v4 では **FK 制約を付けない** に確定。アプリ層で存在確認。Photobook purge 後も通報証跡を残す設計へ
- **P0-11（snapshot 追加）**: `target_public_url_snapshot` / `target_title_snapshot` / `target_creator_display_name_snapshot` の 3 カラムを新規追加。作成時必須埋め（display_name のみ NULL 可）
- **P0-10（reason 拡張）**: `reason` CHECK 制約に `minor_safety_concern` を追加（5 種 → 6 種）。年齢・センシティブ関連の独立カテゴリ
- **P0-5（Moderation 連携）**: `resolved_by_moderation_action_id` カラムを新規追加。Moderation 処置で終端化された場合の参照（FK は張らない）
- **P0-13（Outbox 連携）**: 運営通知は `outbox_events` 経由に確定。`ReportSubmitted` を同一トランザクションで INSERT
- **P1-4（IP ハッシュソルト共有）**: `source_ip_hash` は UsageLimit の `actor_hash` と同じソルトポリシー（version 管理、ローテーション可）を共有する旨を明記
- **優先抽出用 INDEX**: `minor_safety_concern` かつ未対応の通報を高速抽出する部分 INDEX を追加
- **運営 UI 節の再構成**: v3 §5 を「Outbox 経由の運営通知」に再整理し、`minor_safety_concern` の通知レベル引き上げを明記
