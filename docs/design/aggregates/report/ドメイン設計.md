> 上流: [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) §3.6, §3.7, §5.4, §6.11, §7.2, §7.3, §7.4

> 関連: [Outbox 設計書](../../cross-cutting/outbox.md) / [Reconcile スクリプト設計書](../../cross-cutting/reconcile-scripts.md)

## 1. 集約の責務

閲覧者による**通報の受付と記録**を担う。不適切なフォトブック、無断掲載、センシティブ違反、未成年を連想させる不適切表現等、運営が対応するための入力源となる。

### 担う責務

- 閲覧者からの通報を受け付け、永続化する
- 通報理由のカテゴリ（`ReportReason`）を管理する
- 自由記述・連絡先の任意入力を保持する
- 通報対象 Photobook の特定情報を **ID 参照 + snapshot** として保持する（v4 P0-11）
- 運営による対応状況（`submitted` / `under_review` / `resolved_*` / `dismissed`）を追跡する
- `ReportSubmitted` イベントを **Outbox 経由で** 発火する（v4 P0-13）
- Moderation からの処置完了通知を受け、`resolved_action_taken` への遷移を行う（v4 §5.4, P0-5）

### 担わない責務

- 通報に基づく実際の削除・一時非表示・復元等の操作 → **Moderation 集約** が担当
- 運営通知の実送信（メール/Slack 等）→ Outbox ワーカー + 通知アダプタ
- 通報対象 Photobook の存在検証 → アプリケーション層（FK はもたない）
- 運営向け管理 UI → MVP スコープ外（v4 §5.2, §5.4）

## 2. 集約境界と集約ルート

### 集約ルート: `Report`

1件の通報 = 1つの集約。小さな集約として独立させる。

```
Report (集約ルート)
  ├── reason: ReportReason
  ├── detail: ReportDetail (任意)
  ├── reporterContact: ReporterContact (任意)
  ├── status: ReportStatus
  ├── targetPhotobookId: PhotobookId        # ID のみ参照（FK なし）
  └── targetSnapshot: TargetSnapshot         # v4 新規
        ├── publicUrlSlug
        ├── title
        └── creatorDisplayName
```

Photobook 集約とは **ID 参照のみ** で結合する（DB FK は張らない）。Photobook の物理削除（purge）後も通報証跡を残すため、作成時点のスナップショットを保持する。

## 4. 値オブジェクト

### 4.1 `ReportReason`

v4 §3.6 に従い、VRC コミュニティの実情に即した 6 カテゴリ（v3 の 5 種 + `minor_safety_concern`）。

| 値 | 表示名 | 想定ケース |
|----|-------|----------|
| `subject_removal_request` | 被写体として削除希望 | 写っている本人からの削除依頼 |
| `unauthorized_repost` | 無断転載の可能性 | 他人の写真が無断で掲載されている |
| `sensitive_flag_missing` | センシティブ設定の不足 | センシティブ申告なしで不適切表現 |
| `harassment_or_doxxing` | 嫌がらせ・晒し | 個人攻撃、晒し |
| `minor_safety_concern` | 年齢・センシティブに関する問題（未成年を連想させる不適切表現） | 未成年を連想させる性的表現、その他未成年保護関連（v4 新規、§7.4） |
| `other` | その他 | 上記以外 |

**業務ルール**:

- カテゴリは v4 §3.6 に準拠し固定。拡張時は業務知識定義書を先に改版する
- `minor_safety_concern` は **運営通知レベルを上げる**（Outbox ハンドラ側で判定）。`other` の派生ではなく独立カテゴリとして扱う

### 4.2 `ReportDetail`

| 属性 | 制約 |
|------|-----|
| 自由記述 | 0–2000 文字、任意、改行可、コントロール文字禁止 |

### 4.3 `ReporterContact`

| 属性 | 制約 |
|------|-----|
| 連絡先文字列 | 0–200 文字、任意。メールアドレス / X ID / 自由形式 |

**不変条件（運用）**:

- 通報対応以外の用途に用いない（v4 §3.6, §7.2）
- 対応完了後、一定期間経過で削除対象（保持期間はデータモデル設計 §5 に記載）

### 4.4 `ReportStatus`

| 値 | 意味 | 遷移 |
|----|-----|-----|
| `submitted` | 通報受付直後 | → `under_review` / `resolved_action_taken`（Moderation からの直接処置） |
| `under_review` | 運営確認中 | → `resolved_action_taken` / `resolved_no_action` / `dismissed` |
| `resolved_action_taken` | 対応済み（一時非表示・削除等の処置あり） | 終端 |
| `resolved_no_action` | 確認済みだが処置不要と判断 | 終端 |
| `dismissed` | 不適切な通報として却下（スパム等） | 終端 |

### 4.5 `TargetSnapshot`（v4 新規）

通報時点の Photobook 情報のスナップショット。Photobook 物理削除後も通報証跡として必要最小限の文脈を残す。

| 属性 | 制約 |
|------|-----|
| `publicUrlSlug` | 通報時点の公開URL slug。必須（空文字不可） |
| `title` | 通報時点のタイトル。必須（空文字不可） |
| `creatorDisplayName` | 通報時点の作成者表示名。任意（匿名設定可） |

### 4.6 `ReportId`

UUIDv4 or ULID。生成時不変。

## 5. 不変条件（Invariants）

- **I1**: `targetPhotobookId` は必須。通報対象不明の通報は成立しない
- **I2**: `reason` は `ReportReason` の 6 値のいずれかで必須
- **I3**: `status == submitted` への遷移時、`submittedAt` が必ず記録される
- **I4**: `status` が終端値（`resolved_*` / `dismissed`）になった後は再オープンしない（必要なら新規通報を作る）
- **I5**: `reporterContact` が空のとき、通報者への結果通知は行われない（v4 §3.6）
- **I6**: `targetPublicUrlSnapshot` / `targetTitleSnapshot` は通報作成時に必ず埋める（Photobook 物理削除後の証跡保持のため）
- **I7**: `targetPhotobookId` に対する FK 制約は持たない。参照整合性はアプリケーション層で確認する（v4 P0-11）
- **I8**: `resolvedByModerationActionId` が設定されるのは `status == resolved_action_taken` のときに限る

## 6. ドメインサービス

### 6.1 `ReportSubmissionService`

閲覧者からの通報リクエストを受け、新規 `Report` を生成する。

- `targetPhotobookId` を受け取り、Photobook 集約から **現在の slug / title / creatorDisplayName を snapshot として取得**してから Report に埋め込む
- 削除済み Photobook（soft-deleted / purged 直前）も通報対象になりうる（運営確認のため）
- bot 対策・レート制限は UsageLimit 集約で別途検証
- `sourceIpHash` は UsageLimit の `actor_hash` と**同じソルトポリシー** で算出（v4 P1-4）
- 成功時に Report 永続化 + `ReportSubmitted` イベントの Outbox 投入を**同一トランザクション**で行う（v4 P0-13）

### 6.2 `ReportTriageService`（運営操作）

運営が通報を一覧化・優先付けするための読み取り専用サービス。

- 未対応通報（`submitted` / `under_review`）の抽出
- 同一 `targetPhotobookId` に対する通報件数の集計（多数通報の検出）
- **`reason == minor_safety_concern` の通報は最優先キューに分類**（v4 §3.6, §7.4）
- `harassment_or_doxxing` / `sensitive_flag_missing` は高優先度キューに分類

MVP では専用 UI を持たず、DB 直接クエリや運営スクリプト（`scripts/ops/`）で代替（v4 §5.4）。

### 6.3 Moderation からの処置反映（v4 新規）

運営が `ModerationActionExecutor` 経由で Photobook に対して `hide` / `soft_delete` / `purge` を実行する際、`sourceReportId` が付与されていれば、**同一トランザクション内で**対象 Report の `status` を `resolved_action_taken` に遷移させる（v4 §5.4, P0-5）。

- `unhide` / `restore` は Report 状態を自動変更しない（v4 §9 Moderation 設計）。復元後も別途 Report の扱いは運営が手動で判断する
- Report 側は `resolveWithAction(moderationActionId, note?)` という受け口メソッドを提供し、Moderation ユースケースがこれを呼ぶ

## 7. ドメインイベント

すべての外部影響を伴うイベントは **Outbox** に記録する（v4 §6.11）。

| イベント | 発火タイミング | Outbox 経由 | 主な用途 |
|---------|--------------|:----------:|---------|
| `ReportSubmitted` | 通報受付完了時 | はい | 運営通知送信（`minor_safety_concern` 等は通知レベルを上げる） |
| `ReportReviewed` | `under_review` 遷移時 | いいえ（内部記録のみ） | 運営対応の着手記録 |
| `ReportResolved` | 終端状態遷移時 | いいえ（内部記録のみ、通報者通知は別途アダプタ） | 監査ログ |

`ReportSubmitted` のペイロードには `reportId`, `targetPhotobookId`, `reason`, `hasContact`（Bool のみ）を含める。**連絡先や自由記述の本文は Outbox ペイロードに含めない**（漏洩リスク回避）。

## 8. 操作

| 操作 | 呼び出し元 | 説明 |
|------|-----------|-----|
| `Submit(targetPhotobookId, reason, detail?, contact?, targetSnapshot, sourceIpHash?)` | 閲覧者 | 新規通報作成 + `ReportSubmitted` を Outbox に enqueue |
| `startReview()` | 運営 | `submitted` → `under_review` |
| `resolveWithAction(moderationActionId?, note?)` | 運営 or Moderation ユースケース | 処置ありで対応完了（終端） |
| `resolveWithoutAction(note)` | 運営 | 処置なしで対応完了（終端） |
| `dismiss(note)` | 運営 | スパム等として却下（終端） |

## 9. 他集約との境界

| 関係先 | 参照方向 | 内容 |
|--------|---------|-----|
| Photobook | Report → Photobook | `targetPhotobookId`（ID のみ、FK なし）+ snapshot（slug/title/displayName）を保持。Photobook purge 後も Report は残存 |
| Moderation | Moderation → Report | 処置が通報起因の場合、`sourceReportId` で参照。処置完了時に Report を `resolved_action_taken` に遷移（同一 TX） |
| UsageLimit | サービス層で参照 | 通報送信のレート制限（同一 IP からの大量通報抑止）。`sourceIpHash` のソルトポリシーを共有（v4 P1-4） |
| Outbox | Report → Outbox | `ReportSubmitted` を同一トランザクションで enqueue |

## 10. 重大ケースの早期検出（v4 §3.6, §7.4）

v4 §3.6 は `minor_safety_concern` を独立カテゴリとして導入し、優先対応を明文化した。

Report 集約は **検出のヒントだけを提供** する（自動処置はしない）。

- `reason == minor_safety_concern` → **最優先**。Outbox ハンドラ側で通知レベルを引き上げ、運営に即時通知（メール + 別チャネル等）
- `reason == harassment_or_doxxing` / `sensitive_flag_missing` → 高優先度キューに分類
- 同一 `targetPhotobookId` に対する通報件数の急増を運営通知に含める
- 実際の一時非表示（hide）・削除（soft_delete / purge）は **Moderation 集約** が担う。Report は自動では処置しない

## v3 からの主な変更点

- **P0-10**: `ReportReason` に `minor_safety_concern` を追加（未成年保護関連の独立カテゴリ、表示名「年齢・センシティブに関する問題」）。`other` に混ぜない
- **P0-11**: Photobook 参照は **FK なし + snapshot 保持** に確定。v3 で「要検討（候補B推奨）」だった FK 方針を、v4 では candidate C 相当に確定。`targetPublicUrlSnapshot` / `targetTitleSnapshot` / `targetCreatorDisplayNameSnapshot` を追加
- **P0-13**: `ReportSubmitted` は **Outbox 経由** で発火することを明示。Report 永続化と Outbox INSERT は同一トランザクション
- **P0-5**: Moderation からの `hide` / `soft_delete` / `purge` 実行時に `sourceReportId` 経由で Report を `resolved_action_taken` に遷移させる受け口（`resolveWithAction`）を明記。`unhide` / `restore` では自動遷移しない
- **P1-4**: `sourceIpHash` は UsageLimit の `actor_hash` と同じソルトポリシーを共有する旨を明記
- **優先対応キュー**: `ReportTriageService` に `minor_safety_concern` の最優先分類を追加
