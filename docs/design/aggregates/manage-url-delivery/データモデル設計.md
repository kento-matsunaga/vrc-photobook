## 1. 扱うデータ

- 管理URL控えメール送付リクエスト
- 送信試行履歴（成功/失敗）
- 送信先メールアドレス（**最小保持期間のみ**、v4 §7.5）
- 送信時点の管理URLトークンバージョン（**監査・履歴用のスナップショット**）

## 3. `manage_url_deliveries`（v4）

| カラム | 型 | NULL | 既定 | 制約・備考 |
|-------|-----|------|------|----------|
| `id` | `uuid` | NOT NULL | `gen_random_uuid()` | PK |
| `photobook_id` | `uuid` | NOT NULL | - | FK → `photobooks.id` ON DELETE CASCADE |
| `recipient_email` | `text` | NULL | - | 原文。`status=sent` 成功後 + 保持期間で NULL 化 |
| `recipient_email_hash` | `bytea` | NULL | - | 重複検出用ハッシュ（任意）。ソルト付きSHA-256 |
| **`manage_url_token_version_at_send`** | `int` | NOT NULL | - | **送信時点の管理URLトークンバージョンのスナップショット**（v4 P0-6 でリネーム）。監査・問い合わせ対応用。**認可判定には使わない**（検証は Photobook 集約の `manage_url_token_version` が担当） |
| `status` | `text` | NOT NULL | `'pending'` | CHECK: `pending / sending / sent / failed_retryable / failed_permanent / expired` |
| `failure_reason` | `text` | NULL | - | 失敗時のみ。識別可能なキーワードを入れる。例: `permanent_bounce`, `provider_error`, **`expired_during_retry`**（v4 P1-7: `failed_retryable` のまま `expire_at` 到達） |
| `requested_at` | `timestamptz` | NOT NULL | `now()` | 送付リクエスト受付時刻 |
| `last_attempted_at` | `timestamptz` | NULL | - | 最終試行時刻 |
| `completed_at` | `timestamptz` | NULL | - | `sent` 遷移時 |
| `expire_at` | `timestamptz` | NOT NULL | - | 保持期間後の失効時刻（§5 参照） |

### カラム命名の変更（v3 → v4）

| v3 カラム名 | v4 カラム名 | 理由 |
|------------|------------|------|
| `manage_url_token_version` | `manage_url_token_version_at_send` | 「送信時点のスナップショット」であることを明示。Photobook 側の `manage_url_token_version`（現在有効なバージョン）と役割が異なり、混同を防ぐため（v4 P0-6） |

### 索引

| 索引 | カラム | 用途 |
|------|-------|------|
| PK | `id` | |
| INDEX | `photobook_id` | 同一Photobookへの送付履歴参照 |
| INDEX | `status, last_attempted_at` | リトライ対象抽出 |
| INDEX | `expire_at` WHERE `status IN ('sent', 'failed_retryable', 'failed_permanent')` | 失効バッチ（`failed_retryable` も expire_at 到達で `failed_permanent` に落とすため対象に含める） |

### 設計判断

- **`recipient_email` は原文で短期保持**（v4 §7.5）
  - 代替案としてハッシュ保持もあるが、リトライ時に再送できないため MVP は原文保持を採用
  - 送信完了後 **24時間** 経過で NULL 化（バッチ）
  - 失敗確定後も同様
- **`recipient_email_hash`** は同一メールアドレスからの大量送付要求検出に使う（UsageLimit集約での `manage_url_delivery` レート制限の補助）
- **Photobook削除連鎖**: `ON DELETE CASCADE` で送付履歴も連動削除
- **`manage_url_token_version_at_send` は INSERT 後は更新しない**（ドメインの I8 と整合。アプリ層で enforce、必要なら TRIGGER で防御）

## 5. 保持期間と削除ポリシー

v4 §7.5 の最小保持原則に従う。

| 対象 | 保持期間 | アクション |
|------|---------|----------|
| `recipient_email` 原文 | `status='sent'` から **24時間** / `status='failed_permanent'` 確定から **24時間** | NULL 化（バッチ） |
| `failed_retryable` 状態で `expire_at` 到達（v4 P1-7） | 即時 | `status='failed_permanent'`, `failure_reason='expired_during_retry'` に遷移 → 次回バッチで `recipient_email` を NULL 化 |
| `manage_url_deliveries` 行全体 | `expire_at` 到達後 | `status='expired'` に遷移、`recipient_email` / `provider_message_id` を NULL 化、または物理削除 |
| `manage_url_delivery_attempts` | 親行削除で CASCADE | - |

### `expire_at` の初期値

- 送付リクエスト受付時に `requested_at + INTERVAL '30 days'` を設定（運用調整）

### バッチ実装（概念）

```sql
-- (1) failed_retryable のまま expire_at 到達 → failed_permanent に確定（v4 P1-7）
UPDATE manage_url_deliveries
SET status = 'failed_permanent',
    failure_reason = 'expired_during_retry'
WHERE status = 'failed_retryable'
  AND expire_at < now();

-- (2) 24時間経過で原文削除
UPDATE manage_url_deliveries
SET recipient_email = NULL
WHERE recipient_email IS NOT NULL
  AND status IN ('sent', 'failed_permanent')
  AND COALESCE(completed_at, last_attempted_at) < now() - INTERVAL '24 hours';

-- (3) 保持期間経過で expired に遷移 → 個人情報完全除去
UPDATE manage_url_deliveries
SET status = 'expired',
    recipient_email = NULL,
    provider_message_id = NULL
WHERE status IN ('sent', 'failed_permanent')
  AND expire_at < now();

-- (4) 一定期間後に物理削除（監査不要になった expired 行）
DELETE FROM manage_url_deliveries
WHERE status = 'expired'
  AND expire_at < now() - INTERVAL '90 days';
```

バッチの実行順序は `(1) → (2) → (3) → (4)`。`(1)` を先に実行することで、`failed_retryable` でリトライ中だが保持期間を過ぎたものが `failed_permanent` になり、以降のバッチで通常ルートに乗る。

## 6. ログ・監査の配慮（v4 §6.7, §10 管理URL秘匿）

### 永続化しない（ログ・DB両方）

- 管理URL全体（特にトークン部分）
- メール本文全体
- プロバイダAPIへのリクエスト全体（request body）

### 永続化・ログ出力してよい

- `delivery_id`, `photobook_id`, `status`, `attempt_number`, `result`, `provider_message_id`
- メールアドレスは**ハッシュ化した形で**（`recipient_email_hash`）のみログ出力
- 管理URLの **Slug（識別子部分）のみ**ログ可。トークン部分は `***` などでマスク

### v4 新規: プロバイダ側ログも点検範囲

- メール送信プロバイダの送信履歴UI・APIログ・Webhookペイロードに管理URL本文が残る場合、それも漏洩経路になる
- プロバイダ選定時に本文ログの有無・保持期間を確認すること（ドメイン設計 §11 参照）
- バウンス通知に元メール本文が含まれる場合、バウンス処理側でもマスクする

## 7. 重複送信の抑止

同一 `(photobook_id, recipient_email_hash)` に対する短時間の連続リクエストは UsageLimit 集約で抑止される。

DB レベルでの UNIQUE 制約は設けない（同一アドレスへの合法的な再送信＝作成者が複数フォトブックを扱う、Moderation による再発行後の再送付、等が存在する）。

Outbox レベルでは `ManageUrlDeliveryRequested` が `delivery_id` ごとに1件生成されるので、ワーカーが冪等に処理する（[Outbox 設計書](../../cross-cutting/outbox.md) §7 Idempotency 参照）。

## v3 からの主な変更点

- **P0-6**: カラム `manage_url_token_version` を **`manage_url_token_version_at_send`** にリネーム。意味を「送信時点のスナップショット」に限定し、Photobook 側の `manage_url_token_version`（現在有効なバージョン）と区別。認可判定には使わない旨を備考に明記
- **P0-13**: `manage_url_deliveries` への `status=pending` INSERT は**同一トランザクションで `outbox_events` に `ManageUrlDeliveryRequested` を INSERT** する運用を §2, §7 に明記
- **P1-7**: `failure_reason` に `expired_during_retry` パターンを追加。バッチ順序 (1) で `failed_retryable` + `expire_at` 到達時に `failed_permanent` へ確定させるステップを §5 に追加。INDEX の `expire_at` フィルタに `failed_retryable` を追加
- **P1-10**: §6 ログ・監査の配慮を「DB/アプリログ」だけでなく「プロバイダ側ログ・Webhook・バウンス通知」まで点検範囲として拡張（詳細チェック項目はドメイン設計 §11）
- `manage_url_delivery_attempts` の `error_summary` に「管理URL本文を含めない」ことを明記
- バッチSQL例を v4 の運用フロー（失効確定 → メアドNULL化 → expired遷移 → 物理削除）で書き直し
