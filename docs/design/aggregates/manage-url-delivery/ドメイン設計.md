# ManageUrlDelivery 集約 ドメイン設計（v4）

> 上流: [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) §3.5, §6.7, §6.11, §7.5 / [Outbox 設計](../../cross-cutting/outbox.md)

---

## 1. 集約の責務

公開完了時に、作成者が**管理URLの控えをメールで受け取る**ための「送付記録」と「送信処理」を担う。

### 担う責務

- 公開完了時の管理URL控えメール送付要求を受け付ける
- 送付先メールアドレスを（最小限の期間で）保持する
- 送信時点の `manageUrlTokenVersion` を**監査・履歴のためのスナップショット**として記録する
- 送信の結果（成功/失敗/リトライ）を `DeliveryAttempt` として記録する
- プライバシーポリシーに沿ったメールアドレスの取扱いを徹底する
- 状態変更と同一トランザクションで Outbox にイベントをINSERTし、非同期ワーカーで実送信を行う

### 担わない責務

- **管理URL自体の生成・検証** → Photobook集約
  - 管理URL トークンの**検証責務は Photobook 集約**にある。ManageUrlDelivery が保持する `manageUrlTokenVersionAtSend` は「そのメールを送った時点のスナップショット」であり、現在有効なバージョンではない。認可判定には使わない。
- 管理URLの再発行 → Moderation集約（再発行後の送付依頼は再び ManageUrlDelivery に来る）
- メール送信インフラの実装（SES/SendGrid/Resend 等） → インフラアダプタ
- ブラウザローカル保存やコピー操作 → クライアント責務（v4 §3.5 維持）
- メールアドレスのログイン情報化・マーケ転用 → MVPスコープ外かつ禁止（v4 §3.5, §7.5）

## 2. 集約境界と集約ルート

### 集約ルート: `ManageUrlDelivery`

1回の送付リクエスト = 1集約。

```
ManageUrlDelivery (集約ルート)
  ├── id: DeliveryId
  ├── photobookId: PhotobookId
  ├── recipientEmail: EmailAddress?
  ├── manageUrlTokenVersionAtSend: int   # 送信時点のスナップショット（監査用）
  ├── status: DeliveryStatus
  ├── requestedAt / lastAttemptedAt / completedAt / expireAt: Timestamp
  ├── failureReason: string?
  └── attempts: DeliveryAttempt[]
```

Photobook との関係は `photobookId` のみ（ID参照）。Photobook 側の `manage_url_token_version`（現在有効なバージョン）とは**別物**である（§9 参照）。

## 3. エンティティ

### 3.1 ManageUrlDelivery（集約ルート）

| 属性 | 型 | 説明 |
|------|----|----|
| id | `DeliveryId` | 送付ID（UUID/ULID） |
| photobookId | `PhotobookId` | 対象フォトブックID（集約境界越しの参照） |
| recipientEmail | `EmailAddress?` | 送信先メールアドレス。最小保持原則に従い、完了・失敗確定・expire到達で NULL 化される |
| **manageUrlTokenVersionAtSend** | `int` | **メール送信時点の管理URLトークンバージョンのスナップショット**。監査・履歴用。検証には使わない（検証は Photobook 集約の責務） |
| status | `DeliveryStatus` | 送付状態 |
| requestedAt | `Timestamp` | 送付リクエスト受付時刻 |
| lastAttemptedAt | `Timestamp?` | 最終送信試行時刻 |
| completedAt | `Timestamp?` | 送信成功時刻 |
| failureReason | `string?` | 失敗理由（簡潔・識別可能なキーワード。例: `permanent_bounce`, `expired_during_retry`） |
| expireAt | `Timestamp` | レコード自体の失効時刻（保持期間管理） |

#### `manageUrlTokenVersionAtSend` の意味（v4 で明確化）

- **「送信時点のsnapshot」**であり、現在の有効トークンバージョンとは無関係
- Photobook 側で管理URLが再発行された場合、Photobook 側の `manageUrlTokenVersion` は進むが、**ManageUrlDelivery 側の値は送信時のまま変化しない**（履歴として固定される）
- 用途: 「この送付メールは当時のバージョンXで送られた」という監査・問い合わせ対応
- 再発行による再送付は**新しい `ManageUrlDelivery` レコード**として生まれ、新しいバージョンがそこに固定される

### 3.2 DeliveryAttempt

試行ごとの履歴（MVPでは1–3回リトライ想定）。

| 属性 | 型 | 説明 |
|------|----|----|
| attemptNumber | `int` | 試行回数（1, 2, 3...） |
| attemptedAt | `Timestamp` | 試行時刻 |
| result | `AttemptResult` | `sent` / `failed_retryable` / `failed_permanent` |
| providerMessageId | `string?` | 送信プロバイダの追跡ID（管理URL本文ではなくプロバイダ側ID） |
| errorSummary | `string?` | 失敗時の簡潔な要約（スタックトレースや管理URL本文を含めない） |

## 4. 値オブジェクト

### 4.1 `EmailAddress`

| 属性 | 制約 |
|------|-----|
| 文字列 | RFC 5322 準拠の簡易バリデーション、長さ ≤ 254 |

**永続化方針**: MVPは**原文を短期保持**し、送信成功または失敗確定から24時間で NULL 化する（v4 §7.5 最小保持原則）。リトライ時の再送のために必要最小限の期間だけ保持する。

### 4.2 `DeliveryStatus`

| 値 | 意味 | 遷移 |
|----|-----|-----|
| `pending` | 受付直後、未送信 | → `sending` |
| `sending` | 送信中 | → `sent` / `failed_retryable` / `failed_permanent` |
| `sent` | 送信成功 | → `expired` |
| `failed_retryable` | 一時的失敗、リトライ待ち | → `sending` / `failed_permanent`（上限到達 or expireAt到達） |
| `failed_permanent` | リトライ不能（永続失敗 or リトライ中にexpireAt到達） | → `expired` |
| `expired` | 保持期間経過 | 終端、レコード削除候補 |

### 4.3 `AttemptResult`

| 値 | 意味 |
|----|-----|
| `sent` | プロバイダが受理 |
| `failed_retryable` | 一時的エラー（SMTPタイムアウト、レート制限等） |
| `failed_permanent` | 永続的エラー（アドレス無効、バウンス、保持期間切れ） |

### 4.4 `DeliveryId`

UUIDv4 or ULID。

### 4.5 `ManageUrlTokenVersionAtSend`（意味の明確化）

- Photobook 集約の `ManageUrlTokenVersion`（現在有効なバージョン）とは**別の値オブジェクト/意味**
- **不変**: 一度記録したらその ManageUrlDelivery 上では変化しない
- 用途: 監査、問い合わせ対応、「送った当時のバージョン」の特定

## 5. 不変条件（Invariants）

- **I1**: `photobookId` は必須。`recipientEmail` は作成時点では必須、その後の最小保持ルールで NULL 化される
- **I2**: `recipientEmail` は**送信完了または失敗確定から24時間**で削除（NULL化）される（v4 §7.5）
- **I3**: リトライは最大 **3回** まで（MVP仮決め、運用調整可）
- **I4**: `status == sent` になったら、追加の送信試行は行われない
- **I5**: `expireAt` に達したら `status` は `expired` に遷移し、`recipientEmail` と `providerMessageId` は削除（NULL化）される
- **I6**: メール送信の内容は管理URL本体のみを含み、他の目的（マーケティング等）のコンテンツを含まない（v4 §7.5）
- **I7（v4 新規・P1-7）**: `status == failed_retryable` の状態で `expireAt` に到達した場合、**`failed_permanent` に確定**し、`failureReason` に `expired_during_retry` 相当の識別可能な理由を記録する。その後のバッチで `expired` に遷移させ、`recipientEmail` を NULL 化する
- **I8（v4 新規）**: `manageUrlTokenVersionAtSend` は**送信準備時に1回だけ設定**され、以降は不変（履歴として固定）
- **I9（v4 新規・P0-13）**: `status=pending` の新規作成時に、**同一トランザクションで Outbox に `ManageUrlDeliveryRequested` を INSERT** する。状態作成と副作用投入が分離してはならない

## 6. ドメインサービス

### 6.1 `ManageUrlDispatcher`

メール送信プロバイダへの依頼を担うドメインサービス。

```
dispatch(delivery: ManageUrlDelivery, manageUrl: string) -> DispatchResult
```

#### 責務

- Photobook集約から管理URL（Slug + token）を受け取って本文に埋める
- プロバイダ送信APIを呼び、結果を `DeliveryAttempt` として記録する
- **管理URL全体をログ・エラートラッキング・監視ログに残さない**（v4 §6.7, §10 管理URL秘匿）
- プロバイダAPIへのリクエストを構築する際、**内部ログ出力時には管理URLのトークン部分をマスクする**。Slug（識別子部分）のみをログ出力として許可する
- バウンスメール/失敗通知から管理URL本文が漏れないよう、プロバイダ側設定（本文ログ保持期間、送信履歴UIの公開範囲）を選定時に確認する（§11 参照）

### 6.2 `DeliveryRetryScheduler`

`failed_retryable` の `ManageUrlDelivery` を指数バックオフでリトライする。

- 1回目失敗 → 5分後
- 2回目失敗 → 30分後
- 3回目失敗 → `failed_permanent` 確定
- **（v4 新規・P1-7）リトライ待ちの間に `expireAt` に到達した場合、即座に `failed_permanent` に確定させ、`failureReason = expired_during_retry` を記録する**。これにより、リトライ中で保持期間を過ぎたものをメモリリーク的に抱え続けないようにする

### 6.3 `ExpiredDeliveryPurger`

保持期間を過ぎた `ManageUrlDelivery` を失効させ、個人情報を削除する。

- `expireAt < now()` かつ `status IN (sent, failed_permanent)` → `status=expired`, `recipientEmail=NULL`, `providerMessageId=NULL`
- `failed_retryable` のまま `expireAt` 到達したものは、先に `DeliveryRetryScheduler` が `failed_permanent` に確定する（§6.2 参照）ため、このサービスは `failed_permanent` 経由で `expired` に遷移させる

## 7. ドメインイベント

状態変更と同一トランザクションで Outbox に INSERT される（v4 §6.11, Outbox 設計書 §4.3）。

| イベント | 発火タイミング | Outbox event_type | 主な用途 |
|---------|--------------|-------------------|---------|
| `ManageUrlDeliveryRequested` | 公開時に送付要求が作成された（`status=pending` への遷移） | `ManageUrlDeliveryRequested` | メール送信ジョブ作成（非同期ワーカー） |
| `DeliverySent` | 送信成功（`status=sent`） | （Outbox経由不要、内部状態のみ） | UI通知（作成者への完了表示） |
| `DeliveryFailed` | 永続的失敗確定（`status=failed_permanent`） | （Outbox経由不要、内部状態のみ） | 運営通知、作成者への失敗表示 |
| `DeliveryExpired` | 保持期間経過（`status=expired`） | （Outbox経由不要） | 個人情報削除完了 |

**v4 重要**: `ManageUrlDeliveryRequested` は**必ず同一トランザクションで Outbox に INSERT** される（P0-13）。状態変更だけ commit してメール送信ジョブが作成されない、あるいはその逆を起こさない。

---

## 8. 操作

| 操作 | 説明 | 副作用 |
|------|-----|-------|
| `Request(photobookId, email, currentTokenVersion)` | 新規送付リクエスト作成、`status=pending`、`manageUrlTokenVersionAtSend = currentTokenVersion` として**送信時点のバージョンをスナップショット**する（以降不変） | 同一TXで Outbox に `ManageUrlDeliveryRequested` を INSERT |
| `markSending()` | 送信開始（`pending` → `sending`） | - |
| `recordAttempt(attempt)` | 試行結果を `attempts` に追加 | - |
| `markSent(messageId)` | 送信成功（`sending` → `sent`） | - |
| `markFailedRetryable(reason)` | 一時的失敗、リトライ待ち（`sending` → `failed_retryable`） | スケジューラが次回試行を予約 |
| `markFailedPermanent(reason)` | 永続的失敗確定（`failed_permanent`）。`expireAt` 到達で `failed_retryable` から遷移する場合は `reason=expired_during_retry` | - |
| `expire()` | 保持期間経過、`status=expired`、`recipientEmail=NULL`, `providerMessageId=NULL` | - |

`manageUrlTokenVersionAtSend` は **`Request` 時にのみ設定**され、以降は setter を提供しない（不変）。

## 9. 他集約との境界

| 関係先 | 参照方向 | 内容 |
|--------|---------|-----|
| Photobook | ManageUrlDelivery → Photobook | `photobookId` で ID のみ参照。公開時に `manageUrlToken` + 現在の `manageUrlTokenVersion` を受け取り、**送信時のバージョンだけスナップショット**として保持する。**管理URLの検証責務は Photobook 集約**にあり、ManageUrlDelivery は一切の認可判定を行わない |
| UsageLimit | サービス層で参照 | 同一IPからの大量送付要求をレート制限 |
| Moderation | Moderation → ManageUrlDelivery | 管理URL再発行時、Moderation が Photobook の `manageUrlTokenVersion` をインクリメントし、新URLの送付リクエストとして**新しい ManageUrlDelivery レコードを生成**する |
| Outbox | ManageUrlDelivery → Outbox | `status=pending` 作成と同一トランザクションで `ManageUrlDeliveryRequested` を INSERT |

### Photobook 側 `manage_url_token_version` との役割対比

| 項目 | Photobook.`manage_url_token_version` | ManageUrlDelivery.`manage_url_token_version_at_send` |
|------|---------------------------------------|---------------------------------------------------|
| 意味 | **現在有効な**管理URLトークンバージョン | **メール送信時点のスナップショット** |
| 可変性 | 再発行でインクリメント | 送信後は不変 |
| 用途 | 管理URLの認可判定 | 監査・問い合わせ対応 |
| 責務所在 | Photobook 集約 | ManageUrlDelivery 集約 |

## 10. セキュリティ・プライバシー上の注意

### 10.1 管理URLトークンのログ露出禁止（v4 §6.7, §10）

- メール送信プロバイダのログ、バウンスメール、エラーログに `manageUrlToken` が残らないよう注意
- メール本文に管理URLを記載するのは避けられないが、**プロバイダAPIへのリクエストログや本サービスの内部ログには管理URL全体を出さない**
- URLの識別子部分（Slug）のみをログに残し、**トークン部分はマスク**する
- エラートラッキング（Sentry等）に送付時のURLをそのまま送らない（`Referrer-Policy: no-referrer` 相当の意識）

### 10.2 メールアドレスの目的外利用禁止（v4 §3.5, §7.5）

- メールアドレスは控え送付**のみ**に使用
- マーケティング、ログイン情報化、アカウント作成のいずれにも転用禁止
- 送信完了・失敗確定から24時間で NULL 化
- これを利用規約・プライバシーポリシーに明記する

### 10.3 メール送信内容

- 件名例: 「VRC PhotoBook: 管理URLの控え」
- 本文: フォトブックのタイトル、管理URL、紛失時の注意、目的外利用しない旨
- HTMLメール可だが**追跡ピクセル・外部リソース読み込みは禁止**（v4 §6.13 同等の漏洩対策）

## 11. メールプロバイダ選定時のチェック項目（v4 新規・P1-10）

メール送信プロバイダ（SES / SendGrid / Resend / Postmark 等）を選定する際、以下の項目を**必ず事前確認**する。いずれかが要件を満たせない場合、管理URLが漏洩する経路になる。

| # | 確認項目 | 要件 |
|---|---------|-----|
| 1 | 送信履歴UIにメール本文が残るか | 運営コンソールから過去の送信メール本文が見られる場合、管理URLが残ることになる。**残らない or マスクできるプロバイダ**を選ぶ |
| 2 | 管理URL全文がプロバイダログに残るか | プロバイダ側の送信ログ・トレースログに本文が保存されるか。**保存されない or 保持期間を短縮できる**こと |
| 3 | APIログに本文が保存されるか | API呼び出しログ（監査ログ）に request body が残るか。**body をログ対象から除外可能**であること |
| 4 | 本文ログの保持期間を設定できるか | ログ保持期間を最短（7〜30日以内）に設定できるか |
| 5 | バウンス通知に本文が含まれるか | バウンス時にプロバイダから返ってくるエラーレスポンスに元メール本文が含まれるか。**含まれる場合は取り扱い注意** |
| 6 | Webhookペイロードに本文が含まれるか | 送信結果Webhookに本文が含まれる場合、Webhook受信側でのログ取り扱いも要注意 |

選定結果は ADR（`docs/adr/`）に記録する。

## v3 からの主な変更点

- **P0-6**: `manageUrlTokenVersion` を **`manageUrlTokenVersionAtSend`** にリネーム。これは「送信時点のスナップショット」であり、管理URL検証責務は Photobook 集約にあることを §1, §3.1, §4.5, §9 で明示
- **P0-13**: `ManageUrlDeliveryRequested` イベントを**状態作成と同一トランザクションで Outbox に INSERT** する運用を §7, §8, I9 に追加
- **P1-7**: `failed_retryable` のまま `expireAt` に到達したケースを `failed_permanent`（`failureReason=expired_during_retry`）に確定させる運用を I7 と §6.2 に追加
- **P1-10**: メールプロバイダ選定時のチェック項目（送信履歴UI、APIログ、本文保持期間等）を §11 として新設
- `ManageUrlDispatcher` の責務に**プロバイダAPIログ・内部ログで管理URLをマスクする**ことを明記（§6.1, §10.1）
- Photobook 側 `manage_url_token_version` と ManageUrlDelivery 側 `manage_url_token_version_at_send` の役割対比表を §9 に追加
