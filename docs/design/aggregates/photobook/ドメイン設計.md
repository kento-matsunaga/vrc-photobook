# Photobook 集約 ドメイン設計

> 上流: [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) 第2部〜第6部 / [Outbox設計](../../cross-cutting/outbox.md) / [OGP生成設計](../../cross-cutting/ogp-generation.md) / [Reconcile設計](../../cross-cutting/reconcile-scripts.md)
>
> この集約は VRC PhotoBook の中核。サービスの全機能がこの集約を中心に回る。
>
> v4 では以下を取り込む: (1) server draft + `draft_edit_token` 方式、(2) 表紙の `photobooks` インライン化、(3) 楽観ロック、(4) Outbox による集約間イベント伝搬、(5) Slug 復元ルールの明文化、(6) 管理URL／draft編集URLの漏洩対策強化。

---

## 1. 集約の責務

フォトブックという1つの公開可能な作品単位のライフサイクル
（**初回画像アップロードでのdraft化 → 公開 → 閲覧 → 編集 → 非表示／復元／論理削除 → 物理削除**）を一貫して管理する。

### 担う責務

- フォトブックとその構成要素（ページ・写真・メタ情報・表紙）の構造的整合性を保証する
- 公開範囲・センシティブ設定・権利配慮確認の一貫性を保つ
- **draft 状態の管理**（`draft_edit_token`、`draft_expires_at` の発行・延長・失効）（v4 新規）
- 公開URLと管理URLを生成・検証する
- タイプ別既定値（レイアウト・開き方）を適用する
- ライフサイクル（draft / published / deleted / purged）と Slug 復元ルールの管理（v4 強化）
- **楽観ロック（`version`）による同時編集事故の防止**（v4 新規） <!-- 付録C P0-7 -->
- **状態変更と同一トランザクションでの `outbox_events` INSERT**（v4 新規）

### 担わない責務

- 画像ファイルそのもの（バイナリ、リサイズ、配信、EXIF除去）→ **Image集約**
- 通報の受付・カテゴリ運用 → **Report集約**
- レート制限・bot検証 → **UsageLimit集約**
- 管理URL控えメール送付 → **ManageUrlDelivery集約**
- 運営操作（一時非表示・管理URL失効）の**実行記録** → **Moderation集約**
  （本集約は「一時非表示かどうか」の状態フラグは持つが、いつ誰が実行したかは持たない）
- OGP 画像の状態管理 → **OGP 横断コンポーネント** (`photobook_ogp_images`)
- 集約間イベントの配送 → **Outbox 横断コンポーネント** (`outbox_events`)

## 2. 集約境界と集約ルート

### 集約ルート: `Photobook`

集約内のすべてのエンティティ（Page, Photo, PageMeta）と値オブジェクト群は、
`Photobook` を経由してのみ生成・変更・削除される。

```
Photobook (集約ルート)
├── Cover (インライン値オブジェクト)       // v4: photobook_covers を廃止し、photobooksに coverTitle / coverImageId を保持
├── Page [1..*]                           // ページ本体
│   ├── PageMeta (0..1)                   // World/Cast/Photographer/Note
│   └── Photo [1..*]                      // ページに所属する写真
│        └── Caption (0..1)               // 写真単位キャプション
├── CreatorInfo (1)                       // 表示名・XID（avatarIcon は v4 で削除、Phase 2）
├── DraftEditToken (0..1)                 // v4: draft状態の編集トークン（ハッシュで保持）
├── ManageUrlToken (0..1)                 // 公開以降の管理URLトークン（ハッシュで保持）
└── PhotobookVersion                      // v4: 楽観ロック用バージョン
```

### 他集約への参照

- `Photo.imageId` は `ImageId`（Image集約のID）のみを保持し、Image集約本体へは直接触れない。
- `Cover.coverImageId` も同様に Image 集約への参照のみ。
- 所有関係は Image 側の `owner_photobook_id` が本集約の `PhotobookId` を指す（v4）。

### Cover のインライン化（v4 新規、P1-1）

v3 までは `CoverPage` 子エンティティ＋`photobook_covers` テーブルで管理していたが、
MVP では 1 Photobook に対して 1 Cover しか持たず、タイトルと代表画像1枚という簡素な構造のため、
**`photobooks` にインライン化（`cover_title`, `cover_image_id`）する**。

### 3.3 Photo

| 属性 | 型 | 説明 |
|------|----|----|
| id | `PhotoId` | 写真ID |
| imageId | `ImageId` | 画像集約の参照 |
| order | `DisplayOrder` | ページ内表示順序 |
| caption | `Caption` (0..1) | 写真単位のキャプション |

### 3.4 Cover（インライン、v4 新規）

`Photobook` の値オブジェクトとして保持される（独立エンティティではない）。

| 属性 | 型 | 説明 |
|------|----|----|
| coverTitle | `Title` (0..1) | 表紙タイトル。未指定時は `Photobook.title` を流用する |
| coverImageId | `ImageId` (0..1) | 代表画像 |

表紙はタイプと開き方で表示有無が決まるが、データとしては保持する。
MVP では 1 Photobook に対して 1 Cover。複数画像レイアウトや専用テンプレートは Phase 2 以降（v4 §5.2）。

## 4. 値オブジェクト

### 識別子系

| VO | 形式 | 制約 |
|----|------|-----|
| `PhotobookId` | UUIDv4 or ULID | 生成時不変 |
| `PageId` | UUIDv4 or ULID | Photobook内でユニーク |
| `PhotoId` | UUIDv4 or ULID | Page内でユニーク |
| `ImageId` | UUIDv4 or ULID | Image集約の識別子（参照のみ） |

### URL・トークン関連

| VO | 定義 | 不変条件 |
|----|------|---------|
| `PublicUrlSlug` | 公開URLのパス部分（短い読める文字列） | 小文字英数ハイフン、12–20文字程度、一意性は §12 の Slug 復元ルールに従う |
| `ManageUrlToken` | 管理URLのトークン | 暗号論的乱数、最低 **128bit** 相当のエントロピー、URL-safe base64、推測不能、**DBにはハッシュのみ保存（raw token は保存しない）** |
| `ManageUrlTokenVersion` | 管理URLトークンのバージョン番号 | 発行時は 0、再発行で +1。「**現在有効な**バージョン」を保持（v4 更新） <!-- 付録C P0-1 --> |
| `DraftEditToken` | **draft 編集URL 用トークン**（v4 新規） | 暗号論的乱数、**128bit 以上**、URL-safe base64、ManageUrlToken と同等の秘匿性。**DBにはハッシュのみ保存（raw token は保存しない）** <!-- 付録C P0-2 --> |
| `PhotobookVersion` | **楽観ロック用バージョン**（v4 新規） | 非負整数。更新操作で必ず +1。不一致ならエラー <!-- 付録C P0-7 --> |

**不変条件**:
- `ManageUrlToken` は `PublicUrlSlug` から推測できない（独立した乱数源）
- `ManageUrlToken` / `DraftEditToken` は公開URL・OGP・メタタグ・ログのいずれにも露出しない（v4 §6.7, §6.13）
- `DraftEditToken` は **publish 成功と同時に失効**（ハッシュを NULL 化）
- `ManageUrlTokenVersion` は ManageUrlDelivery 側の `manage_url_token_version_at_send` と対になる（v4 P0-6）

### タイプ・表示

| VO | 値 | 業務ルール |
|----|-----|----------|
| `PhotobookType` | `event` / `daily` / `portfolio` / `avatar` / `world` / `memory` / `free` | v4 §2.4 |
| `Layout` | `simple` / `magazine` / `card` / `large` | v4 第4部 の既定と選択肢を強制 |
| `OpeningStyle` | `light` / `cover_first_view` | タイプにより既定値が決まる |

### 公開・モラル

| VO | 値 | 業務ルール |
|----|-----|----------|
| `Visibility` | `public` / `unlisted` / `private` | v4 §2.5。既定は `unlisted`（限定公開） |
| `SensitiveFlag` | `on` / `off` | 既定 `off` |
| `RightsConfirmation` | `{ agreed: bool, agreedAt: Timestamp }` | `agreed == true` でなければ公開不可 |

「非公開(`private`)」は「管理URL保持者のみ閲覧可、公開URLは無効」。

### 作成者情報（v4 更新）

| VO | 属性 | 制約 |
|----|------|-----|
| `CreatorInfo` | `displayName: DisplayName` / `xId?: XId` | `displayName` 必須。**v4 で `avatarIcon` は削除**（Phase 2 へ、P1-2） |
| `DisplayName` | 文字列 | 1–50文字、前後空白トリム、コントロール文字禁止 |
| `XId` | 文字列 | X のスクリーンネーム形式（`@`なし、1–15文字、半角英数とアンダースコア） |

### コンテンツ

| VO | 属性 | 制約 |
|----|------|-----|
| `Title` | 文字列 | 1–80文字、必須 |
| `Description` | 文字列 | 0–500文字、任意 |
| `Caption` | 文字列 | 0–200文字、任意 |
| `DisplayOrder` | 整数 | 0以上の連番 |

### メタ情報 `PageMeta`

v4 §2.1 の「World / Cast / Photographer / Note」の4種。

| フィールド | 型 | 備考 |
|-----------|----|----|
| world | `string?` | ワールド名。任意 |
| cast | `string[]?` | キャスト（複数）。任意 |
| photographer | `string?` | 撮影者。任意 |
| note | `string?` | 補足メモ（v2 までの `Comment` はこれ、v3 で改名） |

**不変条件**: `Comment` フィールドはドメイン・スキーマの両方に存在しない（v4 §2.7 非採用用語）。

### ライフサイクル `PhotobookStatus`

| 値 | 意味 | 遷移 |
|----|-----|-----|
| `draft` | **作成途中（server draft、`draft_edit_token` で編集）**（v4） | → `published`、`draft_expires_at` 経過で削除 |
| `published` | 公開済み | → `deleted` / `hiddenByOperator` 切替可 |
| `deleted` | 論理削除済み、保持期間内 | → `purged`（物理削除） または `restore`（`published` に戻す）（v4） |
| `purged` | 物理削除（本体レコード消滅）。**restore 不可**（v4 §12） | 終端 |

`hiddenByOperator` は `status` と独立したフラグ。`published` かつ `hiddenByOperator == true` のとき「一時非表示」。

## 5. 不変条件（Invariants）

集約ルートがすべての変更操作で維持する。

### 構造

- **I1**: `Page` は最低1件存在する（空のフォトブックは許可しない）
- **I2**: 各 `Page` は最低1枚の `Photo` を持つ
- **I3**: `Page.order` と `Photo.order` は集約内で重複なく連続（0, 1, 2, ...）
- **I4**: `Cover` の有無と表示は `OpeningStyle` に整合する（`cover_first_view` なら `coverImageId` が設定されている）

### タイプとレイアウト

- **I5**: `layout` は `LayoutPolicy.allowedLayouts(type)` に含まれる
- **I6**: タイプが変更された場合、`layout` と `openingStyle` は新タイプの既定値に置換される（v4 §3.1）

### 公開条件

- **I7**: `status == published` にするには、以下をすべて満たす必要がある (参照: v4 §3.2) <!-- 付録C P0-3, P0-4, P0-6 -->
  1. `rightsConfirmation.agreed == true`
  2. `creatorInfo.displayName` が入力されている
  3. `publicUrlSlug` と `manageUrlToken` が発行済み
  4. 画像制限（UsageLimit集約が別途検証）を通過している
  5. **`draftEditToken` は publish 成功と同時に失効する**（v4）
  6. **対象 Photobook の draft session はすべて revoke される**（同一 TX、Session 設計の `revokeAllDrafts(photobookId)` を呼ぶ。詳細は §13.3 / §14） (参照: ADR-0003) <!-- 付録C P0-5 -->
- **I8**: `visibility` のデフォルトは `unlisted`（作成時）
- **I9**: 公開後の `publicUrlSlug` は不変。`manageUrlToken` は Moderation集約による再発行時のみ置換され、その際 `manageUrlTokenVersion` が +1 される

### Draft ライフサイクル（v4 新規）

- **I-D1**: `status == draft` のときに限り `draftEditToken` と `draftExpiresAt` を保持する
- **I-D2**: `status == draft` のとき `publicUrlSlug` と `manageUrlToken` は必ず NULL（未発行）
- **I-D3**: draft は初回画像アップロード時に作成される（`Photo` が1枚以上ある状態でしか生まれない）
- **I-D4**: draft アクセスで `draftExpiresAt = now + 7日` に延長される
- **I-D5**: `draftExpiresAt < now` の draft は Reconcile（`draft_expired.sh`）により削除される
- **I-D6**: publish 成功時、`draftEditToken` のハッシュは NULL 化、`draftExpiresAt` は NULL にクリアされる <!-- 付録C P0-4 -->
- **I-D7**: publish 成功時、対象 Photobook の `session_type='draft'` の session はすべて同一 TX 内で revoke される。session の永続化と revoke 操作の責務は Session 設計（`docs/design/auth/session/`、後続ステップで作成）が持ち、本集約は「publish 時に呼び出す業務ルール」を定めるのみ。 (参照: v4 §3.2 / ADR-0003) <!-- 付録C P0-5 -->

### 楽観ロック（v4 新規） <!-- 付録C P0-7 -->

- **I-V1**: すべての状態変更操作は `expectedVersion` を受け取る
- **I-V2**: `expectedVersion != photobook.version` のとき、操作は `OptimisticLockConflict` エラーで失敗する
- **I-V3**: 成功した状態変更は `version` を +1 する

### 削除と復元（v4 §12 Slug 復元ルール）

- **I-D10**: `softDelete()` は `deletedAt` を必ず記録し `status = deleted` に遷移する
- **I-D11**: `status == deleted` のフォトブックは公開URL経由では閲覧できない（管理URL経由の運営判断で `restore` 可能）
- **I-D12**: `status == deleted` のとき `publicUrlSlug` は保持され、他 Photobook からは使用できない
- **I-D13**: `restore()` は `status = published` に戻し、`publicUrlSlug` は同一値を維持する
- **I-D14**: `purge()` は本体レコードを削除。以降 `publicUrlSlug` は再利用可、**restore 不可**
- **I-D15**: `restore` / `softDelete` / `purge` はすべて ModerationAction として別集約に記録される

### 管理URLの秘匿と漏洩対策

- **I13**: `manageUrlToken` は集約外へのシリアライズ時、原則として露出してはならない
  （例外: 発行直後の `CreatorReceipt`、管理URLアクセス時の検証コンテキストのみ）
- **I14**: OGP生成・閲覧用DTOには `manageUrlToken` も `draftEditToken` も含めない（v4）
- **I15**: `draftEditToken` は `manageUrlToken` と同等の秘匿基準で扱う（v4 §6.13）

### Outbox との整合（v4 新規）

- **I-O1**: 以下の状態変更は、同一DBトランザクション内で `outbox_events` にイベントを INSERT する
  - `publish` → `PhotobookPublished`
  - `updateContent` → `PhotobookUpdated`
  - `hide` → `PhotobookHidden`
  - `unhide` → `PhotobookUnhidden`
  - `softDelete` → `PhotobookSoftDeleted`
  - `restore` → `PhotobookRestored`
  - `purge` → `PhotobookPurged`
  - `reissueManageUrl` → `ManageUrlReissued`

## 6. ドメインサービス

### 6.1 `LayoutPolicy`（値オブジェクト的ポリシー）

v4 第4部のタイプ別既定値・選択肢を強制する純粋関数群。

```
allowedLayouts(type: PhotobookType) -> Layout[]
defaultLayout(type: PhotobookType) -> Layout
defaultOpeningStyle(type: PhotobookType) -> OpeningStyle
```

### 6.2 `PublicUrlSlugGenerator`

衝突を避けつつ読みやすいスラッグを生成する。

- 一意性検証はインフラ層（永続化時の衝突チェック）に委譲
- Slug 復元ルール（§12）により、`deleted` 状態の Photobook が保持するスラッグを再利用しない
- リトライ回数の上限を持つ

### 6.3 `ManageUrlTokenGenerator`

暗号論的乱数で `ManageUrlToken` を生成する。

- 最低 128bit 相当のエントロピーを持つ
- URL-safe base64 でエンコード
- 原文は返却直後に破棄、DB にはハッシュのみ保存

### 6.4 `DraftEditTokenGenerator`（v4 新規）

暗号論的乱数で `DraftEditToken` を生成する。`ManageUrlTokenGenerator` と同等の強度。

- 最低 128bit 相当のエントロピー、URL-safe base64
- 原文は draft 作成直後の一度だけ作成者に提示
- DB にはハッシュのみ保存

### 6.5 `XShareTextGenerator`（読み取り専用サービス）

v4 §3.8 の X共有投稿文テンプレート生成。集約の状態を変更しない。
入力: `Photobook` / 出力: `XShareDraft { text, hashtags, publicUrl }`

### 6.6 `PhotobookPublisher`

公開処理をトランザクション的にまとめるドメインサービス。 (参照: v4 §3.2 / ADR-0003) <!-- 付録C P0-3, P0-4, P0-5, P0-6 -->

- I7 の公開条件を検証
- `publicUrlSlug`, `manageUrlToken` を同一トランザクション内で発行
- `draftEditToken` を失効させる（ハッシュ NULL 化）
- **対象 Photobook の draft session を同一 TX 内で全 revoke する**（Session 設計の `revokeAllDrafts(photobookId)` 相当を呼び出す。本集約は呼び出しのみを規定し、永続化責務は Session 設計に委ねる）
- `outbox_events` に `PhotobookPublished` を INSERT（I-O1）
- 楽観ロック（I-V1〜I-V3）を適用

### 6.7 `OgpRegenerationScheduler`（v4 新規、読み取り専用サービス）

Photobook の更新が OGP 再生成をトリガすべきかを判定する。

- `PhotobookUpdated` イベント発火時、OGP 横断コンポーネントのハンドラから呼ばれる
- 判定ロジックは「タイトル／Cover／Creator表示名／代表画像が変わったか」等
- 本集約の状態を変更しない

## 7. ドメインイベント（v4 Outbox 経由）

集約の状態変更と**同一トランザクション**で `outbox_events` に INSERT される（§5 I-O1）。実際の副作用は非同期ワーカーが `event_type` ごとのハンドラで処理する。

| イベント | 発火タイミング | 主な副作用（ハンドラ） |
|---------|--------------|---------|
| `PhotobookPublished` | 公開確定時 | OGP生成ジョブ投入、ManageUrlDelivery 可能状態 |
| `PhotobookUpdated` | 公開後の編集で即時更新された時 | OGP を `stale` に、OGP再生成、公開ページCDN無効化 |
| `PhotobookHidden` | 運営が一時非表示 | 公開ページ・OGP の CDN 無効化 |
| `PhotobookUnhidden` | 運営が一時非表示を解除 | 公開ページCDN無効化、必要なら OGP 再生成 |
| `PhotobookSoftDeleted` | 論理削除 | 公開ページ・OGP の CDN 無効化 |
| `PhotobookRestored` | 論理削除から復元（v4 新規） | 公開ページCDN無効化、OGP 再生成 |
| `PhotobookPurged` | 物理削除 | 所属 Image と OGP 画像の物理削除ジョブ、CDN 削除 |
| `ManageUrlReissued` | 管理URL再発行（運営経由） | 旧URL無効化、新URL控え送付（任意）、ModerationAction 記録 |

Report / ManageUrlDelivery / Moderation など他集約のイベント（`ReportSubmitted`, `ManageUrlDeliveryRequested` 等）は本集約の責務ではない。

| `changeLayout(layout)` | 3.1 | I5 | - |

## 10. 典型的なユースケース

### 10.1 初回画像アップロード → draft 作成（v4 §3.1）

1. 作成者が最初の画像をアップロード → Image 集約で `images` 行が `owner_photobook_id = NULL` で作成される
2. サービス層が `Photobook.createDraft(type, creatorInfo, firstImageId, draftTokenGen)` を呼ぶ
3. 同一TX内で:
   - `photobooks` に `status=draft`, `draftEditTokenHash`, `draftExpiresAt=now+7日`, `version=0` を INSERT
   - Image の `owner_photobook_id` を発行された `PhotobookId` に更新、`usage_kind='photo'`
4. 作成者に **draft編集URL**（`draft_edit_token` を含む）を提示
5. 以降アクセスのたびに `touchDraft()` で `draftExpiresAt` を `now+7日` に延長

### 10.2 draft 編集 → 公開（v4 §3.2）

1. ページ追加、写真紐付け、メタ情報入力（その過程で Image 集約にアップロードが発生。`owner_photobook_id` は同一 draft Photobook を指す）
2. `agreeRights()` で権利・配慮確認
3. サービス層が UsageLimit 集約の検証を通過させる
4. サービス層が bot 検証（外部サービス）を通す
5. `publishFromDraft(publisher, slugGen, tokenGen)` を同一 TX で実行: (参照: v4 §3.2 / ADR-0003) <!-- 付録C P0-3, P0-4, P0-5, P0-6 -->
   - I7 を検証
   - `publicUrlSlug` 発行（Slug 復元ルール §12 に従い他 Photobook の deleted slug と衝突しない）
   - `manageUrlToken` 発行（ハッシュ保存）、`manageUrlTokenVersion = 0`（draft 中の DEFAULT のまま、再発行で +1）
   - `draftEditTokenHash = NULL`, `draftExpiresAt = NULL`
   - **`session_type='draft'` かつ `photobook_id` が一致する session をすべて `revoked_at = now()` で revoke する**（Session 設計の `revokeAllDrafts(photobookId)` を呼ぶ）
   - `status = published`, `publishedAt = now`
   - `outbox_events` に `PhotobookPublished` を INSERT
   - `version` +1
6. 非同期ワーカーが `PhotobookPublished` を処理し OGP 生成をトリガ（本集約の公開可否には影響しない）

### 10.3 閲覧（v4 §3.3）

- 公開URLから `PublicUrlSlug` で引き、可視性を `visibility` と `hiddenByOperator` と `status` で判定
- `status == deleted` or `purged` → 404 相当
- `hiddenByOperator == true` → 一時非表示の穏当な通知
- `visibility == private` → 公開URLでは無効。管理URL保持者のみ閲覧可
- 全ページ `noindex`、Referrer-Policy は `strict-origin-when-cross-origin`（v4 §7.6、P1-9）

### 10.4 管理URL経由の編集（v4 §3.4）

- サービス層が `manageUrlToken` を検証（ハッシュ比較、`manageUrlTokenVersion` が一致）
- 成功時に Photobook 集約の操作メソッドを呼ぶ（楽観ロック適用）
- 管理URLページは `no-referrer` / 外部リソース禁止 / 破壊的操作は二重確認（§13）
- 編集保存で `PhotobookUpdated` が発火 → OGP stale 化 → 再生成

### 10.5 運営による hide → restore / purge（v4 §5.4、§12）

- `hide()` で `hiddenByOperator=true`、`PhotobookHidden` 発火
- `unhide()` で `hiddenByOperator=false`、`PhotobookUnhidden` 発火
- `softDelete()` で `status=deleted`、`deletedAt` 記録、`PhotobookSoftDeleted` 発火。`publicUrlSlug` は保持（§12）
- 保持期間内に `restore()` → `status=published`、同一 slug で復元、`PhotobookRestored` 発火
- 保持期間経過で `purge()` → 本体削除、`PhotobookPurged` 発火 → Image 連鎖削除 → CDN パージ

## 12. Slug 復元ルール（v4 新規、P1-11）

`public_url_slug` の再利用可否を状態ごとに定める。

| 状態 | `publicUrlSlug` の扱い | 他 Photobook への割当可否 | `restore` 可否 |
|------|-------------------|------------------|--------------|
| `draft` | 未発行（NULL） | - | - |
| `published` | 有効。閲覧可能 | 不可（UNIQUE 制約） | - |
| `deleted` | 保持（SLO: 削除保持期間 = 30日） | **不可**（他 Photobook に再利用させない） | **可能**（同一 slug で復元） |
| `purged` | 解放 | **可能**（他 Photobook が取得しうる） | **不可**（本体が消失） |

**設計意図**:
- `deleted` 内で slug を保持するのは、誤削除・通報対応の巻き戻しを可能にするため
- `purged` 以降は監査証跡が別途残っている前提で、slug を解放して再利用可能にする
- `restore` / `softDelete` / `purge` はすべて ModerationAction として追記記録される

DB 制約は `UNIQUE (public_url_slug) WHERE status IN ('published', 'deleted')` の部分ユニーク（データモデル §3 参照）。

## 13. 管理URLと draft 編集URL の漏洩対策（v4 §6.13）

`ManageUrlToken` と `DraftEditToken` は**同等のリスク水準**で扱う。どちらかが漏洩するとフォトブックの編集／削除権を第三者が握りうる。

### 13.1 DBでの扱い

- 原文は DB に保存しない。**ハッシュ化して保存**（`manage_url_token_hash`, `draft_edit_token_hash`）
- 検証時は入力トークンをハッシュして比較
- アプリケーションログ・外部エラートラッキング（Sentry 等）に URL 全文を送らない

### 13.2 ページ側の対策（管理URLページ / draft 編集URL ページ）

- `Referrer-Policy: no-referrer` を必須化
- 外部画像・外部フォント・外部スクリプトの読み込み禁止（v4 §3.4）
- アクセス解析タグの挿入禁止
- X共有ボタンには**公開URLのみ**を渡す（管理URL混入禁止）
- 破壊的操作（削除、公開範囲変更、管理URL再発行、センシティブ設定変更）には**二重確認**
- GET で状態変更しない（POST/PUT/PATCH/DELETE のみ）
- CORS は自サイトに限定
- `noindex` 必須、robots.txt で `/manage/` と `/draft/` を Disallow（v4 §7.6）

### 13.3 集約面での責務

本集約はトークン原文を外部に**漏らさない**責務を負う。具体的には:

- 閲覧用DTO（OGP 生成・公開ページ向け）に `ManageUrlToken` / `DraftEditToken` を含めない（I14, I15）
- 発行直後の `CreatorReceipt` のみ原文を保持し、即座にクライアントへ返却後は破棄
- トークンの比較は必ずハッシュ経由
- **publish 成功時、対象 Photobook の draft session を同一 TX で全 revoke する**（Session 設計の `revokeAllDrafts(photobookId)` を呼ぶ）。draft session が残ったまま publish 完了すると、既存 Cookie session を持つ端末から draft URL 経由のアクセスで認可状態が混乱するため。 (参照: ADR-0003) <!-- 付録C P0-5 -->

## 14. 次集約への引き継ぎ事項

本集約の P0 反映を後続集約の設計に渡すための申し送り。`docs/spec/vrc_photobook_business_knowledge_v4.md` の付録C 番号を追跡 ID として用いる。

### 14.1 Session 機構（`docs/design/auth/session/`、コミット済み）

- **`revokeAllDrafts(photobookId)` 操作が必要** (参照: ADR-0003) <!-- 付録C P0-5 -->
  - publish 時に Photobook 集約の `publishFromDraft` から同一 TX で呼ばれる
  - 対象は `session_type='draft'` かつ `photobook_id` が一致する session すべて
  - `revoked_at = now()` を設定し、Cookie 自体は端末側に残るが API 認可は失敗する
- **`revokeAllManageByTokenVersion(photobookId, oldVersion)` 操作も必要** (参照: ADR-0003)
  - manage_url_token 再発行（`reissueManageUrl`）時に呼ばれる
  - 対象は `session_type='manage'` かつ `photobook_id` が一致し、`token_version_at_issue <= oldVersion` の session
- **`sessions` テーブルのスキーマは Session 設計側で定義する**。本集約はテーブルを直接持たない

### 14.2 Moderation / Report 集約（既存設計、コミット済み）

- **`Photobook.version` は 4 要素同一 TX（Photobook + ModerationAction + Report + outbox_events）で楽観ロック衝突検知に使う** (参照: v4 §6.20) <!-- 付録C P0-7, P0-19 -->
  - Moderation 経由の `hide / unhide / softDelete / restore / purge / reissueManageUrl` も `expectedVersion` を受け取り、不一致なら `OptimisticLockConflict` を返す
  - Report 状態更新（`resolveWithAction` 等）もこの 4 要素 TX に含まれる
- **`source_report_id` 付きの `hide / soft_delete / purge` では Report.status を `resolved_action_taken` に遷移**させる責務は Moderation/Report 側にある（Photobook 集約は trigger に過ぎない） <!-- 付録C P0-20 -->

### 14.3 ManageUrlDelivery 集約

- **`Photobook.manage_url_token_version` は ManageUrlDelivery の `manage_url_token_version_at_send` と対になる** (参照: v4 §3.5) <!-- 付録C P0-1, P0-24 -->
  - 送信時点の値を ManageUrlDelivery 側で snapshot 保持
  - 後の token 再発行で `manage_url_token_version` がインクリメントされても、過去の delivery レコードは送信時点の値を維持する
- **ReissueManageUrl 時、過去の `recipient_email` は再利用しない** (参照: v4 §3.5) <!-- 付録C P0-26 -->
  - 運営が手動入力した新しい連絡先で新規 ManageUrlDelivery を作る

### 14.4 Image 集約 / OGP 横断

- **OGP 画像実体は Image 集約で管理**（`images.usage_kind = 'ogp'`）。Photobook 集約は OGP 状態（pending/generated/failed/fallback/stale）を直接持たず、`PhotobookPublished` / `PhotobookUpdated` 等のイベント発火のみを責務とする (参照: v4 §6.17) <!-- 付録C P0-13 -->
- **Image の `owner_photobook_id` は publish 時に変更しない**。Draft と Published は同一 Photobook ID なので所有関係は不変 (参照: v4 §6.14)
- **Photobook 物理削除（purge）時の Image 連鎖削除は Outbox 経由で非同期実行**（同期 CASCADE ではない）。本集約は `PhotobookPurged` イベントを INSERT するのみで、実削除は Image 集約側ハンドラの責務

### 14.5 Outbox 横断（`docs/design/cross-cutting/outbox.md`）

- **状態変更と `outbox_events` INSERT は同一 TX**（I-O1） (参照: v4 §6.11) <!-- 付録C P0-27, P0-28 -->
- **本集約が発火する 8 イベント**: `PhotobookPublished` / `PhotobookUpdated` / `PhotobookHidden` / `PhotobookUnhidden` / `PhotobookSoftDeleted` / `PhotobookRestored` / `PhotobookPurged` / `ManageUrlReissued`
- **failed Outbox は Reconcile 対象**: `outbox_failed_retry` 自動 reconciler が拾う (参照: v4 §6.16) <!-- 付録C P0-29, P0-30 -->

### 14.6 未解決事項（後続で確認）

- **`manage_url_token_version` の初期値**: 本集約 §10.2 とデータモデル設計 §3 では DEFAULT 0、再発行で +1（draft 中も 0、publish 時も 0 のまま、初回再発行で 1）。一方 v4 業務知識 §3.2 の記述「`manage_url_token_version = 1` で開始」と表記が不一致。**設計側の DEFAULT 0 を正とし、v4 業務知識を後続で訂正**する想定（次タスクで処理）

## v3 からの主な変更点

- **Cover インライン化（P1-1）**: `CoverPage` 子エンティティを廃止し、`Photobook` の値オブジェクト `Cover` として属性化（`coverTitle`, `coverImageId`）
- **`CreatorInfo.avatarIcon` 削除（P1-2）**: MVP から除外、Phase 2 へ延期
- **Server draft + `DraftEditToken`（P0-9）**: 初回画像アップロードで draft Photobook を作成、`draft_edit_token` と `draft_expires_at` を保持、アクセス毎に延長、publish で失効
- **楽観ロック（P1-6）**: `PhotobookVersion` VO を追加、すべての状態変更操作が `expectedVersion` を受け取る
- **Outbox との統合（P0-13）**: `publish/update/hide/unhide/softDelete/restore/purge/reissueManageUrl` を状態変更と同一TXで `outbox_events` に INSERT
- **`restore` 操作の新設**: `deleted` → `published` への復元を操作として明示化、同一 slug を維持
- **Slug 復元ルールの明文化（P1-11）**: `deleted` 内は同 slug 維持・他者再利用不可、`purged` 以降は解放・restore 不可、§12 に集約
- **`manageUrlTokenVersion` の意味明確化（P0-6）**: 「現在有効なトークンバージョン」。ManageUrlDelivery 側は `manage_url_token_version_at_send` にリネームされ、本集約と対になる
- **管理URL / draft 編集URL の漏洩対策（§6.13）**: `no-referrer`、外部リソース禁止、破壊的操作の二重確認、ログ送信禁止を §13 に集約
- **Image 所有モデルへの追随（§6.14）**: `Image.owner_photobook_id` が本集約の `PhotobookId` を指す前提で記述
- **ドメインイベント列挙の更新**: `PhotobookDeleted` → `PhotobookSoftDeleted`、`PhotobookRestored` 追加。ハンドラは Outbox ワーカーが担当
