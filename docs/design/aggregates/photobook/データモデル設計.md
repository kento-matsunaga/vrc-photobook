## 1. 扱うデータ

- フォトブック本体の属性（タイトル、説明、公開範囲、センシティブ設定、同意、作成者情報、URL類）
- **表紙（タイトル・代表画像）のインラインデータ**（v4 新規、`photobook_covers` 廃止）
- **draft 編集トークン（ハッシュ）と有効期限**（v4 新規）
- **楽観ロック用バージョン**（v4 新規）
- ページ・写真の構造と順序
- ページ単位のメタ情報（World/Cast/Photographer/Note）
- ライフサイクル状態（draft / published / deleted / purged）
- 運営による一時非表示フラグ

画像ファイルそのものは [Image 集約](../image/データモデル設計.md) で管理する。所有関係は Image 側の `owner_photobook_id` が本集約を指す。

### 3.1 CHECK 制約（アプリ＋DB）

- `status IN ('draft', 'published', 'deleted', 'purged')`
- `status = 'draft'` のとき:
  - `public_url_slug IS NULL` AND `manage_url_token_hash IS NULL` AND `draft_expires_at IS NOT NULL` AND `draft_edit_token_hash IS NOT NULL`
- `status IN ('published', 'deleted')` のとき:
  - `public_url_slug IS NOT NULL` AND `manage_url_token_hash IS NOT NULL` AND `draft_edit_token_hash IS NULL` AND `draft_expires_at IS NULL`
- `status = 'purged'` のとき: レコードは原則存在しない（物理削除）。監査証跡は Moderation / Outbox 側

上記は DB の CHECK + アプリ層の不変条件（ドメイン設計 §5）で担保する。

### 3.2 索引

| 索引 | カラム / 条件 | 用途 |
|------|-------|------|
| PK | `id` | 主キー |
| UNIQUE（部分） | `public_url_slug` WHERE `status IN ('published', 'deleted')` | **v4 Slug 復元ルール**: `deleted` でも保持し再利用不可、`purged` では解放 |
| UNIQUE | `manage_url_token_hash` WHERE `manage_url_token_hash IS NOT NULL` | 管理URL トークンの一意性 |
| UNIQUE | `draft_edit_token_hash` WHERE `draft_edit_token_hash IS NOT NULL` | draft 編集トークンの一意性（v4 新規） |
| INDEX | `status, deleted_at` | 論理削除・物理削除バッチ |
| INDEX | `status, draft_expires_at` WHERE `status='draft'` | `draft_expired.sh` による期限切れ検出（v4 新規） |
| INDEX | `published_at DESC` WHERE `visibility='public' AND status='published' AND hidden_by_operator=false` | 将来の一覧機能（Phase 2 以降） |

### 3.3 重要な設計判断

- **管理URLトークンとdraft編集トークンはハッシュ化保存**: v4 §6.7, §6.13 の秘匿方針徹底。原文は DB に残さない
- **表紙のインライン化（v4 P1-1）**: MVP は 1 Photobook あたり 1 Cover（タイトル＋代表画像1枚）のため、別テーブル化のメリットより JOIN コストのデメリットが上回る
- **`public_url_slug` は論理削除後も保持**: 誤削除時の運営復元（`restore`）を同一 slug で行うため。物理削除 (`purged`) で解放（v4 §12 / 付録B P1-11）
- **`status` と `hidden_by_operator` は独立**: v4 §3.4、§5.4
- **楽観ロック（v4 P1-6）**: `version` を UPDATE 時の WHERE 句に含める方式。不一致で 0 行 UPDATE → アプリ側で `OptimisticLockConflict` エラー
- **`manage_url_token_version` の意味（v4 P0-6）**: 「現在有効なトークンバージョン」。ManageUrlDelivery は送信時点の値を `manage_url_token_version_at_send` として別途スナップショット保持する
- **`cover_image_id` は ON DELETE SET NULL**: Image 集約側の孤児 GC と整合させる。ただし通常運用では Image は `owner_photobook_id` に縛られて Photobook 物理削除時に連鎖削除される

## 8. 保持期間と削除ポリシー

v4 §3.4、§3.1（draft）、§6.11（Outbox）、§6.14（画像所有と削除連鎖）、§12（Slug 復元）に従う。

| 状態 | 条件 | 保持期間 | アクション |
|------|------|---------|-----------|
| `draft` | 初回画像アップロード以降 | **`draft_expires_at` まで**（アクセスで延長、最終アクセスから 7日） | `draft_expired.sh` で期限切れを削除。連鎖で Image 集約の該当行（`owner_photobook_id` 一致）も `deleted` 化 |
| `published` | 通常公開 | 無期限 | - |
| `hidden_by_operator=true` | 運営一時非表示 | 無期限 | 運営判断で解除または削除遷移 |
| `deleted` | `softDelete()` 実行後 | **30日**（初期値、運用調整） | 日次バッチで `purge` 候補を抽出。保持期間内は `restore` 可能（v4 §12） |
| `purged` | 物理削除実行後 | レコード自体は削除 | 監査ログ / ModerationAction / Outbox に痕跡。以降 `restore` 不可、slug 解放 |

### 8.1 Draft の GC（v4 新規）

- `scripts/ops/reconcile/draft_expired.sh` が `status='draft' AND draft_expires_at < now()` を検出
- 実行時は以下を同一トランザクションで:
  1. `photobooks` を削除（物理削除で OK、監査不要。業務知識 v4 §3.1）
  2. CASCADE で `photobook_pages` / `photobook_photos` / `photobook_page_metas` が削除
  3. Image 集約側の `owner_photobook_id` 一致行を `deleted` に遷移（Image 側のライフサイクルルールに従う）
- draft は公開物ではないため、保持期間を設けず即時削除してよい

### 8.2 論理削除と復元（v4 §12）

- `softDelete()`: `status=deleted`, `deletedAt=now()`。`outbox_events` に `PhotobookSoftDeleted`。`public_url_slug` は保持
- `restore()`: 保持期間内に運営が復元。`status=published`、同一 slug 維持。`outbox_events` に `PhotobookRestored`
- restore / softDelete / purge はすべて ModerationAction として別集約に記録される

### 8.3 物理削除バッチ（purge）

- 日次バッチで `status='deleted' AND deleted_at < now() - INTERVAL '30 days'` を対象に
  1. 同一トランザクションで `photobooks` を削除（または監査のため `status='purged'` へ）し、`outbox_events` に `PhotobookPurged` を INSERT
  2. Outbox ワーカーが `PhotobookPurged` ハンドラで:
     - `owner_photobook_id` 一致の Image を物理削除通知
     - `photobook_ogp_images` 行の CASCADE 削除に伴い OGP 画像の物理削除
     - CDN キャッシュパージ（公開ページ・OGP・display / thumbnail 画像）
- 保持期間の具体値はプライバシーポリシーに明記する（v4 §3.4）

### 8.4 Outbox との連動

本集約の状態変更（publish / update / hide / unhide / softDelete / restore / purge / reissueManageUrl）は**同一トランザクションで `outbox_events` に INSERT** される（ドメイン設計 §7、Outbox設計 §4.1）。
Outbox 側の保持期間・リトライ・Reconcile は Outbox設計 §8, §9 に従う。

## 9. マイグレーション実施方針

### 9.1 基本

- マイグレーションは 1 機能 1 ファイル（機能単位でレビュー可能に）
- 破壊的変更（列削除、型変更）は 2 ステップでリリース: 追加 → 移行 → 削除
- `manage_url_token_hash` / `draft_edit_token_hash` のハッシュアルゴリズム変更時は `manage_url_token_version` / 新規バージョン列を使い段階的に移行

### 9.2 v3 → v4 マイグレーション手順

1. `photobooks` に v4 新規カラムを追加（すべて NULL 許容 or デフォルト付き）:
   - `cover_title TEXT`, `cover_image_id UUID`
   - `draft_edit_token_hash BYTEA`, `draft_expires_at TIMESTAMPTZ`
   - `version INT NOT NULL DEFAULT 0`
2. 既存 `photobook_covers` の内容を `photobooks.cover_title` / `cover_image_id` へデータコピー
3. アプリを新スキーマ前提にデプロイ（`photobook_covers` は読み書きしない）
4. 動作検証後、`photobook_covers` テーブルを DROP
5. `creator_avatar_image_id` カラムを DROP（MVP から除外、Phase 2 で別カラムで再導入の可能性）
6. CHECK 制約（§3.1）と部分 UNIQUE インデックス（§3.2）を追加

なお、MVP 着手時点で v3 の実装がまだ存在しなければ、上記はスキップして最初から v4 スキーマで DDL を書けばよい。
