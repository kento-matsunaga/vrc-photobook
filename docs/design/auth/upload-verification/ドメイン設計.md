# Upload Verification 認可機構 ドメイン設計

> 上流: [auth README](../README.md) / [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) §2.8, §3.7 / [ADR-0005 画像アップロード方式](../../../adr/0005-image-upload-flow.md)
>
> 本書が扱うのは **Turnstile セッション化機構**。upload-intent 大量発行を防ぎつつ、画像 20 枚アップロード時に毎回 Turnstile を要求しない UX を実現する短期認可機構。
>
> 集約ではなくアプリケーション層の認可機構として扱う理由は [auth README](../README.md) を参照。

---

## 1. 機構の責務

Turnstile（CAPTCHA 相当の bot 検証）の検証成功結果を、**短期間 + 回数制限付き**で再利用できるようにする横断機構。

### 担う責務

- Turnstile 検証成功時に upload_verification_session を発行する
- upload-intent API 呼び出しごとに残り intent 回数を消費する
- 期限切れ・回数超過・Photobook 不一致を検出して再検証を求める
- session_token は 256bit 乱数、DB には SHA-256 ハッシュのみ保存する
- 期限切れ session の GC 候補抽出（実行は Reconcile）

### 担わない責務

- Turnstile API そのものとの通信 → Application 層 / 外部サービス Adapter
- upload-intent API 本体（presigned URL 発行など） → Image 集約 + R2 Adapter
- complete API の検証 → Image 集約
- 画像処理本検証（マジックナンバー、デコード等） → image-processor ワーカー

## 2. 境界と責任

集約ではないため「集約ルート」は存在しない。代わりに **概念単位 `UploadVerificationSession`** とそのライフサイクル操作を定義する。

### 概念単位: `UploadVerificationSession`

```
UploadVerificationSession
├── id                     UploadVerificationId
├── sessionTokenHash       UploadVerificationTokenHash    (Cookie/Header の raw token の SHA-256)
├── photobookId            PhotobookId                    (対象フォトブック)
├── allowedIntentCount     int                            (発行時の許可上限、初期 20)
├── usedIntentCount        int                            (消費済み、初期 0)
├── expiresAt              Timestamp                      (有効期限、発行時刻 + 30 分)
├── createdAt              Timestamp                      (発行時刻)
└── revokedAt              Timestamp?                     (revoke された時刻、NULL なら有効)
```

### Image 集約との関係

- Image 集約は `upload_verification_sessions` テーブルを直接持たない
- Image 集約の `upload-intent` UseCase は本機構の `validateAndConsume(rawToken, photobookId)` を呼び出す
- 本機構は Image 集約のテーブルや内部状態には触れない

### Session 機構との関係

- 同じ auth/ 配下だが、テーブル・ライフサイクル・操作は独立
- session_token のハッシュ方針（256bit 乱数 + SHA-256、bytea 32B）は共通

## 3. エンティティ（仮想）

| 属性 | 型 | 説明 |
|------|----|----|
| `id` | `UploadVerificationId` | DB 内部 ID（UUIDv7、ADR-0001） |
| `sessionTokenHash` | `UploadVerificationTokenHash` | Cookie/Header の raw token の SHA-256（32 バイト） |
| `photobookId` | `PhotobookId` | 対象 Photobook（他 Photobook に流用不可） |
| `allowedIntentCount` | `int` | 発行時の許可上限。MVP 初期値 **20** |
| `usedIntentCount` | `int` | 消費済み回数。発行時 0 |
| `expiresAt` | `Timestamp` | 有効期限。発行時刻 + **30 分** |
| `createdAt` | `Timestamp` | 発行時刻 |
| `revokedAt` | `Timestamp?` | revoke された時刻。NULL なら有効 |

## 4. 値オブジェクト

### 識別子・ハッシュ系

| VO | 形式 | 制約 |
|----|------|-----|
| `UploadVerificationId` | UUIDv7 | DB 内部 ID。クライアント保持値**には載せない** |
| `UploadVerificationToken` | base64url 文字列 | **256bit 以上の暗号論的乱数**を base64url 化（推奨 32 バイト = 43 文字）。Cookie/Header に載る raw 値 |
| `UploadVerificationTokenHash` | bytea 32 バイト | `UploadVerificationToken` の **SHA-256**。DB 保存はこちらのみ <!-- 付録C P0-16 相当方針を upload-verification にも適用 --> |

session 機構と同じ方針:

- raw token は DB に保存しない
- Cookie/Header には 256bit 乱数を base64url 化したものを使う
- DB には SHA-256 ハッシュ（32 バイト）のみを保存

### 利用回数・期限

| VO | 制約 |
|----|-----|
| `AllowedIntentCount` | 正整数。MVP 初期値 20。実装ごとの設定で増減可だが業務知識 v4 §3.7 のデフォルトは 20 |
| `UsedIntentCount` | 非負整数。`UsedIntentCount <= AllowedIntentCount` |
| `ExpiresAt` | `createdAt < ExpiresAt`、MVP は `createdAt + 30 分` |

## 5. 不変条件（Invariants）

`<!-- 付録C P0-17, P0-18 -->`

### セッション状態

- **I-U1**: `sessionTokenHash` は SHA-256（32 バイト固定）。raw token は DB に保存しない (参照: ADR-0005)
- **I-U2**: `revokedAt IS NOT NULL` または `expiresAt < now()` または `usedIntentCount >= allowedIntentCount` のとき、本 session は無効。検証時に拒否し再 Turnstile を求める
- **I-U3**: 同一 `sessionTokenHash` は DB 内に最大 1 件（衝突時は新規発行を再試行）

### 発行時

- **I-U4**: `allowedIntentCount = 20`（MVP 既定）、`usedIntentCount = 0` で発行 <!-- 付録C P0-18 -->
- **I-U5**: `expiresAt = createdAt + 30 分` で発行 <!-- 付録C P0-18 -->
- **I-U6**: `photobookId` は発行時に確定し、不変。他 Photobook の upload-intent には使えない (参照: ADR-0005) <!-- 付録C P0-18 -->

### 消費（intent 1 件）

- **I-U7**: `validateAndConsume(rawToken, photobookId)` 成功時、`usedIntentCount += 1` を **アトミックに** 更新する（同時実行で同じ session を 2 つの intent に使う事故を防ぐ）
- **I-U8**: 消費前に検証する条件: (a) hash 一致、(b) `revokedAt IS NULL`、(c) `expiresAt > now()`、(d) `photobookId` 一致、(e) `usedIntentCount < allowedIntentCount`
- **I-U9**: いずれかが失敗したら 403 相当を返し、エラーコード `turnstile_verification_failed` を返す

### 再検証要求

- **I-U10**: 期限切れ・回数超過・Photobook 不一致のいずれかの場合、クライアントに **再 Turnstile チャレンジ** を要求する
- **I-U11**: 同一 Photobook で複数の有効な upload_verification_session が並存することは許容する（ユーザーが別端末から同時に upload している等）。各 session は独立に消費される

## 6. ドメイン操作

集約ルートではないため、ドメインサービス的な独立関数として定義する。

| 操作 | 呼び出し元 | 効果 | TX |
|------|----------|------|-----|
| `issue(photobookId, turnstileResult)` | API（Turnstile 検証成功直後） | 新規 session を INSERT、raw token を返す | 自身の TX |
| `validateAndConsume(rawToken, photobookId)` | Image `upload-intent` UseCase | I-U8 を検証し成功時 `usedIntentCount += 1` | 呼び出し元の TX に参加（または独立 TX、後述） |
| `revoke(uploadVerificationId)` | 明示破棄 / abuse 検知 | `revokedAt = now()` に更新 | 自身の TX |

### 6.1 `issue(photobookId, turnstileResult)` の詳細

**手順**:
1. Turnstile API のサーバー側検証を実行（外部呼び出し、本機構の責務外）
2. 成功なら、`UploadVerificationToken` を 32 バイト乱数で生成
3. `sessionTokenHash = SHA-256(UploadVerificationToken)` を計算
4. `upload_verification_sessions` に INSERT (`photobookId`, `allowedIntentCount=20`, `usedIntentCount=0`, `expiresAt=now+30分`)
5. raw token を返す（クライアント側で Cookie / Header / メモリのいずれかに保持）

### 6.2 `validateAndConsume(rawToken, photobookId)` の詳細 <!-- 付録C P0-18 -->

**手順**:
1. `rawToken` を SHA-256 で hash 化 → `sessionTokenHash`
2. `upload_verification_sessions` を `(sessionTokenHash, photobookId)` で検索
3. ヒットなければ 403 + `turnstile_verification_failed`
4. `revokedAt IS NOT NULL` なら 403
5. `expiresAt < now()` なら 403
6. `usedIntentCount >= allowedIntentCount` なら 403
7. **アトミックに** `usedIntentCount += 1` を UPDATE（`UPDATE ... SET used_intent_count = used_intent_count + 1 WHERE id=$1 AND used_intent_count < allowed_intent_count` で 0 行なら回数超過）
8. 成功すれば intent 発行を続行（呼び出し元の Image upload-intent UseCase が presigned URL 発行へ）

### 6.3 TX の取り扱い

`validateAndConsume` は 2 通りの実装が可能:

- **案T1: 呼び出し元 TX に参加**: Image `upload-intent` UseCase の TX 内で消費 + Image レコード作成。失敗時は両方ロールバック
- **案T2: 独立 TX**: 消費を先にコミットしてから Image 作成。Image 作成失敗時に消費は戻らない

**MVP 推奨は案T2**。理由:
- 案T1 だと、upload-intent が失敗するたびに消費がロールバックされ、abuse 検知が効きにくい
- 案T2 で「消費したけど upload-intent が失敗した」状態でも、ユーザーは intent 残数を 1 失うだけで、再試行可能（19 回残っている）
- 失敗の頻度が高ければ別途 Reconcile / 監視で対処

### 6.4 `revoke(uploadVerificationId)` の詳細

**手順**:
1. `revokedAt = now()` を UPDATE
2. クライアント側の Cookie / Header はそのまま残るが、検証時に I-U2 で拒否される
3. abuse 検知（短時間で大量失敗等）で運営が呼び出すケースを想定

---

## 7. ドメインイベント

本機構も session 機構と同様、**Outbox にイベントを乗せない** 方針。理由:

- upload-verification の発行・消費・失効は内部認可状態であり、業務イベントではない
- intent 消費の記録は構造化ログで十分
- 監視・abuse 検知は別途メトリクスで実装

## 9. 典型的なユースケース

### 9.1 Turnstile セッション発行 → 連続 upload-intent

```
1. ユーザーが画像アップロード画面に遷移
2. Turnstile ウィジェット表示、ユーザーがチャレンジ通過
3. クライアントが POST /api/auth/upload-verification を Turnstile result とともに呼ぶ
4. サーバーが Turnstile API でサーバー側検証
5. 成功 → issue(photobookId, turnstileResult) を呼ぶ
6. upload_verification_sessions に INSERT (allowed=20, used=0, expires=now+30m)
7. Set-Cookie or レスポンス body で raw upload_verification_token を返す
8. クライアントが画像 1 枚目を選択 → POST /api/photobooks/{id}/images/upload-intent
9. サーバーが validateAndConsume(rawToken, photobookId) を呼ぶ
10. 検証成功、used_intent_count 0 → 1 にアトミック UPDATE
11. Image レコード作成、presigned URL 発行
12. 画像 2 枚目選択 → 同じ raw token で再度 upload-intent
13. validateAndConsume → used 1 → 2
14. ...（最大 20 回）
15. 21 回目で I-U8 (e) に違反、403 + turnstile_verification_failed
16. クライアントが再 Turnstile を表示
```

### 9.2 期限切れ

```
1. ユーザーが Turnstile を通過してから 31 分経過
2. 21 枚目の upload-intent で validateAndConsume が呼ばれる
3. expires_at < now() を検出、I-U8 (c) 違反 → 403
4. クライアントは再 Turnstile を要求
5. 期限切れ session 行は revoked_at NULL のまま残るが、reconciler で物理削除
```

### 9.3 Photobook 不一致

```
1. ユーザーが Photobook A 用に Turnstile を通過
2. 何らかの理由で Photobook B の upload-intent に同じ token を使う（攻撃 or バグ）
3. validateAndConsume(rawToken, photobookId=B) → photobook_id 不一致で I-U8 (d) 違反 → 403
4. usedIntentCount は変更されない（消費されない）
```

### 9.4 abuse 検知時の手動 revoke

```
1. 監視で同一 Photobook に対して短時間で大量の upload-intent 失敗を検出
2. 運営が cmd/ops 経由で revoke(uploadVerificationId) を実行
3. revoked_at = now() に UPDATE
4. 以降の validateAndConsume で I-U2 違反 → 403
5. 異常リクエスト元は再 Turnstile も突破できなければ UsageLimit にも引っかかる
```

## 11. 次設計への引き継ぎ事項

### 11.1 Image upload-intent UseCase

- upload-intent 受付時に `validateAndConsume(rawToken, photobookId)` を呼ぶこと
- 失敗時は 403 + `turnstile_verification_failed` を返し、クライアントに再 Turnstile を要求するメッセージを含める
- 案T2（独立 TX）採用のため、消費後に Image 作成が失敗してもロールバックしない

### 11.2 API ルート（Turnstile 検証発行）

- `POST /api/auth/upload-verification` を新設
- リクエスト body: Turnstile token + photobook_id
- サーバー側で Turnstile API 検証 → `issue(photobookId, turnstileResult)` 呼び出し
- レスポンス: `upload_verification_token`（raw、Cookie or Body）

### 11.3 クライアント実装

- raw upload_verification_token は Cookie または HttpOnly でない一時保持で扱う（セキュリティ要件は Application/API 設計で確定）
- 30 分タイマーで切れる前に再 Turnstile を表示
- 20 intent 消費する前に「再検証が必要です」を出す UI

### 11.4 Reconcile

- **expired upload_verification_session GC**: `expires_at < now() - INTERVAL '1 day'` の行を物理削除（自動 reconciler）
- **revoked GC**: `revoked_at < now() - INTERVAL '7 days'` を物理削除（abuse 検知監査のため 7 日保持）

### 11.5 abuse 検知 / 監視

- 同一 Photobook で 1 分間に N 件の `validateAndConsume` 失敗 → アラート
- `cmd/ops` から `revoke(uploadVerificationId)` を実行可能にする（運営判断）

## 12. 未解決事項

U2〜U4 は session 側の未解決事項を継承（[session/ドメイン設計.md §13](../session/ドメイン設計.md) 参照）。本機構固有の未解決事項は U1 と U5 以降。

### U1: revoke 方式

session 機構と同じく、**MVP は `revoked_at` UPDATE 方式で確定**。INSERT-only 履歴方式は Phase 2 で検討。

### U5: アトミック消費の競合解決（本機構固有）

`UPDATE ... SET used_intent_count = used_intent_count + 1 WHERE id = $1 AND used_intent_count < allowed_intent_count` で 0 行なら回数超過、という方式は PostgreSQL では行ロックで安全。複数同時 upload-intent が走った場合、片方が `used = N → N+1` UPDATE 中に他方が読むと、後者は新しい値を見るためレースは起きない（Read Committed 以上で安全）。

ただし、極端に同時度が高い場合（100 並列等）は最終的に 19 件成功 + 81 件失敗ではなく、ロック競合で性能劣化する可能性がある。**MVP では並列度 20 程度を想定**、性能問題が顕在化したら Phase 2 で対処（advisory lock 等）。

### U6: クライアント保持方式（本機構固有）

Cookie か Body で返すか（Header / Cookie / localStorage の選択）は Application/API 設計で確定。**HttpOnly Cookie が推奨候補**だが、画像処理を JS でハンドルする都合上、JS から読めない HttpOnly Cookie を要求するか、JS が読める通常 Cookie にするか、判断が分かれる。M1 スパイクで実機検証。
