## 1. 扱うデータ

- Turnstile 検証成功後の発行記録
- 残り intent 回数（`allowed_intent_count` - `used_intent_count`）
- 有効期限（`expires_at`、発行時刻 + 30 分）
- 対象 Photobook の固定（`photobook_id`、他 Photobook へ流用不可）
- session_token の hash（**raw token は保存しない**）
- revoke 状態（abuse 検知時の手動 revoke 用）

raw token そのものは DB に存在しない。

## 3. `upload_verification_sessions`

`<!-- 付録C P0-17, P0-18 -->`

| カラム | 型 | NULL | 既定 | 制約・備考 |
|-------|-----|------|------|----------|
| `id` | `uuid` | NOT NULL | - | PK。**UUIDv7** で生成（ADR-0001） |
| `session_token_hash` | `bytea` | NOT NULL | - | **SHA-256(UploadVerificationToken)、32 バイト固定**。raw token は保存しない |
| `photobook_id` | `uuid` | NOT NULL | - | FK → `photobooks.id` ON DELETE CASCADE。他 Photobook には流用不可 <!-- 付録C P0-18 --> |
| `allowed_intent_count` | `int` | NOT NULL | `20` | MVP 既定 20。発行時固定、変更不可 <!-- 付録C P0-18 --> |
| `used_intent_count` | `int` | NOT NULL | `0` | アトミック UPDATE で +1 する <!-- 付録C P0-18 --> |
| `expires_at` | `timestamptz` | NOT NULL | - | 発行時刻 + 30 分（ドメイン側で算出して INSERT） <!-- 付録C P0-18 --> |
| `created_at` | `timestamptz` | NOT NULL | `now()` | 発行時刻 |
| `revoked_at` | `timestamptz` | NULL | - | abuse 検知時の手動 revoke。NULL なら有効 |

### 3.1 CHECK 制約

```sql
allowed_intent_count > 0
used_intent_count >= 0
used_intent_count <= allowed_intent_count
expires_at > created_at
revoked_at IS NULL OR revoked_at >= created_at
```

### 3.2 索引

| 索引 | カラム / 条件 | 用途 |
|------|-------|------|
| PK | `id` | 主キー |
| UNIQUE | `session_token_hash` | I-U3（同一 hash 最大 1 件）。`validateAndConsume` の高速ルックアップ |
| INDEX | `(photobook_id, expires_at)` WHERE `revoked_at IS NULL` | 同一 Photobook の有効 session 一覧（監査・abuse 検知） |
| INDEX | `expires_at` WHERE `revoked_at IS NULL` | 期限切れ session GC |
| INDEX | `revoked_at` WHERE `revoked_at IS NOT NULL` | revoked session の物理削除 |

### 3.3 重要な設計判断

- **`session_token_hash` は SHA-256（32 バイト固定）**: session 機構と同方針。raw token は 256bit 以上の暗号論的乱数のため、ストレッチング・ソルト不要 (参照: ADR-0005 / ADR-0003 共通方針)
- **`id` は UUIDv7**: ADR-0001。クライアント保持値**には載せない**
- **`session_token` を DB に保存しない**: 漏洩時の被害最小化。検証時は入力 token を SHA-256 ハッシュして照合
- **`photobook_id` への FK + ON DELETE CASCADE**: Photobook 物理削除時に session も削除される（孤児防止）
- **`revoked_at` UPDATE 方式（U1）**: MVP は UPDATE 方式で確定
- **`allowed_intent_count` は発行時固定**: 後から変更しない。運用で値を変える場合は別設定として実装し、本テーブルは「発行時の上限」として記録する
- **アトミック消費**: `UPDATE ... SET used_intent_count = used_intent_count + 1 WHERE id=$1 AND used_intent_count < allowed_intent_count` で 0 行なら回数超過。PostgreSQL の行ロックで安全（U5 参照）
- **`sessions` テーブルとは独立**: ライフサイクル・利用回数制限・有効期限すべて異なるため統合しない

## 4. テーブル間の責務分担

### 4.1 Photobook 集約との関係

`upload_verification_sessions.photobook_id` は `photobooks.id` を FK 参照する。Photobook 集約のテーブル本体には触れない。

逆方向で Photobook 集約から本テーブルへの操作はなし（Image 集約からのみ呼ばれる）。

### 4.2 Image 集約との関係

Image 集約の `upload-intent` UseCase が `validateAndConsume(rawToken, photobookId)` を呼び出す。SQL イメージ:

```sql
UPDATE upload_verification_sessions
   SET used_intent_count = used_intent_count + 1
 WHERE session_token_hash = $1
   AND photobook_id = $2
   AND revoked_at IS NULL
   AND expires_at > now()
   AND used_intent_count < allowed_intent_count
RETURNING id;
```

0 行なら検証失敗（403）。1 行なら消費成功 → Image レコード作成 + presigned URL 発行へ。

### 4.3 sessions との関係

別テーブル。共通方針（hash アルゴリズム、bytea 32B、Cookie 属性等）のみ揃える。テーブル統合はしない（理由はドメイン設計 §2 / `sessions` データモデル §4.2）。

## 5. 保持期間と削除ポリシー

### 5.1 期限切れ session の GC

- `expires_at < now()` で無効。検証時に I-U2 で拒否されるが、行は残る
- **自動 reconciler `upload_verification_expired_gc`** が `expires_at < now() - INTERVAL '1 day'` の行を物理削除（バッファ 1 日は監査用）

### 5.2 Revoked session の GC

- `revoked_at IS NOT NULL` の session は無効
- **自動 reconciler `upload_verification_revoked_gc`** が `revoked_at < now() - INTERVAL '7 days'` の行を物理削除
- 7 日保持する理由: abuse 検知の監査・証跡のため

### 5.3 Photobook 削除時

- ON DELETE CASCADE により Photobook 物理削除（purge）時に自動連鎖削除
- 論理削除（`status='deleted'`）では行は残るが、新規 upload-intent は走らないため実害なし

### 5.4 Reconcile（後続）

`docs/design/cross-cutting/reconcile-scripts.md` で詳述する Reconcile スクリプト群:

- 自動 reconciler: `upload_verification_expired_gc`、`upload_verification_revoked_gc`
- 手動 `scripts/ops/reconcile/`: 必要に応じて `upload_verification_audit.sh` 等を後追加

## 6. マイグレーション実施方針

### 6.1 基本

- 1 機能 1 ファイル
- 本テーブルは新規作成のため初版マイグレーションのみ

### 6.2 初版 DDL（参考）

```sql
CREATE TABLE upload_verification_sessions (
    id                       uuid        PRIMARY KEY,
    session_token_hash       bytea       NOT NULL UNIQUE,
    photobook_id             uuid        NOT NULL REFERENCES photobooks(id) ON DELETE CASCADE,
    allowed_intent_count     int         NOT NULL DEFAULT 20 CHECK (allowed_intent_count > 0),
    used_intent_count        int         NOT NULL DEFAULT 0  CHECK (used_intent_count >= 0),
    expires_at               timestamptz NOT NULL,
    created_at               timestamptz NOT NULL DEFAULT now(),
    revoked_at               timestamptz,
    CHECK (used_intent_count <= allowed_intent_count),
    CHECK (expires_at > created_at),
    CHECK (revoked_at IS NULL OR revoked_at >= created_at)
);

CREATE INDEX upload_verification_active_idx
    ON upload_verification_sessions (photobook_id, expires_at)
    WHERE revoked_at IS NULL;

CREATE INDEX upload_verification_expired_gc_idx
    ON upload_verification_sessions (expires_at)
    WHERE revoked_at IS NULL;

CREATE INDEX upload_verification_revoked_gc_idx
    ON upload_verification_sessions (revoked_at)
    WHERE revoked_at IS NOT NULL;
```

## 7. インフラ層のマッピング指針

[.agents/rules/domain-standard.md](../../../../.agents/rules/domain-standard.md) に従う。集約ではないため、ディレクトリ名は `auth/upload_verification` とする。

```
backend/internal/auth/upload_verification/
├── domain/
│   ├── upload_verification_session.go   # 概念単位
│   └── vo/
│       ├── upload_verification_id/      # UUIDv7
│       ├── upload_verification_token/   # raw 256bit 乱数（メモリ中のみ）
│       ├── upload_verification_token_hash/  # SHA-256（DB 保存）
│       ├── allowed_intent_count/
│       └── used_intent_count/
├── infrastructure/
│   └── repository/rdb/
│       ├── upload_verification_repository.go  # 永続化、issue / validateAndConsume / revoke
│       ├── marshaller/
│       │   └── upload_verification_marshaller.go
│       └── tests/
│           └── upload_verification_builder.go
└── internal/
    ├── usecase/
    │   ├── issue_upload_verification.go       # API（Turnstile 検証成功直後）から呼ばれる
    │   ├── validate_and_consume.go            # Image upload-intent UseCase から呼ばれる
    │   └── revoke_upload_verification.go      # 運営 / cmd/ops から呼ばれる
    └── controller/
        └── upload_verification_handler.go     # POST /api/auth/upload-verification
```

`validateAndConsume` は **独立 TX**（案T2、ドメイン設計 §6.3）で実装する。Image 作成失敗時もロールバックしない。
