# Session 認可機構 ドメイン設計

> 上流: [auth README](../README.md) / [業務知識定義書 v4](../../../spec/vrc_photobook_business_knowledge_v4.md) §2.3, §2.4, §3.1, §3.2, §3.4, §6.15 / [ADR-0003 フロントエンド認可フロー](../../../adr/0003-frontend-token-session-flow.md)
>
> 本書が扱うのは **draft / manage の操作 session**（HttpOnly Cookie で配布される短命認可セッション）。Turnstile セッション化（Image アップロード許可）は別設計 [`../upload-verification/`](../upload-verification/ドメイン設計.md) で扱う。
>
> 集約ではなくアプリケーション層の認可機構として扱う理由は [auth README](../README.md) を参照。

---

## 1. 機構の責務

`draft_edit_token` / `manage_url_token`（raw token、URL に含まれる）を、**短命の操作 session に交換する** 横断認可機構。

### 担う責務

- raw token 検証成功時に session を発行する（`session_token` 生成、ハッシュ化して DB 保存）
- API リクエストごとに Cookie session の有効性を検証する
- 編集系 API 成功時の `last_used_at` 更新（draft 延長と連動）
- 明示破棄（共有 PC 対策）の受付と revoke 反映
- Photobook publish 時の draft session 一括 revoke
- manage_url_token 再発行時の旧 version manage session 一括 revoke
- 期限切れ session の GC 候補抽出（実行は Reconcile）

### 担わない責務

- raw token そのものの発行・検証規則 → **Photobook 集約**（`DraftEditTokenGenerator` / `ManageUrlTokenGenerator`）
- Turnstile 検証結果の保持 → **upload-verification 機構**
- API Middleware の middleware 実装そのもの → Application 層 / chi ハンドラ
- 破壊的操作のワンタイム確認トークン → Application/API 設計（U4）

## 2. 境界と責任

集約ではないため「集約ルート」は存在しない。代わりに **概念単位 `Session`** とそのライフサイクル操作を定義する。

### 概念単位: `Session`

```
Session
├── id                    SessionId            (DB 内部 ID、UUIDv7)
├── sessionTokenHash      SessionTokenHash     (Cookie の raw token の SHA-256)
├── sessionType           SessionType          ('draft' | 'manage')
├── photobookId           PhotobookId          (対象フォトブック)
├── tokenVersionAtIssue   TokenVersionAtIssue  (発行時の Photobook.manage_url_token_version)
├── expiresAt             Timestamp            (有効期限)
├── createdAt             Timestamp            (発行時刻)
├── lastUsedAt            Timestamp?           (最後に有効リクエストで使われた時刻)
└── revokedAt             Timestamp?           (revoke された時刻、NULL なら有効)
```

### Photobook 集約との関係

- Photobook 集約は `sessions` テーブルを直接持たない（参照: Photobook ドメイン設計 §13.3, §14.1）
- Photobook 集約の `publishFromDraft` / `reissueManageUrl` は本機構のドメインサービスを **同一 TX で呼び出す**
- 本機構は Photobook 集約のテーブルや内部状態には触れない

## 3. エンティティ（仮想）

集約ルートではないが、`Session` を 1 つの永続化単位として扱う。

| 属性 | 型 | 説明 |
|------|----|----|
| `id` | `SessionId` | DB 内部 ID（UUIDv7、ADR-0001） |
| `sessionTokenHash` | `SessionTokenHash` | Cookie の raw token の SHA-256（32 バイト） |
| `sessionType` | `SessionType` | `draft` / `manage` |
| `photobookId` | `PhotobookId` | 対象 Photobook（Photobook 集約への参照のみ） |
| `tokenVersionAtIssue` | `TokenVersionAtIssue` | 発行時の `Photobook.manage_url_token_version` の snapshot |
| `expiresAt` | `Timestamp` | 有効期限（draft: `draft_expires_at` まで、manage: 24h〜7d） |
| `createdAt` | `Timestamp` | 発行時刻 |
| `lastUsedAt` | `Timestamp?` | 最後に有効リクエストで使われた時刻 |
| `revokedAt` | `Timestamp?` | revoke された時刻。NULL なら有効 |

## 4. 値オブジェクト

### 識別子・ハッシュ系

| VO | 形式 | 制約 |
|----|------|-----|
| `SessionId` | UUIDv7 | DB 内部 ID。Cookie に**は載せない** |
| `SessionToken` | base64url 文字列 | **256bit 以上の暗号論的乱数**を base64url 化（推奨 32 バイト = 43 文字）。Cookie に載る raw 値 |
| `SessionTokenHash` | bytea 32 バイト | `SessionToken` の **SHA-256**。DB 保存はこちらのみ <!-- 付録C P0-16 --> |

**重要**: `SessionId` と `SessionToken` は別物。Cookie には `SessionToken`（raw 乱数）を入れる。DB には `SessionTokenHash` のみを保存し、`SessionId` は内部追跡用。Cookie に `SessionId` を入れない理由は ADR-0001 / ADR-0003 の通り、UUIDv7 は時刻情報を含むため Bearer 的扱いの値としては予測困難性が劣る。

### セッション分類

| VO | 値 | 意味 |
|----|----|----|
| `SessionType` | `draft` | draft 編集 session |
| `SessionType` | `manage` | publish 後の管理 session |

### 期限・連動

| VO | 制約 |
|----|-----|
| `TokenVersionAtIssue` | 非負整数。`Photobook.manage_url_token_version` の発行時 snapshot |
| `ExpiresAt` | `createdAt < ExpiresAt`、draft session は `Photobook.draft_expires_at` を超えない |
| `LastUsedAt` | `createdAt <= LastUsedAt <= ExpiresAt` |
| `RevokedAt` | NULL または `createdAt <= RevokedAt` |

## 5. 不変条件（Invariants）

`<!-- 付録C P0-14, P0-15, P0-16 -->`

### セッション状態

- **I-S1**: `sessionTokenHash` は SHA-256（32 バイト固定）。raw token は DB に保存しない (参照: ADR-0003) <!-- 付録C P0-16 -->
- **I-S2**: `sessionType` は `draft` または `manage` のいずれか
- **I-S3**: `revokedAt IS NOT NULL` または `expiresAt < now()` のとき session は無効。検証時に拒否する
- **I-S4**: 同一 `sessionTokenHash` は DB 内に最大 1 件（衝突時は新規発行を再試行）

### 発行時

- **I-S5**: draft session 発行時、`tokenVersionAtIssue = 0`（draft 中は `manage_url_token_version` が未利用のため）
- **I-S6**: manage session 発行時、`tokenVersionAtIssue = Photobook.manage_url_token_version` の現在値を snapshot として記録 (参照: ADR-0003) <!-- 付録C P0-15 -->
- **I-S7**: draft session の `expiresAt` は `Photobook.draft_expires_at` を超えない
- **I-S8**: manage session の `expiresAt` は発行時刻 + 24h〜7d の範囲（実装初期値: 7d、運用調整可）

### Photobook publish 連動

- **I-S9**: Photobook の `publishFromDraft` 成功時、対象 `photobookId` かつ `sessionType='draft'` かつ `revokedAt IS NULL` の session はすべて `revokedAt = now()` に更新される（同一 TX 内） (参照: v4 §3.2 / ADR-0003)

### Manage URL 再発行連動

- **I-S10**: Photobook の `reissueManageUrl` 成功時、対象 `photobookId` かつ `sessionType='manage'` かつ `tokenVersionAtIssue <= oldVersion` かつ `revokedAt IS NULL` の session はすべて `revokedAt = now()` に更新される（同一 TX 内）
- **I-S11**: I-S10 は draft session には影響しない（draft は publish 済みの場合既に I-S9 で revoke されている、未 publish の場合は manage_url_token がまだ存在しない）

### 検証

- **I-S12**: API リクエスト時の検証は (a) Cookie の raw `SessionToken` を SHA-256 → (b) `sessionTokenHash` で照合 → (c) `revokedAt IS NULL` 確認 → (d) `expiresAt > now()` 確認 → (e) `photobookId` 一致確認 → (f) `sessionType` 一致確認 の 6 段階
- **I-S13**: 上記いずれかが失敗したら 401 相当を返し、Cookie を削除する `Set-Cookie` を発行する
- **I-S14**: 検証成功時、編集系 API のみ `lastUsedAt = now()` を更新する。GET / プレビューでは更新しない（draft 延長ルールと連動）

## 6. ドメイン操作

集約ルートではないため、ドメインサービス的な独立関数として定義する。すべて Application 層 UseCase または API Middleware から呼ばれる。

| 操作 | 呼び出し元 | 効果 | TX |
|------|----------|------|-----|
| `issueDraftSession(photobookId, expiresAt)` | Photobook draft 入場 UseCase | `Session` を新規 INSERT、`SessionToken` を返す | 自身の TX |
| `issueManageSession(photobookId, tokenVersionAtIssue, expiresAt)` | Photobook manage 入場 UseCase | `Session` を新規 INSERT、`SessionToken` を返す | 自身の TX |
| `validate(rawToken, photobookId, sessionType)` | API Middleware | I-S12 を実行、無効なら拒否 | 読み取りのみ |
| `touch(sessionId)` | API Middleware（編集系のみ） | `lastUsedAt = now()` を更新 | 自身の TX |
| `revoke(sessionId)` | 明示破棄 UI | `revokedAt = now()` に更新 | 自身の TX |
| `revokeAllDrafts(photobookId)` | **Photobook `publishFromDraft`** | 対象の draft session を全 revoke | **呼び出し元の TX に参加** |
| `revokeAllManageByTokenVersion(photobookId, oldVersion)` | **Photobook `reissueManageUrl`** | 旧 version 以下の manage session を全 revoke | **呼び出し元の TX に参加** |
| `expireAllForPhotobook(photobookId)` | Photobook purge UseCase | 対象 Photobook の全 session を revoke | 呼び出し元の TX に参加 |

### 6.1 `revokeAllDrafts(photobookId)` の詳細 <!-- 付録C P0-5 -->

Photobook 集約 §14.1 の引き継ぎを実装する操作。

**SQL イメージ**:
```sql
UPDATE sessions
   SET revoked_at = now()
 WHERE photobook_id = $1
   AND session_type = 'draft'
   AND revoked_at IS NULL;
```

**呼び出し規約**:
- 呼び出し元（Photobook `publishFromDraft`）の同一 TX 内で実行する
- 戻り値で affected rows を返してもよいが、業務上は使わない（あれば監査ログには出せる）
- エラーは伝播させる（呼び出し元 TX をロールバックする責務がある）

### 6.2 `revokeAllManageByTokenVersion(photobookId, oldVersion)` の詳細

**SQL イメージ**:
```sql
UPDATE sessions
   SET revoked_at = now()
 WHERE photobook_id = $1
   AND session_type = 'manage'
   AND token_version_at_issue <= $2
   AND revoked_at IS NULL;
```

**呼び出し規約**:
- Photobook `reissueManageUrl` の同一 TX 内で実行
- `oldVersion` は再発行**前**の値（`Photobook.manage_url_token_version` を +1 する**前**の値）
- 新 version 由来の session（同一 TX で発行されたもの）には影響しない

### 6.3 `validate(rawToken, photobookId, sessionType)` の詳細

API Middleware から各リクエストで呼ばれる。

**手順**:
1. `rawToken` を SHA-256 で hash 化 → `sessionTokenHash`
2. `sessions` を `(sessionTokenHash, photobookId, sessionType)` で検索
3. ヒットなければ 401
4. `revokedAt IS NOT NULL` なら 401
5. `expiresAt < now()` なら 401
6. ヒットすれば `Session` を返す（呼び出し元の Middleware が `request.context` に格納）

**性能**: `(session_token_hash)` UNIQUE インデックスで O(1) 相当のルックアップ。

## 7. ドメインイベント

本機構は **Outbox にイベントを乗せない** 方針。理由:

- session 発行・失効は内部認可状態の変化であり、業務イベントではない
- Photobook publish や reissueManageUrl の Outbox イベント（`PhotobookPublished` / `ManageUrlReissued`）の延長として処理されるべきもの
- session GC（Reconcile）も内部状態の整理であり、副作用通知は不要

ただし、**監査ログ**（access log）として「session 発行 / 検証失敗 / revoke」を構造化ログに出す方針は別途整備する（U4 と同じく Application/API 設計に委ねる）。

## 9. 典型的なユースケース

### 9.1 Draft 入場（v4 §3.1, ADR-0003）

```
1. ユーザーが /draft/{draft_edit_token} にアクセス
2. Photobook 集約が draft_edit_token のハッシュ照合で検証
3. 検証成功 → issueDraftSession(photobookId, expiresAt = Photobook.draft_expires_at) を呼ぶ
4. 同一 TX 内で:
   - SessionToken を 32 バイト乱数で生成
   - sessionTokenHash = SHA-256(SessionToken) を計算
   - sessions に INSERT (sessionType='draft', tokenVersionAtIssue=0)
5. Set-Cookie: vrcpb_draft_{photobook_id}=<SessionToken>; HttpOnly; Secure; SameSite=Strict; Path=/
6. /edit/{photobook_id} に redirect（URL から raw token を除去）
7. 以後の API は Cookie session のみで認可
```

### 9.2 Manage 入場（v4 §3.4, ADR-0003）

```
1. ユーザーが /manage/token/{manage_url_token} にアクセス
2. Photobook 集約が manage_url_token のハッシュ照合で検証
3. 検証成功 → issueManageSession(photobookId, tokenVersionAtIssue=Photobook.manage_url_token_version, expiresAt=now+7d) を呼ぶ
4. 同一 TX 内で:
   - SessionToken を 32 バイト乱数で生成
   - sessionTokenHash = SHA-256(SessionToken) を計算
   - sessions に INSERT (sessionType='manage', tokenVersionAtIssue=現在の Photobook.manage_url_token_version)
5. Set-Cookie: vrcpb_manage_{photobook_id}=<SessionToken>; HttpOnly; Secure; SameSite=Strict; Path=/
6. /manage/{photobook_id} に redirect
7. 以後の API は Cookie session のみで認可
```

### 9.3 Publish 時の draft session 全 revoke <!-- 付録C P0-5 -->

```
Photobook publishFromDraft UseCase 内で:
  TX 開始
    1. Photobook の I7 を検証
    2. Photobook を published に遷移、public_url_slug / manage_url_token を発行
    3. revokeAllDrafts(photobookId) を呼ぶ
       → sessions UPDATE で対象の draft session を全 revoke
    4. outbox_events に PhotobookPublished を INSERT
    5. Photobook.version +1
  TX コミット
```

### 9.4 Manage URL 再発行時の manage session 全 revoke

```
Moderation reissueManageUrl UseCase 内で:
  TX 開始
    1. oldVersion = Photobook.manage_url_token_version を取得
    2. Photobook.manage_url_token_hash を新トークンのハッシュに更新
    3. Photobook.manage_url_token_version = oldVersion + 1
    4. revokeAllManageByTokenVersion(photobookId, oldVersion) を呼ぶ
       → sessions UPDATE で旧 version 以下の manage session を全 revoke
    5. ModerationAction を INSERT
    6. ManageUrlDelivery を新規作成（運営入力の recipient_email）
    7. outbox_events に ManageUrlReissued を INSERT
    8. Photobook.version +1
  TX コミット
```

### 9.5 明示破棄（共有 PC 対策）

```
1. ユーザーが管理画面で「この端末から管理権限を削除」をクリック
2. クライアントが POST /api/auth/sessions/revoke を呼ぶ
3. Middleware が Cookie session を validate
4. revoke(sessionId) を呼ぶ → sessions.revoked_at = now()
5. Set-Cookie: vrcpb_manage_{photobook_id}=; Max-Age=0 で Cookie 削除
6. 案内画面へ遷移（管理 URL から再入場可能と伝える）
7. 元の manage_url_token そのものは失効しない（別端末からの再入場を妨げない）
```

### 9.6 期限切れ session の検出

```
1. API Middleware が validate を呼ぶ
2. expiresAt < now() を検出 → 401
3. Set-Cookie で Cookie 削除
4. クライアントは「管理 URL または Draft URL から再入場してください」画面を表示
5. 期限切れレコード自体は session_expired_gc reconciler が定期削除（Reconcile §11）
```

## 10. CSRF / Cookie / ヘッダ方針（ADR-0003 抜粋）

### Cookie 属性

- `HttpOnly: true`
- `Secure: true`
- `SameSite: Strict`
- `Path: /`
- `Domain`: 未指定が基本候補（U2、M1 スパイクで確定）
- 期限: draft session は `Photobook.draft_expires_at` まで、manage session は 24h〜7d（初期値 7d）

### CSRF 対策

- `SameSite=Strict` 一次対策
- 状態変更 API では `Origin` ヘッダ検証必須
- CORS は自サイトのみ
- 破壊的操作（削除・公開範囲変更・管理 URL 再発行・センシティブ変更）には**ワンタイム確認トークン**を要求（実装は本設計では深追いしない、U4）

### ログ・エラートラッキング

- raw `SessionToken` / `sessionTokenHash` どちらも構造化ログの**禁止フィールド**に登録
- Sentry の `beforeSend` で URL の token セグメントをスクラブ
- `authorization`, `cookie`, `set-cookie` ヘッダ値もログ非出力

## 12. 次設計への引き継ぎ事項

### 12.1 Photobook publish UseCase

- `publishFromDraft` の同一 TX 内で `revokeAllDrafts(photobookId)` を呼ぶこと <!-- 付録C P0-5 -->
- 戻り値は無視してよい（監査ログ用に affected rows を取りたければ別途）
- エラー伝播で TX をロールバック

### 12.2 Moderation `reissueManageUrl` UseCase

- 同一 TX 内で `revokeAllManageByTokenVersion(photobookId, oldVersion)` を呼ぶこと
- `oldVersion` は再発行**前**の `Photobook.manage_url_token_version`
- 4 要素同一 TX（Photobook / ModerationAction / ManageUrlDelivery / Outbox）に session UPDATE が **5 番目** として参加する

### 12.3 API Middleware / Route Handler

- 各リクエストで Cookie から `SessionToken` を抽出 → `validate(rawToken, photobookId, sessionType)` を呼ぶ
- Cookie 名は `vrcpb_draft_{photobook_id}` または `vrcpb_manage_{photobook_id}`、photobook_id は URL パスから抽出
- 編集系 API のみ `touch(sessionId)` を呼ぶ
- 検証失敗時は 401 + Cookie 削除

### 12.4 Photobook 編集系 UseCase の draft 延長

- 編集系 API 成功時、Photobook 集約の `touchDraft()` で `draft_expires_at = now + 7日` に延長
- 同一 TX 内で `sessions.expires_at` も `Photobook.draft_expires_at` に合わせて更新するかは**実装判断**（基本は `expiresAt = min(原値, Photobook.draft_expires_at)` で運用、後続で精査）
- GET やプレビューでは **延長しない**

### 12.5 明示破棄 UI

- Photobook 管理画面・編集画面に「この端末の編集/管理権限を削除」ボタンを置く
- POST /api/auth/sessions/revoke を呼ぶ
- レスポンスは Cookie 削除 + 案内画面遷移指示

### 12.6 Reconcile

- **expired session GC**: `expires_at < now() - INTERVAL '7 days'` の session は物理削除候補（自動 reconciler）
- **revoked session GC**: `revoked_at < now() - INTERVAL '30 days'` の session も物理削除候補
- 詳細は `docs/design/cross-cutting/reconcile-scripts.md` に集約

### 12.7 破壊的操作のワンタイム確認トークン（U4）

本設計では深追いしない。Application/API 設計で:

- スコープ: 1 操作 1 トークン
- 有効期限: 5〜10 分
- 保存先: `sessions` テーブル拡張 or 別テーブル（決定は後続）

## 13. 未解決事項

### U1: revoke 方式

**MVP では `revoked_at` UPDATE 方式で確定**（INSERT-only 履歴方式は Phase 2 で検討）。

理由:
- session 寿命が短い（最大 7 日）ため履歴量が膨大にならない
- INSERT-only 方式は監査ログ的価値が高いが、MVP の運営規模では過剰
- 30 日経過した revoked session は物理削除し、それ以降の追跡は構造化ログ（access log）に委ねる

### U2: Cookie の Domain 属性

M1 スパイクで Cloudflare Pages の Cookie 挙動を確認後に確定。

**現時点の基本候補**: `Domain` 未指定（= 発行ホストのみ有効、サブドメインに広がらない）。

サブドメインで運営画面を別ホストに切り出す場合は別途設定が必要だが、MVP は単一ドメイン前提（運営は `cmd/ops` で HTTP API なし）。

### U3: 別端末同時編集時の楽観ロック衝突 UX

**ドメイン上の挙動**: 複数 manage session が同 Photobook で並行 UPDATE すると、後勝ちで `Photobook.version` 不一致 → `OptimisticLockConflict` を返す。

**画面文言・復旧 UX**: M7 で確定。本設計では「衝突時はエラーを返す」のみ規定。

### U4: 破壊的操作のワンタイム確認トークン

**本設計では深追いしない**。Application/API 設計で:

- 配置: `sessions` テーブル拡張 or 別テーブル `confirmation_tokens` 新設
- 期限: 5〜10 分
- スコープ: 1 操作 1 トークン
- 削除トリガ: 消費後即削除、期限切れは Reconcile
