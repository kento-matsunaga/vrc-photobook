# 失敗→ルール変換スキル

## 入力
- テスト失敗ログ、レビュー指摘、またはインシデントレポート

## 手順

### Step 1: 失敗の分類

| カテゴリ | 例 | 対策タイプ |
|---------|-----|----------|
| テスト構造 | フラットt.Run、Builder誤用 | ルール（testing系） |
| ドメインモデル | 不変条件破壊、型安全性欠如 | ルール（domain系） |
| セキュリティ | 認証バイパス、テナント漏洩 | ルール（security系） |
| インフラ | DB接続、マイグレーション | スキル + テスト |
| プロセス | テスト未実行、レビュー漏れ | フック |

### Step 2: 根本原因分析

```markdown
## 5 Whys 分析

1. なぜ失敗したか？ → {直接原因}
2. なぜ {直接原因} が起きたか？ → {間接原因}
3. なぜ {間接原因} が起きたか？ → {構造的原因}
4. なぜ {構造的原因} が起きたか？ → {プロセス的原因}
5. なぜ {プロセス的原因} が起きたか？ → {根本原因}
```

### Step 3: ルール/スキルの生成

#### ルールを生成する場合:

```markdown
---
description: "{ルールの1行説明}"
globs: ["{適用対象のファイルパターン}"]
---

# {ルール名}

## 必須/禁止事項
- {具体的なルール}

## 正しい例
```code
// ✅ こうする
```

## 誤った例
```code
// ❌ これは禁止
```

## Why
{根本原因の説明 — なぜこのルールが必要になったか}
```

#### スキルを生成する場合:
`SKILL.md` を `.agents/skills/{name}/` に作成

### Step 4: テストの追加
生成したルール/スキルが機能することを検証するテストを追加する。

### Step 5: 失敗ログの更新
`harness/failure-log/` の該当エントリに対策完了を記録する。

### Step 6: 品質スコアの更新
`harness/QUALITY_SCORE.md` を更新する。

## 出力
- 新しいルールファイル or スキルディレクトリ
- 検証テスト
- 更新された失敗ログ
- 更新された品質スコア
