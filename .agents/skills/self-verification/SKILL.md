# 自己検証スキル

## 入力
- 完了したタスクの内容
- 変更したファイルの一覧

## 手順

### Step 1: ビルド検証
```bash
# プロジェクトのビルドが通ることを確認
# 言語/フレームワークに応じたビルドコマンドを実行
```
- [ ] ビルドエラーなし
- [ ] 新しい警告なし

### Step 2: Lint 検証
```bash
# リンターを実行
```
- [ ] lint エラーなし
- [ ] 型チェックエラーなし

### Step 3: テスト検証
```bash
# テストを実行
```
- [ ] 既存テストが全て通る
- [ ] 新規テストが追加されている
- [ ] 新規テストが全て通る

### Step 4: テストガイドライン準拠
- [ ] テーブル駆動テストである
- [ ] description フィールドに Given/When/Then がある
- [ ] メソッドテストで Builder を使用している
- [ ] コンストラクタテストで Builder を使用していない
- [ ] ヘルパー関数を *_test.go に追加していない
- [ ] fixture.go を作成していない

### Step 5: ドキュメント整合性
- [ ] docs/ の業務知識と実装が矛盾していない
- [ ] 新しいビジネスルールが docs/ に反映されている

### Step 6: アーキテクチャ準拠
- [ ] ドメイン層が外部依存を持っていない
- [ ] レイヤー間の依存方向が正しい（外→内）
- [ ] セキュリティガードルールに違反していない

### Step 7: ハーネスフィードバック
- [ ] 品質スコアへの影響を確認
- [ ] 失敗があれば harness/failure-log/ に記録
- [ ] 新しいルール/スキルの必要性を検討

## 出力

```markdown
## 自己検証レポート

### 結果: ✅ PASS / ❌ FAIL

### 検証サマリー
| Step | 結果 | 備考 |
|------|------|------|
| ビルド | ✅/❌ | |
| Lint | ✅/❌ | |
| テスト | ✅/❌ | |
| ガイドライン | ✅/❌ | |
| ドキュメント | ✅/❌ | |
| アーキテクチャ | ✅/❌ | |
| フィードバック | ✅/❌ | |

### 発見事項
- （検証中に発見した問題や改善点）

### 次のアクション
- （必要な追加作業）
```
