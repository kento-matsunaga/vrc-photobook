# ドキュメントガーデニングスキル

## 入力
- チェック対象のディレクトリ（省略時: `docs/` + `.agents/rules/`）

## 手順

### Phase 1: テスト対応チェック
ドキュメントに記載されたBDDシナリオ（Given/When/Then）に対応するテストが存在するか確認する。

1. `docs/` 配下の業務知識ドキュメントからシナリオを抽出
2. `docs/ディレクトリマッピング.md` でコード側の対応ディレクトリを特定
3. テストファイル内の `description` フィールドとシナリオを照合
4. 未対応のシナリオをリストアップ

### Phase 2: データモデル整合性チェック
ドキュメントに記載されたデータモデルと実際のスキーマ/マイグレーションの整合性を確認する。

1. データモデルドキュメントのテーブル・カラム定義を抽出
2. 実際のスキーマファイル（SQL, HCL, migration）と照合
3. 不一致（追加・削除・型変更）をリストアップ

### Phase 3: ルール品質チェック
`.agents/rules/` のルールファイルの品質を検証する。

1. **重複検出**: 同じ内容を異なるルールが記述していないか
2. **矛盾検出**: ルール間で矛盾する指示がないか
3. **陳腐化検出**: コードの変更により不要になったルールがないか
4. **フォーマット検証**: フロントマター、Whyセクションの存在

### Phase 4: リンク整合性チェック
Markdownファイル内のリンクが有効か確認する。

1. `docs/` と `.agents/` 配下の全Markdownファイルをスキャン
2. 内部リンク（相対パス）の参照先が存在するか確認
3. 壊れたリンクをリストアップ

### Phase 5: フィードバックログ分析
`harness/failure-log/` の蓄積パターンを分析し、ルール/スキルへの昇格を提案する。

1. 失敗ログの全エントリを読み込み
2. カテゴリ・根本原因でグルーピング
3. **3回以上** 同じパターンが出現 → ルール/スキルへの昇格を推奨
4. `PENDING` ステータスのエントリに対策を促す

## 出力

```markdown

日時: YYYY-MM-DD

## サマリー
| Phase | チェック項目 | 問題数 |

| 1 | テスト対応 | N件 |
| 2 | データモデル整合性 | N件 |
| 3 | ルール品質 | N件 |
| 4 | リンク整合性 | N件 |
| 5 | フィードバック昇格候補 | N件 |

## Phase 1: テスト対応
- 未対応シナリオ: {リスト}

## Phase 2: データモデル整合性
- 不一致: {リスト}

## Phase 3: ルール品質
- 重複: {リスト}
- 矛盾: {リスト}
- 陳腐化: {リスト}

## Phase 4: リンク整合性
- 壊れたリンク: {リスト}

```
