# 2026-04-26 PR2 完了報告の後半で出力フォーマット崩れ

## 発生状況

- **何をしようとしていたか**: M2 本実装 PR2（backend config / logger / graceful shutdown / readyz）の完了報告をユーザーへ提出。
- **どのファイル/モジュールで発生したか**: Claude のチャット出力（リポジトリ内ファイルではない）。

## 失敗内容

- 完了報告の後半セクション（§9 Secret 確認 / §10 実リソース操作 / §11 コミット ID / §12 次工程）で、Markdown 表の表崩れまたは長文の折り返しが発生し、ユーザー側で視認性が下がった。
- ユーザーから「報告フォーマットが崩れた原因を確認してから PR3 へ」と指摘あり。

## 根本原因

- 長い完了報告で **Markdown 表を多用しすぎた**（PR2 報告は §1〜§12 の 12 セクション、うち多くが 2〜3 列の表）。
- 表が連続するとターミナル / Markdown レンダラ側で折り返しが起きやすく、特に Bash の引用ブロックや絵文字を含む列で行が崩れる。
- 検証ログ（curl 出力 + slog JSON 出力）を表内に詰め込むと等幅幅に収まらない。

## 影響範囲

- **リポジトリ内ファイル / コミット内容には影響なし**（git status クリーン、コミット `c85b663` 存在、`go vet` / `go build` 両方 OK、ファイル先頭行に文字化けなし）。
- ユーザー側の確認コストが増えた（同じ情報を再度短い形式で要求された）。
- 今後の長文報告で同種の問題が再発する可能性。

## 対策種別

- [x] ルール化（運用ルールの追加）
- [ ] スキル化
- [ ] テスト追加
- [ ] フック追加

### 採用する報告フォーマット規約（自己ルール）

長い完了報告では以下を守る:

1. **表を減らし、箇条書き中心**にする
2. **コミット ID と検証結果は最後に短い形式で再掲**する（ユーザーがスクロール戻りせずに済む）
3. **コードブロック / curl 出力 / slog JSON 出力を表の中に入れない**（必ず表の外、独立した行で）
4. **8 セクション以上の長い完了報告**は、冒頭に「結論 3 行（コミット ID / 主要結果 / 次工程）」を置いて短く読めるようにする
5. **絵文字 ✅ / ❌ は列の最後の値としてのみ**使う（表中央や複数列に散らさない）

### コードへの反映

不要（チャット出力フォーマットの問題、リポジトリ側の修正なし）。

## 教訓

- **完了報告は「ユーザーが読みやすい」を最優先**。Claude 側で構造化した気になっていても、表の連続は視認性を下げる。
- 報告が長くなりがちな PR / 計画書承認系では、最初に **結論 3 行**を置く運用が効く。
- 確認用ログ（curl / slog 出力 / vet 結果）は **表の外**、コードブロックで独立配置するのが安全。
- ユーザーから「報告崩れた」と言われたら **リポジトリ健全性をまず確認**（コミット / ファイル / vet）し、コードに影響がなければ純粋に報告フォーマット側の改善で完結させる。
